# テーマ11 2Dダイナミックライティング / 影 ― LittleJS 版

暗いトップダウンの部屋を、プレイヤーの灯りと多数の動的な色付き光源が照らし、矩形の柱（オクルーダ）が**影**を落とす
ライティングベンチを **LittleJS** で実装したもの。**多数の動的光源 ＋ ライトマップ合成（加算→乗算）＋ 影生成**を
同時に回し、「同時光源数」と「影 ON/OFF の描画コスト差」を測る。

## 起動方法

CDN 読込のため `file://` でも動くが、`../assets/` の相対参照を安定させるため簡易HTTPサーバ経由を推奨。

```bash
# このフォルダ (11/LittleJS) で
python -m http.server 8000
# → ブラウザで http://localhost:8000/ を開く
```

`../assets/` が空でも起動する（光源の放射グラデはコードで生成、床/柱/プレイヤーは図形フォールバック）。

## 使用バージョン / CDN

- **LittleJS**（`@latest` 解決時点 / classic global build）
- 主URL: `https://unpkg.com/littlejsengine`
  失敗時フォールバック: `https://cdn.jsdelivr.net/npm/littlejsengine/dist/littlejs.min.js`
- 読込失敗時のみ `error` イベントを捕捉して jsDelivr ミラーへ切替。

## 操作

| キー | 動作 |
|---|---|
| `W`/`A`/`S`/`D` または 矢印 | プレイヤー（＝白色光源 半径240）を移動（220 px/s, 壁/柱で停止）|
| `+` / `-`（メイン行 or テンキー）| 動的光源の数を 6 ずつ増減（性能比較の主軸）|
| `L` | 影 ON / OFF 切替 |
| `R` | リセット |

## 仕様数値（全エンジン共通）

| 項目 | 値 |
|---|---|
| キャンバス / タイル | 960 x 540 / 32px（部屋 30 x 17）|
| ambient | 0.10 |
| プレイヤー光源 半径 / 速度 | 240 / 220 px/s |
| 動的光源 半径 / 速度 | 160 / ~120 px/s 相当の決定的軌道 |
| オクルーダ | 矩形・約16個 + 外周壁 |
| 動的光源数 | 初期 12・±6・下限 0・上限 120 |

## 実装メモ

### ライティングの実装方式（このテーマの肝）

LittleJS には高レベルな 2D ライト機構が無いため、**オフスクリーン 2D canvas にライトマップを自前生成 → multiply 合成**
という現実的経路を採る（SPEC のライトマップ方式）。毎フレーム:

1. オフスクリーン canvas（960x540）を `ambient`（0.10 の灰）でクリア。
2. 各光源を**加算合成（`globalCompositeOperation='lighter'`）**の放射グラデで描く。
   **影 ON** 時は、その光源から見た矩形オクルーダの**シルエット**を**黒（=光ゼロ）の影ポリゴン**として塗り、光を削る（ハードシャドウ）。
3. 出来たライトマップを、LittleJS が描いた暗いシーン（床・柱）へ **`multiply`** で重ねる（`gameRenderPost` で overlay/main の
   context に `globalCompositeOperation='multiply'` を設定して blit）。

### 座標系 / Y軸（最重要・このテーマの主な罠）

- LittleJS のワールドは **Y軸上向き**（画面下=Y小, 上=Y大）。一方、ライトマップを描く 2D canvas は **Y下向き**（左上原点）。
- 2つの座標系を毎フレーム混ぜると事故るので、本実装は**ライトマップの計算を最初から最後までスクリーン空間
  （左上原点・Y下向き・px）で完結**させる。
  - ゲーム論理（光源・プレイヤー・オクルーダの位置）も**スクリーン空間（Y下向き）で保持**。
  - `worldFromScreen()` で1回だけスクリーン→ワールドに変換して LittleJS の `drawRect`/`drawTile` に渡す。
  - ライトマップ canvas へはスクリーン座標をそのまま使う（変換不要）。
- `cameraScale=1` / カメラ中心=ワールド中心に固定。部屋高さ 544 はキャンバス 540 より僅かに高く上下 2px ずつはみ出すが、
  下地が暗いので視覚的に問題なし。スクリーン↔ワールドは線形 1:1。

### 決定的生成 / フォールバック

- 柱配置・光源軌道・色は `mulberry32` の決定的乱数（`Math.random` 不使用）。
- `light_glow.png` が無くても canvas の放射グラデを生成して必ず点灯。床/柱/プレイヤーも図形フォールバック。

## Codex 生成所感（軽量エンジン × 自前ライティングの所感・罠）

- **「ライトマップは LittleJS の外（2D canvas）で作る」と割り切るのが正解**。LittleJS は WebGL で世界を描くが、
  multiply 合成や放射グラデの加算は Canvas2D の `globalCompositeOperation` が一番素直。エンジンが描いた絵の上へ
  最後に1枚かぶせる、という構成にすると見通しが良かった。
- **最大の罠は二重の Y 軸**。「世界=Y上向き / ライトマップ canvas=Y下向き」を毎フレーム行き来すると、影が反対側に伸びる/
  光源がカーソルとズレる等が頻発する。**ライトマップ計算を完全にスクリーン空間に閉じ込め、世界へ渡す時だけ1回変換**する、
  という規約に固定して回避した（テーマ3/5 の Y 反転の知見と同根）。
- **WebGL レイヤと Canvas 合成の橋渡し**は、LittleJS が公開する `mainCanvas`/`mainContext`/`overlayContext` を使って
  blit する。ここはエンジンの内部構造を知らないと当てづらく、README に明記した（HUD は別途 HTML overlay）。
- **コストの所在が明快**。光源数ぶんの放射グラデ描画＋（影ONなら）影ポリゴン塗りが Canvas2D の塗り面積に比例するため、
  `+/-` と `L` で FPS が素直に動く。GPU ネイティブの Godot `PointLight2D` と、CPU/Canvas 合成の本実装で
  「同じライティングを別経路で出したときの差」を観測する題材として扱いやすい。
