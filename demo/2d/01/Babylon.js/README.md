# 弾幕STG（縦スクロールSTG） ― Babylon.js 版

テーマ1 共通仕様（`../SPEC.md`）を **Babylon.js** で実装したもの。
3Dエンジンである Babylon.js を使い、**正射影(Orthographic)カメラ + Sprite/Plane** で 2D シューティングを構成している。

## 使用バージョン
- Babylon.js: **CDN 最新安定版** `https://cdn.babylonjs.com/babylon.js`（グローバル `BABYLON`）
- 追加ライブラリなし・ビルド不要

## 起動方法
画像アセットを `fetch`/`Image` で読むため、`file://` では CORS/読込制限で
アセットが読めない場合がある（その場合でもフォールバック図形で**必ず起動**する）。
アセット画像をきちんと表示したい場合はローカルサーバ経由推奨。

```bash
# このフォルダ（1/Babylon.js/）で
python -m http.server 8000
# → ブラウザで http://localhost:8000/index.html
```

ダブルクリック（`file://`）でも単色フォールバック図形で動作する。

### フォルダ構成 / アセット
- アセットは `../assets/`（= `1/assets/`）から SPEC のファイル名で読み込む。
- 画像が存在しない・読込失敗時は **単色図形にフォールバック**：
  - 自機 = 水色三角、小型敵/大型敵 = 赤/桃の円、自機弾 = 黄円、敵弾 = 橙円、爆発 = 橙円
  - 背景画像が無ければ星空（流れる点）で代替

## 操作
- 移動: 矢印キー / WASD（8方向・画面内クランプ）
- 発射: オート連射（150ms ごと）
- `+` / `-`: 最大同時敵数を ±10（上限300）
- Game Over 後: Enter またはクリックでリスタート

## 実装メモ（SPEC 準拠の数値）
- 自機弾速 600px/s（上）、連射間隔 150ms
- 敵: 上端ランダムX出現、下方向 80〜140px/s、自機方向へ敵弾（200px/s）
- 敵スポーン: 初期同時最大 40 体（同時数を維持するよう小刻みに補充）
- 当たり判定: すべて**円判定**（自機弾×敵 / 敵弾×自機 / 敵×自機）
- 被弾 HP-1（初期3）、撃破 +10、撃破時 explosion を一瞬表示
- HP 0 で Game Over

### Babylon 特有の設計
- **座標系**: 正射影カメラを `orthoLeft=0, orthoRight=960, orthoTop=0, orthoBottom=540`
  に設定し、ワールド座標を**画面ピクセル座標（左上原点・y下向き）**に一致させた。
  これで「2Dゲームの座標計算をそのまま 3D 空間で書ける」ようにしている。
- **描画**: テクスチャがあれば `SpriteManager`/`Sprite`（GPU バッチ描画で大量弾に強い）。
  無ければ `Mesh`（三角ポリゴン / `CreateDisc`）+ `StandardMaterial(emissive, disableLighting)`
  にフォールバック。両者を同一インターフェース（`setPos/setVisible/setSize/...`）でラップ。
- **プール**: 弾・敵・エフェクトは dispose せずプールで再利用し GC スパイクを抑制。
- **HUD**: HTML オーバーレイ div。FPS（指数移動平均）/ Objects（弾+敵+FX 合計）/
  Score / HP / 最大敵数 / 描画モード（Sprite or Plane フォールバック）を常時表示。
- フォールバック Mesh は `backFaceCulling=false`（カメラが背面側から見るため）。

## Codex / AI コーディングでの生成しやすさ所感（3Dエンジンで2Dを書く難しさ）

- **最大のつまずきは座標系**。Babylon は 3D 前提で y が上向き・原点中央が標準。
  2D STG の「左上原点・y下向き・ピクセル単位」に合わせるには正射影カメラの
  `orthoTop/orthoBottom` を**意図的に反転**（top=0, bottom=540）させる必要があり、
  AI が素直に書くと y が上下逆・座標がワールド単位（数メートル）になりがち。
  ここは「画面ピクセル=ワールド単位」と最初に決め打ちしないと延々ズレる。
- **2Dの当たり判定/移動ロジック自体は 3D と無関係**なので生成は容易。難所は
  「2D 用に 3D API をどう設定するか」という**エンジン作法**の部分に集中する。
- **Sprite vs Mesh の二択**。大量描画なら `SpriteManager` がバッチで効くが、
  テクスチャ前提なので「画像なしフォールバック」を求めると Mesh + Material 経路が
  別途必要になり、**2系統のレンダリングパスを統一ラップ**する一手間が発生する。
  AI は片方だけ書きがちで、フォールバック要件を明示しないと欠落しやすい。
- **Sprite は常にカメラへビルボード**するため 2D では扱いやすい一方、Mesh は
  カメラの向き次第で裏面（カリングで消える）になる罠がある。`backFaceCulling=false`
  か向きの統一が必要で、これも明示しないと「図形が見えない」事故になりやすい。
- **ライティング**。デフォルトのマテリアルはライト依存で暗くなるため、2D では
  `emissiveColor` + `disableLighting=true` が定石。AI はライト設置で解決しようと
  して色が安定しないことがある。
- 総じて、**ロジックより「2D 化のための 3D 初期設定」が生成の肝**。プロンプトで
  「正射影・ピクセル座標・y下向き・emissive unlit・フォールバック必須」を
  明示すると一発で通りやすい。Pixi/Phaser のような純2Dエンジンに比べると、
  AI 生成時に与えるべき前提知識が一段多い。
