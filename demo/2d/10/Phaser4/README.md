# テーマ10 マッチ3パズル ― Phaser 4 版

ぷよ/ツムツム/Bejeweled 風のマッチ3パズル。NxN グリッドの宝石(6種)を隣と入れ替え、
横/縦に同種3連以上が揃うと消え、上から落ちて補充され連鎖する。
**描画は単純な矩形/丸だが、ロジック(O(N²) マッチ走査・連鎖・盤面状態管理)と
大量の同時トゥイーン(落下/消滅を「セル単位」で張る)が主役**な、描画/物理が重い既存テーマとは
対照的な負荷を測る比較用サンプル。このエンジンの比較対象機構は **Phaser Tweens**。

## 起動方法

ブラウザのセキュリティ制約(file:// での CORS / アセット読込)を避けるため、
簡易 HTTP サーバ経由での起動を推奨します。

```bash
# このフォルダ (10/Phaser4/) で
python -m http.server 8000
# → http://localhost:8000/ をブラウザで開く
```

または任意の静的サーバ(`npx serve` など)でも可。
アセット画像が無くてもフォールバック描画で必ず起動します(初期状態はオート ON で自動進行)。

## 使用バージョン

- Phaser **4**(CDN: `https://cdn.jsdelivr.net/npm/phaser@4/dist/phaser.min.js`)
- 追加ライブラリなし。`index.html` + `game.js` のみ。
- **トゥイーン機構 = Phaser Tweens**(`this.tweens.add({...})`)。物理エンジンは未使用。

## 操作

| キー / 操作 | 動作 |
|---|---|
| クリック2回 | 隣接2セルを選択して手動スワップ(隣接のみ有効) |
| `Space` | オートプレイ ON/OFF トグル(初期 **ON**) |
| `+` / `-`(テンキー可) | 盤面サイズ N を ±2(下限 6・上限 40) |
| `R` | 盤面リセット(決定的に再生成) |

- オート ON の間、**0.25s 間隔で盤面を左上から走査し、最初に見つかった「マッチを生む隣接スワップ」**を
  自動実行。有効手が無ければ盤面を決定的シャッフル。マウス無しでベンチが回り続ける。
- 無効手(スワップしてもマッチしない)は元に戻す(スワップ往復のトゥイーン)。

## 実装メモ

### 状態機械(IDLE / SWAP / CLEAR / FALL)

- **IDLE**: 入力/オートソルバ待ち。オート ON なら 0.25s 間隔で `autoStep()`。
- **SWAP**: 選択2セルを `swapGrid()` で論理交換し、2スプライトを互いの座標へトゥイーン(0.15s)。
  完了時に `swapWouldMatch()` 相当でマッチ有無を判定し、無ければ往復トゥイーンで**元に戻す**。
- **CLEAR**: `findMatches()` の結果(消去セル集合)を各セルごとに縮小+フェード トゥイーン(0.2s)。
  `Score += 消去数 × 10 × 連鎖段数`。全消滅完了後に落下へ。
- **FALL**: 列ごとに下から詰め、空いた上部を上端外から決定的補充。**1宝石ごとに**落下トゥイーン
  (距離比例 0.2s/セル)。全落下完了後に再度 `findMatches()` で新マッチがあれば CLEAR へ戻り連鎖
  (`chain++`)、無ければ IDLE。
- 連鎖の各段は**トゥイーンの `onComplete` を数えて**次段へ進む(残数カウンタで全完了を検知)。

### マッチ判定(O(N²) 全走査)

- 横ラン・縦ランをそれぞれ 1 パスずつ走査(計 2N² 程度)。連続 3 以上の区間をすべて `Set` に集約。
- スワップ妥当性は盤面全走査ではなく `localMatchAt()`(交換した2セル周辺の横/縦連結だけ)で軽量判定。
- オートソルバの探索は**毎セルで右隣・下隣の仮スワップ**を試すため最悪 O(N²) の `localMatchAt` 呼び出し。
  N が大きいほどここが支配的になる。

### Phaser Tweens(このエンジンの比較対象機構)

- **すべてのアニメ(スワップ/消滅/落下)を `this.tweens.add({...})` で実装**。落下/消滅は仕様どおり
  「1列まとめて」ではなく**セル単位**でトゥイーンを張り、同時進行トゥイーン数を稼ぐ。
  大連鎖時には**数百本**のトゥイーンが同時に走る。
- 進行中トゥイーン数は `this.tweens.getTweens().length` を HUD に表示(`Active tweens`)。
  自前カウンタ(`activeTweenCount`)も併用して整合を取っている。
- Phaser の TweenManager は内部で全トゥイーンを 1 リストで毎フレーム更新する。`ease`(`Quad.easeInOut`/
  `Back.easeIn`/`Quad.easeIn`)もプリセット文字列で指定でき、`onComplete` で破棄/次段遷移を素直に書ける。
- N/リセット変更時は `this.tweens.killAll()` で取り残しトゥイーンを一掃してから再生成する。

### 決定的生成(シード固定 / Math.random 不使用)

- 盤面・補充・シャッフルはすべて `mulberry32` 製の決定的疑似乱数で生成。
- **初期盤面**(seed にリセット回数と N を混合): 各セルで「左2連・上2連と同種を避ける」貪欲生成で
  **初期マッチ無し**を保証(保険として生成後に全走査し、残れば最大 8 回振り直す。実測 0 回で達成)。
- **補充**(seed に手数・連鎖段数を混合): 列ごとに上端外から新規宝石を決定的に補充。
- **シャッフル**: 有効手が無いとき、種別配列を Fisher-Yates(PRNG)で再配置。直後にマッチがあれば連鎖解決。

### アセット / フォールバック

- `../assets/`(= `10/assets/`)から SPEC のファイル名(`gem_red/blue/green/yellow/purple/white.png`,
  `bg_board.png`)で読込。`BootScene` で `loaderror` を捕捉し、欠落キーは `Graphics.generateTexture` で
  焼いた図形にフォールバック(宝石=色付き角丸矩形+ハイライト / 盤面背景=濃紺グリッド)。
- フォールバック宝石は 64px 固定で焼き、表示時に**セルサイズに合わせて拡縮**(`gemScale()`)。
  N を変えるとセルサイズが変わるため、拡縮率も都度再計算する。**画像が 1 枚も無くても必ず起動する。**

### N を上げたときの破綻点

- 負荷は2軸: ①マッチ走査 O(N²) と ②同時トゥイーン数(大連鎖で爆発的に増える)。
- N=12(144セル)〜20(400セル)は余裕。N=30(900セル)あたりから大連鎖時にトゥイーン数が
  数百に達し、TweenManager の更新とスプライト描画で FPS が揺れ始める。
- N=40(1600セル)では、盤面全消し級の連鎖で同時トゥイーンが**数百〜千本**規模になり、
  オート探索の `localMatchAt` 全走査コストも重なって、低スペック環境では 60fps を割り込む。
  これが Phaser Tweens(全トゥイーン 1 リスト毎フレーム更新)の実用上の破綻点。

## Codex 生成所感(大量トゥイーン × ロジック主体の実装しやすさ)

- **Phaser Tweens の使い勝手**: `this.tweens.add({ targets, x, y, duration, ease, onComplete })` の
  宣言的 API が単純で、スワップ/消滅/落下のどれも数行で書け、`onComplete` での破棄や次状態遷移が
  自然に繋がる。セル単位で大量に張っても API は同じで、本テーマの「同時トゥイーン数を稼ぐ」目的に素直。
- **状態機械との相性**: トゥイーン完了を `onComplete` の残数カウントで集約し、CLEAR→FALL→再判定の
  連鎖ループを駆動した。ポーリングではなくコールバック駆動で書けるため、状態遷移が見通し良くまとまった。
- **killAll() の安心感**: N 変更やリセットで取り残しトゥイーンを一掃できる `killAll()` が
  あるおかげで、盤面再生成時の整合(座標ずれ/破棄漏れ)を気にせず作れた。
- **ロジック側は素の JS**: マッチ走査・連鎖・補充はエンジン非依存の純粋ロジックで、Phaser は
  「座標を持つ Image にトゥイーンを張る」だけ。比較対象が描画でなくトゥイーン機構になる本テーマの趣旨に合致。
- 総じて Phaser 4 は「単純描画 + 大量トゥイーン + 状態機械」を宣言的に短く書け、
  API 名(tweens.add / getTweens / killAll / generateTexture)も安定していて迷いの少ない実装になった。
