# テーマ10 マッチ3パズル ― PixiJS v8 版

Bejeweled / ツムツム風のマッチ3パズルを **PixiJS v8** で実装。NxN 盤面の宝石を
隣と入れ替え、3つ以上揃うと消え、上から落ちて補充され連鎖する。
**描画は軽いが、ロジック（マッチ判定・連鎖・状態管理）と大量の同時トゥイーン
（落下/消滅アニメ）が主役**の性能比較用デモ。盤面サイズ N（＝セル数）を上げるほど、
**同時に動くトゥイーン数**と**マッチ走査コスト O(N²)** が増える。

## 起動方法

ローカルにHTTPサーバを立ててブラウザで開く（`PIXI.Assets.load` が `file://` では CORS で失敗するため）。

```bash
# このフォルダ(10/PixiJS/)の1つ上(10/)で実行すると ../assets/ も解決できる
cd c:/work/claude/local-works/web-game/10
python -m http.server 8000
# → http://localhost:8000/PixiJS/ を開く
```

`../assets/` に画像が無くても **Graphics 図形フォールバック** で必ず起動する。
（むしろ既定はアセット無しでの起動を前提に作ってある。）

## 使用バージョン

- PixiJS **v8**（CDN: `https://cdn.jsdelivr.net/npm/pixi.js@8/dist/pixi.min.js`）
- 追加ライブラリなし。`index.html` + `game.js` の2ファイル構成。
- **v8 注意**: 初期化は `const app = new PIXI.Application(); await app.init({...});` の2段階。
  キャンバスは `app.view` ではなく **`app.canvas`**。Graphics は新チェーンAPI
  （`gr.roundRect(...).fill(color)` / `.stroke({width,color})`）。フォールバックは
  `app.renderer.generateTexture({ target: graphics })` でテクスチャ化して Sprite で再利用。
- **トゥイーン機構は未使用＝自前実装**（後述）。

## 操作

| 入力 | 動作 |
|---|---|
| クリック2回 | 隣接2セルを選択して手動スワップ（隣接のみ有効。マッチしなければ自動で元に戻る） |
| `Space` | オートプレイ切替（初期 **ON**） |
| `+` / `-`（テンキー可） | 盤面サイズ N を ±2 増減（初期12・下限6・上限40） |
| `R` | 盤面リセット（決定的に再生成。スコア・手数もリセット） |

## 仕様数値（全エンジン共通）

- キャンバス **960×540**（固定）、盤面は中央の正方領域。
- 盤面 **N×N**（初期 **12**）。セル1辺 = `floor(min(520/N, 56))` px で動的決定。
- 宝石 **6種**（色/形）。初期盤面・補充・シャッフルはすべて固定シード `mulberry32` で
  **決定的生成**（`Math.random` 不使用）。初期盤面は**生成直後に既存マッチが無い**状態。
- 状態機械 **IDLE / SWAP / CLEAR / FALL**:
  - **SWAP**: 選んだ2セルを **0.15s** トゥイーンで入替。入替後にマッチが無ければ**元に戻す**（無効手）。
  - **CLEAR**: マッチしたセルを **0.2s** で縮小+フェード消滅。`Score += 消去数 × 10 × 連鎖係数`。
  - **FALL**: 上の宝石を空きへ落下（**0.2s/セル**・距離比例・加速付き）。上端から決定的に補充。
  - 落下後に新たなマッチがあれば CLEAR へ戻り**連鎖**（`Chain++`、連鎖係数 = 連鎖段数）。無ければ IDLE。
- マッチ判定: 横/縦に**同種3連以上**を全消し。判定は毎回**盤面全走査 O(N²)**。
- オートプレイ: **0.25s 間隔**で「マッチを生む隣接スワップ」を**左上から走査して最初の有効手**を実行。
  有効手が無ければ**決定的シャッフル**。マウス無しでベンチが回り続ける。

## HUD（HTMLオーバーレイ `#hud`・約120msごと更新）

仕様の必須項目をそのまま表示する。

- `FPS`（直近60フレームの移動平均）
- `Board`（N x N = セル数）
- `Active tweens`（**自作トゥイーンマネージャが現在保持しているトゥイーン本数**）
- `State`（IDLE / SWAP / CLEAR / FALL）/ `Chain`（現在の連鎖段数）
- `Score` / `Moves`（実行した手数）/ `Auto`（ON/OFF）
- 画面下に操作ヒント行。

HUD はキャンバス外の HTML `<pre>` 要素で描画。Pixi の描画負荷と切り離して計測できる。

## 実装メモ

### ★ 手書きトゥイーンマネージャ（本テーマの比較対象）

**PixiJS には Phaser の `this.tweens.add(...)` のような組込みトゥイーン機構が無い。**
そこで `makeTweenManager()` で最小のトゥイーンマネージャを自作している。

- 1トゥイーン = `{ target, props:{key:{from,to}}, dur, t, ease, onComplete }`。
  `add()` 時に `target` の現在値を `from` として記録する。
- 毎 Ticker フレーム `update(dt)` で全件の `t += dt` を進め、`k = t/dur` を
  イージング（線形 / easeOutQuad / easeInQuad）に通して `target[key] = from + (to-from)*e` を**直接書き戻す**。
- `k >= 1` で完了。配列から除去し `onComplete` を**補間更新が全部済んだ後にまとめて呼ぶ**
  （コールバック内で次のトゥイーンを `add` しても安全なように）。
- **状態遷移はこの `onComplete` チェーンで駆動**する:
  SWAP完了→マッチ判定→`startClear`→CLEAR完了→`startFall`→FALL完了→マッチ判定→連鎖 or IDLE。
- HUD の `Active tweens` は `tweenMgr.count()`。**落下/消滅が同時多発するほど増える**のが負荷の核。

> Phaser版は `this.tweens.add({ ... onComplete })` をエンジン任せにできるが、Pixi では
> 「進行管理・イージング・完了コールバック・配列のライフサイクル」をすべて自前で書く。
> 逆に言えば**トゥイーン更新コストが完全に自分のコードに乗る**ため、エンジン差が素直に出る。

### 落下はセル単位でトゥイーンを張る（同時数を稼ぐ）

仕様の「1列まとめてではなくセル単位」に従い、**移動する宝石1つにつき1トゥイーン**を発行する。
さらに上端からの補充宝石も**1つ1つ別トゥイーン**で画面外上方から落とす。よって CLEAR→FALL の
1サイクルで「消えた数 ＋ 落ちた数 ＋ 補充数」ぶんのトゥイーンが同時に走り、N が大きいほど
`Active tweens` が跳ね上がる。落下時間は**落下距離に比例**（`0.2s × 距離セル数`）＋ `easeInQuad` で加速感。

### 状態機械と連鎖

- `grid[r][c]` = 宝石種別(0..5) または -1(空き)。**論理座標(r,c)と描画座標(x,y)を分離**し、
  トゥイーンは描画座標だけを動かす。論理スワップ/詰めはトゥイーン完了時に確定する。
- スワップは2セル分のトゥイーンが両方終わってから論理確定→マッチ判定。マッチが無ければ
  もう一度逆スワップ（`revert`）して IDLE。
- CLEAR は `chain++` して `Score += 消去数 × 10 × chain`。FALL 後にまたマッチがあれば
  `startClear` を再入して連鎖を伸ばす。

### マッチ判定 O(N²)

`findMatches()` は横（各行）・縦（各列）を1パスずつ走査し、同種3連以上のランをマーク集合に入れる。
計算量は O(N²)。オートプレイの有効手探索は各隣接ペアで「一時入替→`findMatches`→戻す」を行うため
**O(N⁴) 相当**になり、ここが N を上げたときの最大のCPU負荷になる（下記「破綻点」）。

### 決定的生成

- 初期盤面: 左上から順に、その時点で横/縦3連を作らない種別を `mulberry32` の出目順に選んで埋める
  （`wouldMatchAt` で左2つ・上2つだけ見れば十分）。**生成直後に既存マッチ無し**を保証。
- 補充: `(盤面シード ^ N ^ moves ^ chain)` から派生したシードで列ごとに種別を決定。
- シャッフル: 全セルを Fisher–Yates（決定的）で並べ替え、即マッチになるセルだけ種別を1つずらす。

### スプライトプール

宝石スプライトは `spritePool` で使い回す。盤面リサイズ（`+`/`-`）やリセットでも
`getSprite`/`releaseSprite` で**生成は初回ぶんだけ**。N=40（1600セル）でも追加生成は最初の1回に収まる。

### フォールバック描画

各アセットを `PIXI.Assets.load` で試行し、失敗時は `generateTexture` 生成の図形テクスチャに差し替え。
宝石は種別ごとに色（赤/青/緑/黄/紫/白）を変え、偶数種別=角丸四角・奇数種別=丸で**色に頼らず識別**できる。
盤面背景は暗い角丸タイル、キャンバス背景は濃紺。セルサイズが動的に変わるため、画像（64px固定）は
セルサイズに合わせて Sprite を拡縮して描く。

## N を上げたときの破綻点

- **描画は終始軽い**（単純な Sprite を最大1600個動かすだけ・ドローコールも少ない）。
- 効いてくるのは **CPU 側**:
  1. **オートプレイの有効手探索が O(N⁴) 相当**（全隣接ペア × `findMatches` O(N²)）。
     N=12 で約 12²×2 ペア × 144 ≈ 4万判定/手だが、N=40 では約 40²×2 ペア × 1600 ≈ 1000万判定/手に達し、
     0.25s ごとのオート手探索が**1フレーム内のスパイク**になりやすい。実測では概ね **N≈30 を超えたあたり**から
     オート手探索のフレームでカクつきが出始める（環境依存）。
  2. **同時トゥイーン数**: 大規模連鎖時に数百〜千本が同時進行。`tweenMgr.update` は線形なので
     本数に比例。これは描画より軽いが、O(N⁴) 探索と重なるフレームで合算負荷になる。
- まとめ: PixiJS版の破綻点は**描画ではなく「自前ロジック（特にオート手探索）＋自前トゥイーン更新」の
  CPU コスト**。これは描画/物理が重い既存テーマと**対照的な負荷プロファイル**で、テーマ10の狙いどおり。

## v8 の落とし穴（マッチ3を組む観点）

- **初期化の罠**: `new PIXI.Application(opts)` は無視され `app.canvas` が生えない。
  必ず `new PIXI.Application()` → `await app.init(opts)` の2段階。`app.view` も `app.canvas` に改名。
- **トゥイーンは無い**: v8 にアニメ補間の組込みは無い。`Ticker` の `deltaMS` を使った自前補間が必須。
  本実装はそれを比較対象として明示的に手書きしている。
- **新 Graphics API**: `g.beginFill/drawRect` 系は廃止。`g.roundRect(...).fill(color)` /
  `.stroke({ width, color, alpha })` のチェーン記法。フォールバックテクスチャは
  `app.renderer.generateTexture({ target, width, height, resolution })`。
- **ポインタ座標の換算**: キャンバスは CSS で等倍縮小表示するため、`getBoundingClientRect()` で
  実表示サイズを取り `VIEW_W/rect.width` 倍して論理座標へ戻してからセルを引く。

## Codex 生成所感

- v8 は「描画は超軽量・ロジックとアニメは完全自前」という分担が明快で、**トゥイーン機構を持つ
  Phaser/Godot との対比**がはっきり出る題材だった。Phaser なら `tweens.add` 一発の所を、Pixi では
  進行・イージング・完了コールバック・配列ライフサイクルまで自分で設計する必要がある。
- 反面、`onComplete` チェーンで状態機械を駆動する設計は素直で、SWAP→CLEAR→FALL→連鎖の流れを
  トゥイーン完了イベントだけで組めた。論理座標と描画座標を分離したのも見通しを良くした。
- 負荷の主役が**描画ではなく自前CPUコスト**（特にオート手探索 O(N⁴)）に出るのは想定どおりで、
  N を上げたときの破綻が「ドローコール」ではなく「JS の走査」に現れる点が、他テーマと逆で面白い。
