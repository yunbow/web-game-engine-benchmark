# テーマ10 マッチ3パズル(ロジック主体・軽描画 × 大量トゥイーン) ― LittleJS 版

ぷよ/ツムツム/Bejeweled 風のマッチ3パズルを **LittleJS** で実装したもの。
N×N 盤面の宝石を隣と入れ替え、3つ以上揃うと消え、上から落ちて補充され連鎖する。
描画自体は単色の矩形/円と軽いが、**マッチ判定(O(N²) 全走査)・連鎖・盤面状態管理のロジック**と、
**落下/消滅を表す大量の同時トゥイーン**が主役で、既存テーマ(描画/物理が重い)とは対照的な負荷を測る。

LittleJS には Phaser Tweens や Godot Tween のような高レベルなトゥイーン機構が無いため、
**小さな手動トゥイーンマネージャ**を自前実装している。この「手動補間方式そのもの」が
他エンジン(Phaser Tweens / Pixi 手動 or ticker / Babylon Animation / Godot Tween)との比較対象。

## 起動方法

CDN 読込のため `file://` で直接開いても動くが、`../assets/` の相対参照と
ブラウザのキャッシュ挙動を安定させるため簡易HTTPサーバ経由を推奨。

```bash
# このフォルダ (10/LittleJS) で
python -m http.server 8000
# → ブラウザで http://localhost:8000/ を開く
```

`../assets/` フォルダは**空でも起動する**(画像欠落時は単色図形フォールバック)。
SPEC のアセット(`gem_red.png` 等)を `../assets/` に置けば自動で使用される。

## 使用バージョン / CDN

- **LittleJS**(`@latest` 解決時点 / classic global build)
- 主URL: `https://unpkg.com/littlejsengine`
  失敗時フォールバック: `https://cdn.jsdelivr.net/npm/littlejsengine/dist/littlejs.min.js`
- 素の unpkg は classic global build(グローバル `engineInit` 等を生やす版)を解決するため
  `<script>` 直読みできる。読込失敗時のみ `error` イベントを捕捉して jsDelivr ミラーへ切替。

## 操作

- **クリック2回**(隣接2セルを選択)で手動スワップ。隣接(上下左右1マス)のみ有効。
  非隣接を選ぶと選択し直し、同じセルを再クリックで選択解除。
- **Space**: オートプレイ ON/OFF トグル(初期 **ON**)。ON の間、0.25s 間隔で決定的に
  「マッチを生む隣接スワップ」を左上から探索して自動実行。有効手が無ければ決定的シャッフル。
- **+ / -**(メイン行 or テンキー): 盤面サイズ N を **±2**(下限 6・上限 40)。性能比較の主軸。
- **R**: 盤面リセット(決定的に再生成・スコア/手数も初期化)。

## 仕様数値(全エンジン共通)

| 項目 | 値 |
|---|---|
| キャンバス | 960 x 540 固定 |
| 盤面 | N x N(N 初期 12, ±2, 6..40)・中央配置 |
| セル1辺 | `floor(min(520 / N, 56))` px |
| 宝石種別 | 6(赤/青/緑/黄/紫/白) |
| スワップ | 0.15s(マッチ無しなら元に戻す無効手) |
| 消滅(CLEAR) | 0.2s 縮小+フェード / Score += 消去数 × 10 × 連鎖係数 |
| 落下(FALL) | 基準 0.2s/セル(落下距離比例)+ 上端から決定的補充 |
| 連鎖 | 落下後に新マッチがあれば CLEAR へ戻り chain++(1段目=1, 2段目=2 …) |
| マッチ | 横/縦 同種3連以上を全消し・毎回 O(N²) 全走査 |
| オート間隔 | 0.25s |

## HUD(常時表示・左上, HTML `#hud` overlay)

- `FPS`: 実測の移動平均(エンジン `frameRate` を指数移動平均で平滑化)。
- `Board`: N x N = セル数。
- `Active tweens`: 現在進行中のトゥイーン本数(手動トゥイーンマネージャの配列長そのもの)。
- `State`: IDLE / SWAP / CLEAR / FALL。
- `Chain`: 現在の連鎖段数。
- `Score` / `Moves`(実行した有効手数) / `Auto`(ON/OFF)。
- `[sprites]` / `[shapes fallback]`: 画像使用かフォールバックかを明示。
- 画面下に操作ヒント行。

## 実装メモ

### 手動トゥイーンマネージャ(本テーマの比較対象)
LittleJS は薄い描画/入力 API のみで、トゥイーン機構を持たない。そこで自前で実装した。

- 各トゥイーンは `{ obj, props:{key:[from,to]}, t, dur, ease, onDone }` のレコード。
  `tweens` 配列に積み、`gameUpdate` の `updateTweens(timeDelta)` で **実時間(秒)** 基準に
  `t` を進め、`ease(t/dur)` で `obj[key]` を線形補間する。完了したら配列から除去し `onDone` を呼ぶ。
- イージングは `easeOutQuad / easeInQuad / easeInOutQuad` を用意(スワップは inOut, 落下/消滅は in)。
- HUD の `Active tweens` は **この配列長そのもの**。エンジン差を測る一次指標になる。
- **落下は1列まとめてではなく「セル単位」でトゥイーンを張る**(SPEC: 同時トゥイーン数が負荷の核)。
  1回の FALL で動いた宝石+補充宝石それぞれに1本ずつ。N が大きく連鎖が深いほど同時本数が増える。
- 宝石オブジェクトは `{ type, dx, dy, scale, alpha }`。トゥイーンは描画**オフセット** `dx/dy`、
  消滅は `scale`/`alpha` を動かす。論理盤面(`board[row][col]`)は**先に**確定させ、
  見た目だけ補間で追従させる方式(LittleJS には可逆トゥイーンが無いため、スワップは
  「論理交換 → 相手位置から元位置へオフセットを戻す」で表現)。

### 状態機械(IDLE / SWAP / CLEAR / FALL)
- **IDLE**: 入力/オート待ち。オート ON なら 0.25s ごとに 1 手。手動クリックもここで受理。
- **SWAP**: 2セルを論理交換し、`dx/dy` を 0.15s で 0 へ補間。完了時 `findMatches()` で判定。
  マッチ無し(無効手)なら `revert=true` で逆スワップを張り、戻し完了で IDLE。有効手のみ `Moves++`。
- **CLEAR**: マッチセルを `scale:1→0, alpha:1→0` で 0.2s 消滅。`Score += 消去数 × 10 × chain`。
  全消滅トゥイーン完了で論理 `null` 化 → FALL。
- **FALL**: 各列を下から詰め(コンパクション)、上端の空きを決定的補充。動いた各宝石に
  落下距離比例(`0.2s × セル数`)の `dy→0` トゥイーン。全着地完了で再度 `findMatches()`。
  マッチがあれば CLEAR へ戻り `chain++`(連鎖)、無ければ IDLE。

各状態遷移は**トゥイーンの `onDone` カウンタ**で駆動する(「N本中 N本完了したら次へ」)。
タイマや固定待ちフレームを使わないので、N やデルタタイムが変わっても破綻しない。

### マッチ判定(O(N²) 全走査)
`findMatches()` は横走査(各行で連続同型ランを数える)+ 縦走査(各列で同様)を行い、
3連以上のセルを `mark[row][col]=true` に立てて返す。盤面全体を毎回走査するので **O(N²)**。
オートプレイの有効手探索 `findFirstValidMove()` は各セルで「右/下と仮交換 → `findMatches` →
戻す」を試すため、最悪 O(N²) × O(N²) = **O(N⁴)** になる(N=40 でここが効いてくる)。

### 座標系 / Y軸
LittleJS は **Y軸が上向き**。`cameraScale=1`(1ワールド単位 = 1px)・`cameraPos=(0,0)` とし、
盤面を**原点中心**に正方形配置する。盤面ロジックは「row=0 が一番上」という素直なグリッド添字で持ち、
Y反転は変換関数に閉じ込めた:

- `boardToWorld(col,row)`: row が増える(下の行)ほど worldY が減る → `worldY = boardTop - row*cell - cell/2`。
- `worldToBoard(wx,wy)`: マウス入力(`mousePos` は y-up ワールド座標)を同じ式系で逆変換し、クリックセルを得る。
- 落下トゥイーンの `dy` は「真上 dist セル分のオフセット(正=上)から 0 へ」。y-up なので
  上から落ちる = 正の `dy` が 0 に近づく、で素直に表現できる。

符号を一箇所(`boardToWorld`/`worldToBoard`)に閉じ込めることで、「クリック位置がずれる」
「落下が上向きに見える」事故を防いだ。

### 決定的生成 / フォールバック
- `mulberry32` ベースの決定的乱数。盤面生成・シャッフル用(`boardSeed`)と補充用(`refillSeed`)で
  独立した進行シードを持ち、`Math.random` は**不使用**。シードは N から導出するので
  「同じ N なら毎回同じ初期盤面」。
- 盤面初期化は各セルで「左2つ/上2つと同型なら型をずらす」を決定的に繰り返し、
  **生成直後に既存マッチが無い**状態にする。
- 詰み(有効手なし)時は `deterministicShuffle()`(フィッシャー-イェーツを `boardSeed` で駆動)し、
  残ったマッチは型をずらして潰す。
- 画像は `textureInfos[i].size.x > 1` で読込判定(`spriteReady`)。読込済みなら `drawTile`、
  未読込なら宝石=**色付き矩形+中央ハイライト円**、盤面=暗いセル下地、背景=濃紺の
  `drawRect`/`drawCircle` フォールバック。セルサイズが N で動的に変わるため図形も毎フレーム
  セルに合わせて拡縮する。**画像が1枚も無くても必ず起動する。**

## N を上げたときの破綻点

- **N=12(144セル)**: 同時トゥイーンは連鎖時でも一桁〜十数本程度。常時 60fps。
- **N=20〜28(400〜784セル)**: 大連鎖時に落下トゥイーンが数十〜百本規模。まだ余裕。
- **N=40(1600セル)**: 連鎖が深いと落下トゥイーンが**数百〜千本**同時に走る。
  - ボトルネックは2つ: ① 毎手の `updateTweens` の本数、② オート探索 `findFirstValidMove` の
    **O(N⁴) 仮交換ループ**。後者の方が先に効く傾向で、N=40 では 1 手の探索コストが跳ねる。
  - 描画自体は単純な矩形/円なので軽く、**ロジック+トゥイーン管理のオーバーヘッド**がそのまま
    フレーム時間に出る。これがエンジン比較の狙い(描画が軽い分、機構差が露出する)。
- LittleJS は抽象が薄く、トゥイーン配列も探索ループも完全に自前なので、どこが重いかが
  プロファイラ無しでも `Active tweens` と `State` の HUD から読み取れるのが利点。

## Codex 生成所感(高レベル機構の無いエンジンで大量トゥイーンを捌く所感・罠)

- **「トゥイーン機構が無い」ことが逆に素直**。Phaser/Godot の宣言的 Tween に慣れていると
  手書きは面倒に見えるが、`{obj, props, t, dur, ease, onDone}` の 30 行ほどのマネージャで
  必要十分だった。本数(=負荷)が配列長として丸見えなので、ベンチ用途ではむしろ好都合。
- **状態遷移を「待ちフレーム」でなく `onDone` カウンタで駆動**したのが効いた。
  「N本中 N本完了で次へ」にすると、落下距離がセルごとに違っても・N が変わっても・
  デルタタイムが揺れても破綻しない。固定待ちにすると N 依存でズレる。
- **論理を先に確定し見た目を後から補間**する設計が肝。スワップ/落下とも `board[][]` は
  即座に正しい状態にし、`dx/dy/scale/alpha` のオフセットだけトゥイーンする。これで
  マッチ判定はいつでも「確定済み盤面」に対して走らせられ、アニメ中の判定ズレが起きない。
- **Y軸上向き × グリッド**は、変換関数に閉じ込めれば素直。テーマ5(横スクロール物理)ほど
  符号が絡まないが、「マウス → セル」と「落下オフセットの向き」を `boardToWorld` と
  同じ式系で統一したのは同じ思想。
- **最初に効く負荷はトゥイーン本数より O(N⁴) のオート探索**だったのは想定外。
  描画が軽いテーマだけに、ロジックの計算量がそのまま見える。SPEC が「ロジックとトゥイーン
  管理のオーバーヘッドがエンジン差として出る」と言う通りの挙動になった。
- 総じて LittleJS は「薄い API を正しく組む」エンジン。高レベル機構が無いぶん、
  トゥイーン・状態機械・マッチ判定のコストがすべて自前コードに集約され、
  性能の所在が HUD(`Active tweens`/`State`/`FPS`)にそのまま出るのがベンチ題材として扱いやすかった。
