# テーマ10: マッチ3パズル (ロジック主体・大量トゥイーン) ― Babylon.js 版

Bejeweled / ツムツム風のマッチ3パズルを **Babylon.js** で実装したもの。
2Dゲームだが Babylon は3Dエンジンのため、**正射影(Orthographic)カメラ + SpriteManager/Sprite**
で 2D 盤面を再現している。描画は単純な丸/矩形と軽い一方、**マッチ判定 (盤面全走査) と
大量の同時トゥイーン (落下/消滅アニメ)** が負荷の主役という、既存テーマ (描画/物理が重い) とは
対照的な性能特性を測るのが主眼。**マウス操作無しでもオートソルバがベンチを回し続ける**。

## 起動方法

`../assets/` を相対パスで読むため、ローカルHTTPサーバ経由で開く (file:// でも起動はするが
画像が読めずフォールバック図形になる)。**現状アセットは未配置なので、図形フォールバックで動く**。

```bash
# このフォルダ (10/Babylon.js) の1つ上 (10/) で実行すると ../assets も配信される
cd c:/work/claude/local-works/web-game/10
python -m http.server 8000
# ブラウザで http://localhost:8000/Babylon.js/ を開く
```

または VS Code の Live Server などでも可。
画像アセットが無い環境でも **単色図形 (種別ごとの色付き円) でフォールバック**して必ず起動する。

## 使用バージョン

- Babylon.js: CDN 最新版 (`https://cdn.babylonjs.com/babylon.js`)
- 追加ビルド・依存なし。`index.html` + `game.js` の2ファイルのみ。

## 操作

| キー / 操作 | 動作 |
|---|---|
| クリック2回 | 隣接2セルを選択して手動スワップ (隣接のみ有効。手動操作するとオート OFF) |
| `Space` | オートプレイ切替 (初期 **ON**) |
| `+` / `-` | 盤面サイズ N を ±2 (下限 **6**・上限 **40**) |
| `R` | 盤面リセット (決定的に再生成) |

## 仕様準拠 (数値)

- キャンバス **960x540** 固定。盤面は中央に正方形領域で配置。
- 盤面 **N x N** (N 初期 **12**)。セル1辺 = `floor(min(520 / N, 56))` px で動的決定 → 常に中央に収まる。
- 宝石 **6 種類** (赤/青/緑/黄/紫/白)。盤面・補充は固定シードの mulberry32 で決定的生成
  (`Math.random` 不使用)。**初期生成時は既存マッチが出ないよう、各セルで禁止種を避けて配置**
  (横左2つ/縦上2つと同種になる置き方を避けて種をずらす → 生成直後マッチ無しを保証。N=6/12/40 で検証済)。
- 状態機械 **IDLE / SWAP / CLEAR / FALL**:
  - **SWAP** (~0.15s): 2セルの位置をトゥイーンで入れ替え。入れ替え後にマッチが無ければ**元に戻す** (無効手)。
  - **CLEAR** (~0.2s): マッチセルを縮小+フェードで消去。**Score += 消去数 × 10 × 連鎖係数** (連鎖係数 = 連鎖段数)。
  - **FALL** (~0.2s/セル, 落下距離比例): 上の宝石が空きへ落下 + 上端から決定的補充。**セル単位**でトゥイーンを張る。
  - 落下後に新マッチがあれば CLEAR へ戻り**連鎖** (chain++)。無ければ IDLE。
- マッチ判定: 横/縦に**同種3つ以上連続**を全消し。判定は状態遷移時に盤面全走査 **O(N²)**。
- **オートプレイ** (初期 ON): IDLE 中 **0.25s 間隔**で、盤面を左上から走査して**最初に見つかった有効手**
  (隣接スワップでマッチが生まれる手) を打つ。有効手が無ければ盤面を**決定的にシャッフル**。
- パズルのため勝敗は無い (無限プレイ)。`Moves` は確定した手数。

## HUD (HTMLオーバーレイ, 約0.1s更新)

- `FPS` (指数移動平均)
- `Board` (N x N = セル数) / `Active tweens` (自前マネージャの進行中トゥイーン数)
- `State` (IDLE / SWAP / CLEAR / FALL) / `Chain` (現在の連鎖段数)
- `Score` / `Moves` (実行した手数) / `Auto` (ON / OFF)
- 描画モード: `Sprite(tex)` (6種すべて読込成功) / `Disc(fallback)` (1つでも欠落 → 全フォールバック)

## 実装メモ ― トゥイーン機構の選択 (自前 vs Babylon.Animation)

- **採用: 自前の軽量トゥイーンマネージャ**。`tweens` 配列に `{ obj, props:[{key,from,to}], t, dur, ease, onDone }`
  を積み、毎フレーム `update(dt)` で線形に時間を進めて `obj[key]` を直接書き換える (ease は easeOutQuad / easeInQuad)。
  完了トゥイーンは配列から除去し、`Active tweens` に件数をそのまま出す。落下/消滅は**1セル1トゥイーン**で張るため、
  N=40 では1回の連鎖で**数百のトゥイーンが同時進行**する。この規模の短命トゥイーンでは、
  生成/破棄が安く・件数を正確に数えられる配列管理が最も**計測しやすく軽い**と判断した。
- **idiomatic な代替: `BABYLON.Animation`**。Babylon には `BABYLON.Animation` + `scene.beginAnimation()` /
  `Animatable` という宣言的トゥイーン機構があり、キーフレーム・補間種別 (`ANIMATIONTYPE_FLOAT` 等)・
  イージング (`BABYLON.QuadraticEase` 等) を持つ。1個1個のアニメをエンジンに登録して回す設計で読みやすい反面、
  Sprite (SpriteManager 配下) は Animation のターゲットにしづらく、毎セルに `Animatable` を作る/破棄するオーバーヘッドと
  「同時アクティブ数」をエンジン横断で正確に数える手間が大量短命トゥイーンでは効いてくる。本ベンチの主旨
  (同時トゥイーン数 = 負荷) の計測には自前管理が素直なため、こちらを採用し Animation 版は本 README の記載に留めた。
- **状態機械と onDone 連鎖**: 状態遷移 (SWAP→CLEAR→FALL→…) はトゥイーンの `onDone` で駆動する。
  各操作で残トゥイーン数 `remaining` をカウントし、最後の1本が完了したら次状態へ進む。
  `updateTweens` は**そのフレーム開始時点の件数だけ**を進め、onDone 内で積まれた次状態のトゥイーンは
  末尾に積んで次フレームから進めることで、同フレーム二重進行を防いでいる。
- **マッチ判定 O(N²)**: 横走査・縦走査を各1パスずつ行い (各行/各列を端まで run-length で見る)、3連以上を
  消去マスク `Uint8Array` に立てる。毎フレームではなく**状態遷移時のみ**呼ぶ。オートソルバの有効手探索は
  各隣接ペアで仮スワップ→`findMatches`→戻す、を左上から行うため、N が大きいと O(N⁴) 規模になり主要コストになる。
- **正射影2D**: `orthoLeft/Right/Top/Bottom` を `0..960 / 0..540` に設定し (`orthoTop < orthoBottom` で **y 下向き**)、
  画面座標 = ワールド px を 1:1 対応させた。盤面ロジックは終始「左上原点・+y 下方向」の px 空間で完結し、座標変換不要。
  セル `(r,c)` 中心は `originX + c*cell + cell/2` / `originY + r*cell + cell/2`。落下補充は盤面上端より上 (負の row) から
  落とすことで、画面外から降ってくる演出になる。
- **スプライトプール / フォールバック**: 宝石は種別ごとに `SpriteManager` を1つ持ち、その配下に Sprite を割り当てる。
  テクスチャが1つでも欠ければ全種を**単色 Disc (円)** のクローンに切り替える (`createGemVisual` が Sprite/Mesh を
  統一ラッパ `{setPos,setSize,setAlpha,setVisible}` で隠蔽)。Disc はマテリアル共有のため、フェードは
  `material.alpha` でなく `mesh.visibility` (per-instance) で行う。盤面背景は暗い矩形、選択中セルは半透明の白枠。
  セルサイズは N で動的に変わるので、画像/図形は毎フレーム `setSize(cell*scale*0.92)` で拡縮して描く。
- **z 重なり順**: 盤面背景 = 5 (奥)、宝石 = 0、選択ハイライト = -2 (手前) を手動割当て。

## N を上げたときの破綻点

- **N=12 (144セル)**: 連鎖時でも同時トゥイーンは数十程度。60fps 余裕。
- **N=20〜28**: 大連鎖で同時トゥイーンが数百に達するが、自前マネージャの更新は単純配列ループのため描画より軽い。
- **N=40 (1600セル)**: 1連鎖で**数百〜千のトゥイーン**が同時進行。ボトルネックはトゥイーン更新ではなく
  むしろ **(a) 毎フレーム全セルの `setSize`/`setPos` 反映** と **(b) オートソルバの有効手探索 O(N⁴ 相当)**。
  Sprite モードなら SpriteManager のインスタンシングが効いて描画は耐えるが、フォールバック Disc モードは
  1600 個の個別 Mesh となり描画コールが増えるため、N=40 ではフォールバック時の方が先に fps が落ちやすい。
  実用上 N=40 でも IDLE 中は問題なく回り、負荷のピークは「大連鎖の落下フレーム」と「ソルバ探索フレーム」に出る。

## Codex 生成所感 ― Babylon でマッチ3を書く

- **3Dエンジンで「描画が軽くロジックが主役」のゲームを作る逆転構図**が面白い。テーマ5 (横スクロール) では
  カリングや自前物理が主コストだったが、本テーマは描画は丸を並べるだけで、**状態機械・マッチ判定・トゥイーン管理**が
  コードと負荷の中心になる。Babylon の SpriteManager は「同種大量スプライト」に強く、宝石を種別ごとに1 manager へ
  まとめる設計と相性が良い。
- **トゥイーンは自前が素直**。Babylon.Animation は宣言的で読みやすいが、Sprite ターゲット・大量短命・件数計測の
  3点で本ベンチには噛み合わず、`dt` 駆動の配列トゥイーンに落ち着いた。状態遷移を onDone カウンタで駆動する設計は、
  「最後の落下が終わったら次の連鎖判定」という流れをそのまま表現できて分かりやすい。
- **決定的性の担保が地味に効く**。盤面生成 (初期マッチ回避)・補充・シャッフル・ソルバ探索のすべてを
  mulberry32 固定シードで決定的にしたことで、N を変えても R リセットしても同じ盤面・同じ手順が再現でき、
  性能比較のベンチとして揺らがない。`Math.random` を一切使わないのが前提。
- 総じて、Babylon は2Dマッチ3に対して「正射影 px 等倍 + SpriteManager」で素直に土台を組めるが、
  トゥイーン/状態機械/マッチ判定といった**ゲームロジック部分は結局自前**で、エンジン差はそのロジックと
  描画反映のオーバーヘッドに出る、というのが本テーマの性能比較の肝になる。
