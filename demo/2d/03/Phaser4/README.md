# テーマ3 トップダウンRPG探索 ― Phaser 4 版

見下ろし型RPGの探索パート。100x100 タイル(3200x3200px)のマップをカメラ追従でスクロールし、
**可視範囲のタイルのみ描画(カリング)** して大規模タイルマップの描画効率を測る性能比較用サンプル。

## 起動方法

ブラウザのセキュリティ制約(file:// での CORS / アセット読込)を避けるため、
簡易 HTTP サーバ経由での起動を推奨します。

```bash
# このフォルダ (3/Phaser4/) で
python -m http.server 8000
# → http://localhost:8000/ をブラウザで開く
```

または任意の静的サーバ(`npx serve` など)でも可。
アセット画像が無くてもフォールバック描画で必ず起動します。

## バージョン

- Phaser **4.1.0** (CDN: `https://cdn.jsdelivr.net/npm/phaser@4/dist/phaser.min.js`)
- 追加ライブラリなし。`index.html` + `game.js` のみ。

## 操作

| キー | 動作 |
|---|---|
| WASD / 矢印 | 4方向移動 (160 px/s) |
| Shift | ダッシュ (2倍速) |
| `+` / `-` | エンティティ(NPC+敵)を 10 体ずつ増減 |

- 壁(3) / 水(2) / 木(4) で衝突。軸別解決で壁ずり移動。
- スライムに接触するとプレイヤーがノックバック。

## 実装メモ

- **マップ生成**: `mulberry32` による決定的疑似乱数(seed 固定 `0xC0B0`)。
  草ベースに水たまり(楕円クラスタ)・道の幹線(等間隔グリッド)・木(まばら)・壁の塊・外周壁を生成。
  中央 5x5 をスポーン用に草地確保。種別 0=草 1=道 2=水 3=壁 4=木。
- **カリング(性能の核)**: カメラの `worldView` から可視タイル範囲 (x0..x1, y0..y1) を毎フレーム算出。
  範囲が前フレームから変化したときだけ Blitter を再構築し、可視タイルぶんの Bob のみ生成。
  描画タイル数を HUD `Tiles drawn` に表示。可視範囲(約 31x18 + 余白)ぶんのみ描画するため、
  マップが 100x100=1万タイルでも常に 600 タイル前後しか描かない。
- **描画方式**: Phaser の **Blitter** を地形種別(草/道/水/壁)ごとに 1 枚ずつ用意。
  Blitter は単一テクスチャに対し多数の Bob を高速バッチ描画できるためタイル描画に好適。
  木は 32x48 で足元基準に別 Blitter、下に草タイルも敷く。
- **アセット**: `../assets/`(= `3/assets/`)から SPEC のファイル名で読込。
  読込失敗 (`loaderror`) を記録し、欠落キーは `Graphics.generateTexture` で生成した
  単色/図形テクスチャにフォールバック(草=緑 / 道=茶 / 水=青 / 壁=灰 / 自機=白 / NPC=黄 / slime=緑丸)。
- **HUD**: FPS(移動平均) / Tiles drawn / Entities(NPC+敵+木) / 自機タイル座標 (x,y) を常時表示。
  `setScrollFactor(0)` でカメラ非追従に固定。
- **エンティティ**: 初期 NPC+敵=60 体。スライム(約6割)は一定間隔でランダムに向きを変える簡易徘徊、
  壁衝突で反転。`+`/`-` で動的に増減し負荷を調整可能。

## Codex 生成所感(タイルマップ / カリング実装のしやすさ)

- **タイルマップ**: Phaser には Tilemap API も存在するが、本サンプルでは「描画方式を自前で
  制御してカリング数を可視化する」目的のため **Blitter** を採用した。Blitter は
  「1 テクスチャ + 多数 Bob」というモデルが明快で、可視タイルだけ Bob を作る/clear する
  だけでカリングが完結し、実装が非常に素直だった。種別ごとにテクスチャが分かれる都合上
  Blitter を種別数だけ用意する必要がある点だけ注意。
- **カリング**: カメラの `worldView`(ワールド座標の可視矩形)がそのまま使えるため、
  可視タイル範囲の算出が 1 行レベルで書ける。範囲が変化したフレームのみ再構築する最適化も
  容易で、「描画タイル数」を HUD に出す要件と相性が良い。Phaser のカメラ追従
  (`startFollow` + バウンド設定)も 2 行で完了し、大規模スクロールの土台作りは短時間で済んだ。
- **フォールバック**: `Graphics.generateTexture` で実行時にテクスチャを焼ける仕組みが標準で
  あるため、アセット欠落時の図形フォールバックが他エンジンより書きやすい。
- 総じて Phaser 4 は「広いマップ + カメラ + 自前カリング」をシンプルに書けるエンジンで、
  Codex による生成でも API 名(blitter / worldView / generateTexture)が安定しており、
  迷いの少ない実装になった。
