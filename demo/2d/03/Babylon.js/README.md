# テーマ3: トップダウンRPG探索 ― Babylon.js 版

見下ろし型RPGの探索パートを **Babylon.js** で実装したもの。
2Dゲームだが Babylon は3Dエンジンのため、**正射影(Orthographic)カメラ + SpriteManager/Sprite** で
2Dタイルマップを再現している。「3Dエンジンで2Dタイルマップ＋カリングを書く」評価が主眼。

## 起動方法

`../assets/` を相対パスで読むため、ローカルHTTPサーバ経由で開く（file:// でも起動はするが
画像が読めずフォールバック図形になる）。

```bash
# このフォルダ (3/Babylon.js) の1つ上 (3/) で実行すると ../assets も配信される
cd c:/work/claude/github/virtual_office/research/3
python -m http.server 8000
# ブラウザで http://localhost:8000/Babylon.js/ を開く
```

または VS Code の Live Server などでも可。
画像アセットが無い環境でも **単色図形でフォールバック**して必ず起動する。

## 操作

| キー | 動作 |
|---|---|
| `WASD` / 矢印 | 4方向移動 (160 px/s) |
| `Shift` | ダッシュ (2倍速) |
| `+` / `-` | 敵/NPC 数を ±10 |

## バージョン

- Babylon.js: CDN 最新版 (`https://cdn.babylonjs.com/babylon.js`)
- 追加ビルド・依存なし。`index.html` + `game.js` の2ファイルのみ。

## 仕様準拠

- キャンバス **960x540** 固定 / タイル **32x32** / マップ **100x100**（=3200x3200px）。
- マップは決定的擬似乱数 (mulberry32, seed=1337) で生成。外周は壁。
  - `0=草, 1=道, 2=水, 3=壁, 4=木`。水/壁/木は進入不可。
- 正射影カメラが自機を追従（マップ端でクランプ）。
- **可視範囲のタイルのみ描画（カリング）**。HUD の "Tiles drawn" で実数を確認可能。
- NPC/敵スライム 初期60体。簡易ランダム徘徊。スライム接触でノックバック。
- アセットは `../assets/`（SPEC のファイル名に厳密一致）。欠落時はフォールバック
  （草=緑/道=茶/水=青/壁=灰/木=濃緑/自機=白/NPC=黄/slime=緑丸）。
- HUD（HTMLオーバーレイ）: FPS（30フレーム移動平均）/ Tiles drawn / Entities（NPC+敵+可視木）/ 自機タイル座標。

## 実装メモ

- **座標系の橋渡し**: ゲームロジックは「左上原点・+y下方向」の px 空間で持ち、
  描画時に Babylon の「中心原点・+y上方向」へ `toBabylonX/Y()` で変換する。
  WASD の上下も Babylon の y 反転に合わせている。
- **カリングの実装**: タイル種別ごとに `SpriteManager` を1つ用意し、各 manager 内に
  **スプライトプール**を持つ。毎フレーム、カメラ可視範囲のタイルだけをプールのスプライトへ
  割り当て直し（位置更新＋`isVisible`）、余ったプールは非表示にする。
  これにより毎フレームの Sprite 生成/破棄を避けつつ、描画対象を可視範囲に限定している。
- **木 (32x48)**: タイルより背が高いので底辺を合わせて少し上にオフセットし、z を手前にして重なりを表現。
- **エンティティ**: NPC/スライムも視野外は `isVisible=false` で描画から外す（簡易カリング）。
- **フォールバック**: 別途 `new Image()` で各アセット URL を試し、`onerror` で
  `DynamicTexture` に差し替え（タイルは矩形、スライムは円）。SpriteManager の cell サイズも更新。

## Codex 生成所感 ― 3Dエンジンで2Dタイルマップ/カリングを書く難しさ

- **「2Dなのに3D座標」二重メンタルモデルが最大の難所**。Babylon は中心原点・y軸上向き・
  z奥行きを持つため、タイルマップの素直な「左上原点・行列インデックス」と常に変換が要る。
  ロジック側は終始 px 空間で完結させ、描画の最後でだけ Babylon 座標へ写すと破綻しにくい。
- **カリングを自前で書く必要がある**。2D専用エンジン（Phaser/Pixi 等）ならタイルレイヤや
  ビューポートカリングが用意されているが、Babylon の Sprite は「全部出して個別に
  isVisible を切る」前提に近い。100x100=1万タイルを毎フレーム全生成すると重いので、
  **種別ごとの SpriteManager + スプライトプール**で可視分だけ使い回す設計が事実上必須になった。
  ここが「3Dエンジンで2Dタイルマップ」を書くときに一番コードが増える部分。
- **正射影カメラの ortho 範囲設定**（orthoLeft/Right/Top/Bottom）でピクセルパーフェクトな
  960x540 を作るのは、慣れれば素直。ピクセル等倍にしたいだけなら 3D の遠近を切るこの方法が確実。
- **z バトル/重なり順**は 2D タイルでは z 値の手動管理になる。地面=0、木=-1、エンティティ=-2、
  自機=-3 のように layer を z に割り当てて解決した。
- 総じて、Babylon は3D表現力が高い反面、2Dタイルマップ用途では
  「描画パイプラインとカリングを自分で組む」コストがかかる。逆に言えば、
  SpriteManager のインスタンシングは大量スプライトに強く、エンティティ数を `+` で増やしても
  破綻しにくい点は3Dエンジン由来の強み。
