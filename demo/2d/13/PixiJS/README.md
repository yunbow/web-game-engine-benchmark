# テーマ13 大量動的テキスト / UI描画 ― PixiJS v8 版

「システムログ / データダッシュボード」風のデモを **PixiJS v8** で実装。
画面いっぱいに多数のテキストラベル（`OBJ#0042 v=137` 風）が決定的に流れ、各ラベルの
数値部が毎フレーム更新される。**多数のテキストオブジェクト ＋ 動的な文字列更新
（グリフのテクスチャ再生成）**のスループットを測る性能比較用デモ。

## 起動方法

ローカルにHTTPサーバを立ててブラウザで開く（CDN 読込のため `file://` でも概ね動くが、
HTTP 起動を推奨）。

```bash
cd c:/work/claude/local-works/web-game/13
python -m http.server 8000
# → http://localhost:8000/PixiJS/ を開く
```

**アセット不要**。フォントは既定/システムフォント（`Consolas, monospace`）を使用し、
BitmapText 用のフォントは `PIXI.BitmapFont.install` でランタイム生成する（外部ビットマップ
フォント不使用）。`assets/.gitkeep` のみ。

## 使用バージョン

- PixiJS **v8**（CDN: `https://cdn.jsdelivr.net/npm/pixi.js@8/dist/pixi.min.js`）
- 追加ライブラリなし。`index.html` + `game.js` の2ファイル構成。
- **v8 注意**: 初期化は `const app = new PIXI.Application(); await app.init({...});` の2段階。
  キャンバスは `app.view` ではなく **`app.canvas`**。
- **v8 Text API**: `new PIXI.Text({ text, style })`（v7 の `new PIXI.Text(str, style)` から変更）。
  `style` は `new PIXI.TextStyle({...})` またはプレーンオブジェクト。`.text` 代入で内容更新。

## 操作

| 入力 | 動作 |
|---|---|
| `+` / `-`（テンキー可） | テキスト数を ±100 増減（初期200・下限0・上限5000） |
| `U` | 動的更新 ON/OFF（ON=毎フレーム文字列更新 / OFF=位置のみ・文字列固定） |
| `B` | テキスト機構切替（**Canvas Text** ⇄ **BitmapText**） |
| `R` | リセット（位置・速度・色・サイズを初期決定値へ。機構・更新状態は維持） |

## 仕様数値（全エンジン共通）

- キャンバス **960×540**、背景 **暗色 `#0b0d16`**、デルタタイム駆動。
- テキストアイテム数 `N`: 初期 **200**、`+/-` で **±100**（下限 **0**・上限 **5000**）。
- 各アイテムは固定シード（`mulberry32(20250613)`）で**決定的**に初期位置・速度・色・サイズを割当。
  - 水平: 端で**バウンド**（vx 反転）。垂直: 緩やかな**下方向スクロール**、下端を越えたら上へ**ラップ**。
  - 文字列は `OBJ#` + 4桁番号 + ` v=` + 0〜999 の数値（例 `OBJ#0042 v=137`、約 8〜20 文字）。
    数値 `v` は `frame` と `index` から決定的に算出され、Update=ON では**毎フレーム変化**する。
  - 色は8色パレットから、サイズは **12〜22px** から決定的割当。
- 画面右上に**複数行の統計パネル**（1つの `PIXI.Text`）を置き、**毎フレーム更新**（多行の動的
  再レイアウトも踏ませる）。FRAME / N / CHARS / MODE / UPD / FPS / TICK。

## HUD（HTMLオーバーレイ `#hud`・約120msごと更新）

- `FPS`（直近60フレームの移動平均）
- `Texts`（テキストアイテム数 現在 / 上限5000）
- `Chars`（概算総グリフ数 = N × 平均文字数14 + 統計パネル分）
- `Render`（使ったテキスト機構: `Canvas Text` / `BitmapText`）
- `Update`（`dynamic` = 毎フレーム文字列更新 / `static`）
- 操作ヒント行（`+/-=テキスト数  U=動的更新  B=機構切替  R=リセット`）

HUD はキャンバス外の HTML `<pre>` 要素で描画。Pixi のテキスト描画負荷と HUD 更新を切り離して
計測できる（統計パネルは仕様要件のため Pixi 側に別途1つ置く）。

## 使ったテキスト機構と崩れ始める点

### Canvas Text（既定 / `PIXI.Text`）

PixiJS の標準テキスト。内部で **HTMLCanvas にグリフを描画してテクスチャ化**し、その
テクスチャを Sprite として表示する。

- **`.text` を書き換えると、そのテクスチャが再生成（再ラスタライズ）される**。
  Update=ON では毎フレーム `N` 個ぶんのラスタライズ + テクスチャアップロードが走る。
  **これが本テーマの測定軸そのもの**（テキスト/グリフ描画の「既知の崖」）。
- Update=OFF（文字列固定）なら、テクスチャは初回1回だけ生成され以降キャッシュが効く。
  位置更新だけになるので、同じ `N` でも **FPS は大幅に改善**する。
- 崩れ始める点（参考・環境依存）: Update=ON では概ね **N ≈ 800〜1500** 付近から 60fps を割り、
  N が増えるほど線形以上に重くなる（ラスタライズ + アップロードがフレームを食う）。
  Update=OFF なら数千個でも比較的粘る。

### BitmapText（`B` で切替 / `PIXI.BitmapText`）

`PIXI.BitmapFont.install({ name, style, chars })` で**ランタイム生成**したフォントアトラス
（ASCII 可視範囲を内蔵フォントから1回だけ焼く）を使う。各グリフは事前生成テクスチャの
**部分矩形を引くだけ**で、`.text` を書き換えても**グリフの再ラスタライズは起きない**
（頂点バッファの組み直しのみ）。

- Update=ON でも **Canvas Text より桁違いに軽い**。同じ `N` で FPS の落ち方を直接対比できる。
- 制約: 焼いた文字集合（ASCII）外は出ない。色はグリフを白で焼いて表示時に `tint` で着色、
  サイズはベース24pxを `scale` で調整（アトラス1枚を全アイテムで共有）。
- 本デモの **B 切替の核心**: 「同じ N・同じ Update=ON」で Canvas Text → BitmapText に切り替えると、
  毎フレームのラスタライズが消えて FPS が跳ね上がる ＝ 崖の正体が**グリフのテクスチャ再生成**で
  あることが体感できる。

## 実装メモ

- **プール（破棄/生成しない）**: `PIXI.Text` と `PIXI.BitmapText` をそれぞれ最大 `N_MAX=5000` 個
  まで**遅延生成**し、`+/-` の増減では破棄せず `visible=false` で退避するだけ。毎フレームの
  生成/破棄は一切しない（v8 の Text 構築コストを増減操作の外に追い出す）。
- **論理と表示の分離**: 各アイテムの論理状態 `{x,y,vx,vy,base,color,size}` を `items[]` に持ち、
  表示は現在の機構（Canvas/Bitmap）に対応するプールへ反映。機構切替（`B`）は非アクティブ側を
  全部隠し、アクティブ側を `count` 個だけ可視化して `.text` を入れ直す。
- **決定的文字列生成（ホットパス）**: `makeText(it, i)` は `'OBJ#' + pad4(base) + ' v=' + ((frame*7+i*13)%1000)`。
  `pad4` は `slice` ベースの軽量ゼロ埋め。Update=ON ではこの結果を毎フレーム `.text` に代入する。
- **Update=OFF の固定**: ON→OFF 切替・機構切替・リセット時に `refreshAllText()` で一度だけ全件の
  `.text` を現在値に固定し、以降は位置（`o.x/o.y`）だけ動かす。
- **統計パネル**: 仕様要件のため Pixi 側に複数行 `PIXI.Text` を1つ置き、毎フレーム更新。Canvas Text の
  場合はこの1個も毎フレーム再ラスタライズされる（多行再レイアウトのコストも踏む）。
- **決定的乱数**: 初期化は `mulberry32(20250613)` のみ。`Math.random` は不使用。`R` で再シードして
  初期状態へ戻す。

## Codex 生成所感（v8 で大量動的テキストをベンチする観点）

- **本テーマは「描画ライブラリのテキスト実装差」を素直に突く**。テーマ1〜12の HUD は数行だけで
  テキスト描画は誰も負荷にしていなかったが、Pixi の `PIXI.Text` は**Canvas ラスタライズ → GPU
  アップロード**という重い経路を持ち、`.text` 書き換え（Update=ON）でそれが毎フレーム走る。
  N とこの ON/OFF の2軸だけで FPS が綺麗に動くので、ベンチ題材として素直で良い。
- **v8 Text API の変更点が地味に効く**: `new PIXI.Text(str, style)`（v7）が
  `new PIXI.Text({ text, style })` に変わっている。誤って旧シグネチャで渡すと無言で空文字になりやすい。
- **BitmapText の install が外部アセット不要で済むのが嬉しい**: `PIXI.BitmapFont.install({chars})` で
  内蔵フォントから ASCII アトラスを1回焼くだけ。色は白焼き + `tint`、サイズは `scale` で吸収でき、
  「同条件で機構だけ差し替え」が綺麗に作れた。崖の正体（グリフ再ラスタライズ）が `B` 一発で
  可視化できるのが、このデモの一番の見どころ。
- **プール設計の勘所**: Text の構築は Canvas/Bitmap とも安くないので、生成は遅延 + 上限 5000 まで、
  増減は `visible` トグルのみ。これで `+/-` の負荷比較がフレーム生成ノイズに汚されず安定して取れる。
- **HUD は HTML オーバーレイのまま**（テーマ5踏襲）。一方で「複数行統計パネル」は仕様要件なので
  Pixi 側に別途1つ置き、これも動的更新の負荷に含めた。計測の主役（アイテム群）と HUD の表示更新を
  混ぜないことで、FPS が何のコストを映しているかが明快になる。
