# テーマ13 大量動的テキスト / UI 描画 ― Phaser 4 版

「システムログ / データダッシュボード」風のデモ。画面いっぱいに多数のテキストラベルが
決定的に流れ、各ラベルの数値部分が刻々と更新される。**多数のテキストオブジェクト ＋
毎フレームの文字列更新（再レイアウト/グリフのテクスチャ再生成）** のスループットを測る比較用サンプル。
テーマ1〜12では HUD の数行しかテキストを使っておらず、テキスト/グリフ描画という
**既知のパフォーマンスの崖**を計測してこなかった。本テーマはその未計測軸を埋める。

## 起動方法

ブラウザのセキュリティ制約(file:// での CORS)を避けるため、簡易 HTTP サーバ経由を推奨します。

```bash
# このフォルダ (13/Phaser4/) で
python -m http.server 8000
# → http://localhost:8000/ をブラウザで開く
```

または任意の静的サーバ(`npx serve` など)でも可。**外部アセットは一切不要**で必ず起動します。

## 使用バージョン

- Phaser **4**(CDN: `https://cdn.jsdelivr.net/npm/phaser@4/dist/phaser.min.js`)
- 追加ライブラリなし。`index.html` + `game.js` のみ。
- 画像アセット不要(既定の Consolas/monospace システムフォントを使用)。

## 操作

| キー | 動作 |
|---|---|
| `+` / `-`(テンキー可) | テキストアイテム数(負荷)を ±100 増減(0〜5000 にクランプ) |
| `U` | 文字列の毎フレーム更新 ON/OFF 切替(計測の核) |
| `B` | テキスト機構切替 Canvas Text ⇄ BitmapText |
| `R` | リセット(N=200・Update=ON・Canvas へ戻し、シードから再生成) |

## 仕様数値表(全エンジン共通)

| 項目 | 値 |
|---|---|
| 画面 | 960 x 540(固定) |
| 背景 | `#0b0d16`(暗色) |
| テキスト数 N | 初期 200 / ±100 / 下限 0・上限 5000 |
| アイテム文字列 | `OBJ#0042 v=137` 形式(8〜20文字)。`v=` の数値が毎フレーム更新 |
| 速度 | 30〜130 px/s(角度・速さは決定的) |
| サイズ | 11/12/13/14/16 px から決定的に割当 |
| 色 | 8 色から決定的に割当 |
| 動き | 上下左右バウンド(端で速度反転) |

## HUD(左上・必須項目)

- `FPS`(実測・移動平均 30 サンプル)
- `Texts`(現在のアイテム数 / 上限 5000)
- `Chars`(概算の総グリフ数 = アイテム数 × 平均文字数 14 ＋ 統計パネル分)
- `Render`(使ったテキスト機構: `Canvas Text` または `BitmapText`)
- `Update`(`dynamic` = 毎フレーム文字列更新 / `static` = 固定)
- 画面下に操作ヒント行(+/- / U / B / R)。

別途、右上に**複数行のライブ統計パネル**(frame 番号・経過秒・active 数・`setText`/f・mode・dyn)を
1 つ置き、これも毎フレーム `setText` して**多行テキストの再レイアウト**も踏ませている。

## 実装メモ

### 使ったテキスト機構と崩れ始める点

- **既定は Canvas Text**(`this.add.text`)。これが各エンジンの「素直な選択」で、本テーマの比較対象そのもの。
  Phaser の `Text` は **`setText` のたびに内部 Canvas を再ラスタライズ**する(文字列の計測 →
  Canvas クリア → グリフ描画 → GPU テクスチャ再アップロード)。**Update=ON ではこれが毎フレーム
  N 回走る**のがコストの主軸。
- **`B` で BitmapText**(`this.add.bitmapText`)に切替可能。BitmapText はグリフを 1 枚のフォントテクスチャから
  クアッドで貼るだけなので、`setText` してもテクスチャ再生成は起きず、頂点バッファの再構築のみ。
  そのため **Update=ON でも Canvas Text より大幅に軽い**。
- **崩れ始める点(目安・環境依存)**:
  - **Canvas Text / Update=ON**: 数百〜1000 個あたりから 60fps を割り始める。毎フレームの
    Canvas ラスタライズが支配的で、N に対してほぼ線形に重くなる。2000 個超では一桁 fps になりやすい。
  - **Canvas Text / Update=OFF**: 文字列を作り直さない(再ラスタライズ無し)ため、各 Text は
    キャッシュ済みテクスチャを貼るだけになり、同じ N でも大幅に軽い。崩れる点は数倍 N 側へ移る。
  - **BitmapText / Update=ON**: 再ラスタライズが無いぶん、Canvas Text の Update=ON より数倍多い N まで耐える。
    ここが「Canvas テキスト(毎回ラスタライズ)」と「ビットマップ/キャッシュ系」の `FPS` の落ち方の差。

### BitmapText を外部フォントアセット無しで用意する方法

- SPEC は「外部ビットマップフォントは使わない(動的生成の範囲で)」とするため、**起動時(`BootScene`)に
  ランタイムでビットマップフォントを焼いている**:
  1. `this.textures.createCanvas` で空テクスチャ用 Canvas を確保。
  2. ASCII 32(' ')〜122('z') を**等幅セル**(12x20px)グリッドに白で `fillText` して焼く
     (表示文字列 `OBJ#0042 v=137` の小文字 `v` まで網羅)。
  3. Phaser 標準の **`RetroFont.Parse`**(固定グリッド用ビットマップフォントのパーサ)に
     `{image, width, height, chars, charsPerRow}` を渡し、返ったキャッシュエントリ
     `{data, frame, texture}` を `this.cache.bitmapFont.add` で登録。
  - グリフは白で焼き、表示時に `setTint` でアイテムごとの色を付ける(Canvas Text と同じ 8 色)。
  - `RetroFont` が無いビルドでは `bmpFontReady=false` とし、**`B` を no-op**(何も起きない)にしている。
    その場合 HUD の `Render` は `Canvas Text (bmp N/A)` を表示する。

### プール(生成/破棄を毎フレームしない)

- テキストオブジェクトは **Canvas Text 用 / BitmapText 用の 2 系統をプール**し、cap(=5000)まで
  **遅延生成**する。`+`/`-` での増減は「先頭 N 個を可視・残りを `setVisible(false)`」で切り替えるだけで、
  オブジェクトの生成/破棄は一切行わない(SPEC のプール要件)。
- アイテムの**論理データ**(座標・速度・色・サイズ・基準文字列)は描画オブジェクトと分離して保持。
  `B` でモードを切り替えても論理データは共有され、配置・動きは決定的に維持される。

### 動的更新(コストの主軸)

- **Update=ON(既定)**: メインループで可視中の各アイテムについて文字列 `OBJ#xxxx v=NNN` を作り直して
  `setText` する。`v=` の数値は `(baseVal + frame) % 1000` で毎フレーム変化させる(Canvas Text なら
  ここで全アイテムが再ラスタライズ)。`setText` 呼び出し数は統計パネルの `setText/f` に出している。
- **Update=OFF**: 文字列は固定のまま `setPosition` で位置だけ動かす。`setText` を呼ばないため
  Canvas Text の内部テクスチャはキャッシュが効き、描画は貼り直しのみになる。
- どちらのモードでも**位置更新は常に**行う(端でのバウンド＋速度反転)。これにより「テキスト数だけ多いが
  文字列は静的」「文字列も毎フレーム動的」のコスト差をそのまま比較できる。

### 決定的生成(シード固定)

- 初期位置・速度(角度＋速さ)・色・サイズ・基準数値はすべて `mulberry32`(seed `0x13ADBEEF`)製の
  **決定的疑似乱数**で決め、`Math.random` は不使用。`R` で同じシードから再生成するため、
  毎回まったく同じ初期状態・同じ動きに戻る。

## Codex 生成所感(大量動的テキストの実装しやすさ)

- **Canvas Text の素直さと崖**: `this.add.text` は数行で出せて見た目も綺麗だが、`setText` のたびに
  Canvas を丸ごと再ラスタライズする内部実装のため、「多数 ＋ 毎フレーム更新」では一気に崖が来る。
  この崖こそが本テーマの計測対象で、Phaser では**意図せず踏みやすい落とし穴**でもある。
- **BitmapText のための動的フォント生成**: Phaser は外部 `.fnt`/`.xml` を読む前提の API が多く、
  「アセット無しでビットマップフォントを用意する」には一工夫が要った。`Texts.createCanvas` でグリフを焼き、
  固定グリッド用の `RetroFont.Parse` に流して `cache.bitmapFont` へ登録する経路に落ち着いた。
  XML を手組みして `ParseXMLBitmapFont` に渡す方法もあるが、その関数はテクスチャフレームを第 2 引数に
  要求するため、フレーム解決まで自前で面倒を見る必要がある。**等幅グリッドなら `RetroFont.Parse` が
  最短**で、返り値がそのままキャッシュエントリ形なのも扱いやすかった。
- **プール構造**: Text/BitmapText いずれも `setVisible` で表示数を制御し、cap まで遅延生成して使い回す
  構造はテーマ5の敵プールと同型で、Codex 生成でも安定して書けた。`+`/`-` での増減が生成/破棄を伴わない点が
  ベンチの再現性に効く。
- **計測のしやすさ**: `setText/f`(毎フレームの再ラスタライズ回数)を統計パネルに出すことで、Update=ON/OFF・
  Canvas/Bitmap の違いがそのまま数字に現れ、`FPS` の落ち方と突き合わせやすい。
- 総じて Phaser 4 は「素直な Canvas Text」と「速いが準備が要る BitmapText」のコントラストが明確で、
  本テーマ(テキスト描画の崖)の比較サンプルとして題材適性が高かった。
