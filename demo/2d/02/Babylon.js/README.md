# トップダウン・サバイバー ― Babylon.js 版

テーマ2「トップダウン・サバイバー」の Babylon.js 実装。
3Dエンジンである Babylon.js を **正射影カメラ + SpriteManager/Sprite** で 2D ゲームとして動かし、
数百〜千体の敵スプライトを捌く性能を比較するためのもの。

## 起動方法

ローカルファイルでもおおむね動くが、`../assets/` の画像読込やテクスチャ生成の都合上、
簡易HTTPサーバ経由での起動を推奨。

```bash
# このフォルダ(2/Babylon.js)で
python -m http.server 8000
# → ブラウザで http://localhost:8000/index.html
```

または `index.html` をブラウザに直接ドラッグ＆ドロップでも起動する
（画像アセットが無い場合は単色図形フォールバックで必ず起動する）。

- Babylon.js は CDN `https://cdn.babylonjs.com/babylon.js`（最新安定版）を読込。
- キャンバスは **960 x 540 固定**。

## 操作

| 入力 | 動作 |
|---|---|
| `W A S D` / 矢印 | 自機移動（8方向・180 px/s） |
| 攻撃 | オート（最近接敵へ 400ms ごと・弾速 350 px/s） |
| `+` / `-` | 敵スポーン上限を ±50（0〜1000） |
| `Enter` | ゲームオーバー後のリスタート |

## HUD

HTML オーバーレイで表示：`FPS`（移動平均）/ `Enemies`（生存数）/ `Objects`（敵+弾+gem）/
`Time`（生存秒）/ `Kills` / `HP`。スポーン上限(target)も併記。

## 仕様準拠（SPEC.md の数値）

- 移動 180 px/s、攻撃 400ms 間隔、弾速 350 px/s（命中で敵HP-1）。
- 敵速度 60〜90 px/s、接触で自機HP-1（無敵 0.5s）、自機HP初期5。
- 敵HP = 1(bat/小・速) / 3(zombie/大・遅)。撃破で xp_gem ドロップ、接触取得で Kill+1。
- 同時敵数 初期150、`+`/`-`で±50、上限1000、10秒ごと +25 自動増加。
- 当たり判定は円。敵同士の重なりは許容。

## 実装メモ

- **正射影カメラ**: `ORTHOGRAPHIC_CAMERA` で `orthoLeft/Right/Top/Bottom` を毎フレーム
  自機中心に合わせ、ワールド px と画面 px を 1:1 に対応（カメラ追従＝無限スクロール）。
- **SpriteManager をエンティティ種別ごとに用意**（bat/zombie/projectile/gem/player）。
  同一 manager のスプライトは 1 ドローコールでバッチ描画されるため、
  千体規模でも描画は manager 数（=数個）のドローで済む。
- **プール再利用**: 敵・弾・gem の各オブジェクトとその `Sprite` をフリーリストで使い回す。
  ゲーム中の `new` を実質ゼロにし、GC スパイクを抑制。非アクティブ Sprite は
  `isVisible=false` かつ画面外退避で再利用待機。
- **フォールバック**: `../assets/` の各 PNG を `Image` で先読みし、失敗時は
  Canvas で動的生成した単色図形（自機=白丸/bat=紫丸/zombie=緑丸/弾=黄丸/gem=水色菱形）の
  DataURL を使用。画像が 1 枚も無くても必ず起動する。
- **地面**: `ground_tile.png` を WRAP したテクスチャの大型 Plane をカメラ追従させ、
  UV オフセットで無限スクロール表現（画像なしは単色）。
- **当たり判定**: 弾×敵・敵×自機・gem×自機をすべて円判定。負荷測定が目的のため、
  弾の命中は全敵走査の素直な総当たり（空間分割なし）。

## Codex 生成所感（3Dエンジンで大量2Dスプライトを捌く所感）

- Babylon.js は本来 3D エンジンだが、`SpriteManager` のおかげで 2D 大量スプライトは
  かなり素直に書ける。**種別ごとに manager を分けて 1 テクスチャ＝1 バッチ**にする設計が肝で、
  これだけで「敵1000 + 弾 + gem」でもドローコールが数個に収まる。3Dエンジン特有の
  メッシュ/マテリアルの重さを意識せずに済むのが大きい。
- 一方で **2D を 3D エンジンでやる違和感**もある。正射影カメラの ortho 範囲を手で毎フレーム
  更新したり、Sprite の z 順ではなく manager の生成順で重なりを制御したりと、
  「2D の常識」がそのままは通らない。座標系も Y が上方向なので、画面座標感覚との
  読み替えが要る。専用2Dエンジン（Pixi 等）なら何も考えず済む部分にひと手間かかる。
- 性能面では、ボトルネックは描画よりも **CPU 側のゲームロジック**（敵の追尾移動と
  弾×敵の総当たり当たり判定 O(弾×敵)）に出やすい。SpriteManager がバッチで描画を
  引き受けてくれるぶん、相対的にロジックの粗さが露呈する構図で、これは
  「3Dエンジンでも2Dは描画が軽い」ことの裏返しと言える。実用上は当たり判定の
  空間分割（グリッド/quadtree）を入れれば千体級でも余裕がありそう、という手応え。
- 総じて、**Babylon.js は 3D 資産を活かしつつ 2D 物量も現実的に捌ける**。
  ただし「2D に最適化された書き味」は専用2Dライブラリに一日の長があり、
  本タスクのような純2D負荷比較では Babylon の 3D 機能はほぼ出番がない、という所感。
