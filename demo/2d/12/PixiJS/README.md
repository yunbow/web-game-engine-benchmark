# テーマ12 フォーリングサンド / セルオートマトン ― PixiJS v8 版

砂・水・壁のセルが落下・堆積・流動する「フォーリングサンド」型のセルオートマトンを
**PixiJS v8** で実装。`COLS×ROWS` の格子をセル単位で更新し、**毎フレーム全面の RGBA
ピクセルバッファを生成して 1 枚のテクスチャへアップロード**、960×540 へニアレスト拡大表示する。
ベンチの核は「**グリッド解像度（セル数）＝ 1 フレームあたりのセル更新数 ＋ テクスチャ転送量**」。

## 起動方法

ローカルに HTTP サーバを立ててブラウザで開く（CDN 読込のため `file://` でも概ね動くが、
他テーマと揃えてサーバ経由を推奨）。

```bash
# このフォルダ(12/PixiJS/)の1つ上(12/)で実行
cd c:/work/claude/local-works/web-game/12
python -m http.server 8000
# → http://localhost:8000/PixiJS/ を開く
```

**アセットは不要**（セルの色はコードで定義し、毎フレーム生成する）。

## 使用バージョン

- PixiJS **v8**（CDN: `https://cdn.jsdelivr.net/npm/pixi.js@8/dist/pixi.min.js`）
- 追加ライブラリなし。`index.html` + `game.js` の2ファイル構成。
- **v8 注意**: 初期化は `const app = new PIXI.Application(); await app.init({...});` の2段階。
  キャンバスは `app.view` ではなく **`app.canvas`**。

## 使ったテクスチャ更新機構（本テーマの肝）

**`ImageData→canvas Texture, source.update()`**

1. オフスクリーン `<canvas>`（サイズ = `COLS×ROWS`）を 1 枚用意。
2. その 2D コンテキストの **`ImageData`（= RGBA の `Uint8ClampedArray`）** に毎フレーム
   全セルの色を書き込む（`grid[i]` の素材 → 砂色/水色/壁色/背景色 ＋ 決定的な濃淡）。
3. `ctx.putImageData(imageData, 0, 0)` で canvas へ反映。
4. その canvas を `PIXI.Texture.from(canvas)` でテクスチャ化し、
   `texture.source.scaleMode = 'nearest'` を設定（拡大時にドットがくっきり）。
5. 毎フレーム **`texture.source.update()`** で GPU へ再アップロード。
6. `Sprite` を `setSize(960, 540)` で全画面へニアレスト拡大して表示。

→ 「**`COLS×ROWS×4` バイトの RGBA を毎フレーム生成 ＋ GPU 転送**」が、まさに測られるコスト。
HUD の `Upload` 行に機構名と 1 フレームあたりの転送量（KB）を表示している。

## 操作

| 入力 | 動作 |
|---|---|
| 左ドラッグ | 現在のブラシ素材を描き込む（円形・半径3セル） |
| 右ドラッグ | 消去（empty で塗る・`contextmenu` は抑止） |
| `1` / `2` / `3` | ブラシ素材を 砂 / 水 / 壁 に切替 |
| `+` / `-`（テンキー可） | グリッド解像度 `COLS` を ±40（下限80・上限640・決定的に再構築） |
| `C` | 全消去（構造を消す。エミッタは毎フレーム供給され続ける） |
| `R` | リセット（決定的初期状態へ） |

## 仕様数値（全エンジン共通）

- キャンバス **960×540** 固定。グリッド **COLS×ROWS**（`ROWS = round(COLS×540/960)`）。
- `COLS` 初期 **160**（→ ROWS=90, 14400 セル）。`+/-` で **±40**（**80**〜**640**）。
  - 上限 640（→ ROWS=360, 230400 セル, ≒900 KB/frame の転送）で更新＋転送の双方が効く。
- セル素材: `0=空気 / 1=砂 / 2=水 / 3=壁`。**場外は壁扱い**（落下が端で止まる）。
- 素材色（SPEC 基準）: 砂 `#d9c067` 系・水 `#3a7bd5` 系・壁 `#888`・空気 `#0b0d12`。
  砂/水はセルごとに決定的な濃淡を焼き込み、堆積が見やすいようにしている。
- 時間はデルタタイム基準だが、**シミュレーションは固定タイムステップ**（毎フレーム1ステップ）。

## シミュレーション規則（決定的・`Math.random` 不使用）

- **走査順**: 下の行から上へ。各行は左右交互スキャン（行の偶奇 ＋ フレーム位相で反転）にして
  偏りを抑える。乱択（左下/右下のどちらを先に試すか）は **`mulberry32`** で決定的に行う。
- **砂(sand)**: 真下が空なら落下。真下が水なら**入れ替え**て沈む。塞がれていれば左下・右下
  （空 or 水なら移動/入れ替え）。
- **水(water)**: 真下が空なら落下。塞がれていれば左下・右下、それも塞がれていれば左右へ広がる。
- **壁(wall)**: 不動。**空気(empty)** も不動。
- **エミッタ（無入力でもベンチが回る）**: 上部に4個（砂2・水2）を **COLS への比率で**決定的配置。
  毎フレーム少量供給するので、マウスを触らなくても常にセルが動き、ベンチが安定する。
  解像度を変えても比率配置なので相対位置を維持する。

## HUD（HTMLオーバーレイ `#hud`・約120msごと更新）

- `FPS`（直近60フレームの移動平均）
- `Grid`（`COLS x ROWS = セル数`）
- `Active`（空気以外のセル数）＋ そのフレームで移動したセル数 `moved/frame`
- `Brush`（現在のブラシ素材: sand/water/wall）
- `Upload`（テクスチャ更新機構 `ImageData→canvas Texture, source.update()` ＋ 転送量 KB/frame）

HUD はキャンバス外の HTML `<pre>` 要素で描画し、Pixi の描画・転送負荷と切り離して計測できる。

## 実装メモ

- **グリッド表現**: 素材は flat な **`Uint8Array`**（`grid[y*cols+x]`）。描画用の濃淡は
  `Int8Array tint[]` に決定的に焼き込み、移動時に素材と一緒に運ぶ（`moveTo`/`swap`）。
- **更新（`Sandbox.step`）**: 下→上、各行は左右交互スキャン。砂は `moveTo`/水とは `swap`、
  水は落下→斜め下→横拡散。場外参照は `get()` が `WALL` を返すので端で自然に止まる。
- **テクスチャ書き換え（本デモの核・`TexUploader.upload`）**: `grid[]` を走査して `ImageData`
  へ RGBA を直書き → `putImageData` → `source.update()`。`COLS×ROWS×4` バイトの生成＋GPU転送が
  解像度に比例して増えるので、`+/-` で更新コストと転送コストの双方をスイープできる。
- **解像度変更**: `setSize(cols)` で `ROWS=round(cols×540/960)` を算出し、状態を**決定的に再構築**。
  同時に canvas / ImageData / Texture も作り直し、旧テクスチャは `destroy(true)` で破棄して張り替える。
  エミッタは比率配置なので解像度に追従する。
- **拡大表示**: `COLS×ROWS` のテクスチャを `Sprite.setSize(960,540)` で拡大。
  `source.scaleMode = 'nearest'` でセルがドットとしてくっきり見える。
- **入力**: ポインタは `app.canvas` の pointer events で取得。`getBoundingClientRect()` で
  CSS 縮小表示ぶんを比率換算してセル座標へ。ドラッグは線分補間で塗り、速い動きでも途切れない。
  右ボタンは消去、`contextmenu` は `preventDefault` で抑止。

## Codex 生成所感（v8 で動的テクスチャ書き換えを組む観点）

- **テーマ1〜11の未計測軸**: 他テーマは既製スプライト/図形/ライトを「描く」だけで、
  **毎フレーム CPU でピクセルを生成して GPU へ上げる**処理は誰も行っていない。本テーマはその
  軸を埋める。Pixi の通常の描画パスではなく **テクスチャソース更新**のスループットが主役になる。
- **v8 の素直な機構**: `PIXI.Texture.from(canvas)` で canvas をソース化し、毎フレーム
  `source.update()` するのが最も率直。v7 までの `BaseTexture.update()` 相当が v8 では
  `texture.source.update()` に整理されており、迷いどころが少ない。`scaleMode` は文字列
  `'nearest'` で `source` に設定する（v7 の `PIXI.SCALE_MODES.NEAREST` 定数ではない点に注意）。
- **コストの所在**: 14400 セル（160×90, 約56KB/frame）では FPS は張り付くが、640×360
  （230400 セル, 約900KB/frame）まで上げると **ImageData への書き込みループ**と
  **`source.update()` の GPU 転送**が両方効いてくる。JS ループのセル走査と RGBA 直書きが
  支配的で、Pixi のドローコール自体は Sprite 1枚ぶんしかない（描画より「生成＋転送」が律速）。
- **決定性の確保**: 走査の左右交互＋フレーム位相反転で砂山の偏りを抑え、左下/右下の選択は
  `mulberry32` で行うことで `Math.random` を完全排除。初期状態・エミッタも固定シード/比率配置で、
  解像度を変えても再現性のある比較ができる。
- 総じて v8 は「**1枚の canvas テクスチャを毎フレーム更新するだけ**」という最小構成で
  動的テクスチャ書き換えベンチが組め、`Texture.from`→`source.update()` の経路が明快だった。
  比較軸（セル数＝更新＋転送量）が `+/-` でそのまま負荷に直結するのも分かりやすい。
