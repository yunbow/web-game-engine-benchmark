# テーマ12 フォーリングサンド / セルオートマトン ― KAPLAY 版

砂・水・壁のセルが落下・堆積・流動する「フォーリングサンド」型のセルオートマトンを
**KAPLAY 3001** で実装。`COLS×ROWS` の格子をセル単位で更新し、**毎フレーム全面の RGBA
ピクセルバッファを生成して 1 枚のテクスチャへアップロード**、960×540 へニアレスト拡大表示する。
ベンチの核は「**グリッド解像度（セル数）＝ 1 フレームあたりのセル更新数 ＋ テクスチャ転送量**」。

正準リファレンスは `../PixiJS/game.js`。シミュレーション規則・定数・色は全エンジンで同一。

## 起動方法

ローカルに HTTP サーバを立ててブラウザで開く（CDN 読込のため `file://` でも概ね動くが、
他テーマと揃えてサーバ経由を推奨）。

```bash
# このフォルダ(12/KAPLAY/)の1つ上(12/)で実行
cd c:/work/claude/local-works/web-game/12
python -m http.server 8000
# → http://localhost:8000/KAPLAY/ を開く
```

**アセットは不要**（セルの色はコードで定義し、毎フレーム生成する）。

## 使用バージョン

- KAPLAY **3001.0.19**（CDN: `https://unpkg.com/kaplay@3001.0.19/dist/kaplay.js`）
- `global: false` で名前空間 `k.*` を明示利用。`index.html` + `game.js` の2ファイル構成。
- 全体設定 `texFilter: 'nearest'` ＋ `crisp: true` で拡大時のドットをくっきりさせる。

## 使ったテクスチャ更新機構（本テーマの肝）

**`ImageData→canvas, Texture.update()`（KAPLAY 内部 GL テクスチャ）**

1. オフスクリーン `<canvas>`（サイズ = `COLS×ROWS`）を 1 枚用意。
2. その 2D コンテキストの **`ImageData`（= RGBA の `Uint8ClampedArray`）** に毎フレーム
   全セルの色を書き込む（砂色/水色/壁色/背景色 ＋ 決定的な濃淡）。
3. `ctx.putImageData(imageData, 0, 0)` で canvas へ反映。
4. その canvas を **初回のみ** `k.loadSprite(name, canvas)` でスプライト化し、返ってきた
   `SpriteData` から **KAPLAY 内部の `Texture`（GL テクスチャ）** を取り出して保持する。
5. 毎フレーム **`tex.update(canvas, 0, 0)`** で GPU へ再アップロード（`loadSprite` を毎フレーム
   呼ばず、テクスチャだけを更新する）。
6. スプライトを `width/height = 960/540` で全画面へニアレスト拡大して表示。

→ 「**`COLS×ROWS×4` バイトの RGBA を毎フレーム生成 ＋ GPU 転送**」が、まさに測られるコスト。
HUD の `Upload` 行に機構名と 1 フレームあたりの転送量（KB）を表示している。

## 操作

| 入力 | 動作 |
|---|---|
| 左ドラッグ | 現在のブラシ素材を描き込む（円形・半径3セル） |
| 右ドラッグ | 消去（empty で塗る・`contextmenu` は抑止） |
| `1` / `2` / `3` | ブラシ素材を 砂 / 水 / 壁 に切替 |
| `+` / `-`（テンキー可） | グリッド解像度 `COLS` を ±40（下限80・上限640・決定的に再構築） |
| `C` | 全消去（構造を消す。エミッタは毎フレーム供給され続ける） |
| `R` | リセット（決定的初期状態へ） |

## 仕様数値（全エンジン共通）

- キャンバス **960×540** 固定。グリッド **COLS×ROWS**（`ROWS = round(COLS×540/960)`）。
- `COLS` 初期 **160**（→ ROWS=90, 14400 セル）。`+/-` で **±40**（**80**〜**640**）。
- セル素材: `0=空気 / 1=砂 / 2=水 / 3=壁`。**場外は壁扱い**（落下が端で止まる）。
- 素材色（SPEC 基準）: 砂 `#d9c067` 系・水 `#3a7bd5` 系・壁 `#888`・空気 `#0b0d12`。
  砂/水はセルごとに決定的な濃淡を焼き込み、堆積が見やすいようにしている。
- 時間はデルタタイム基準だが、**シミュレーションは固定タイムステップ**（毎フレーム1ステップ）。

## シミュレーション規則（決定的・`Math.random` 不使用）

- **走査順**: 下の行から上へ。各行は左右交互スキャン（行の偶奇 ＋ フレーム位相で反転）。
  乱択（左下/右下のどちらを先に試すか）は **`mulberry32`** で決定的に行う。
- **砂(sand)**: 真下が空なら落下。真下が水なら**入れ替え**て沈む。塞がれていれば左下・右下。
- **水(water)**: 真下が空なら落下。塞がれていれば左下・右下、それも塞がれていれば左右へ広がる。
- **壁(wall)** / **空気(empty)**: 不動。
- **エミッタ**: 上部に4個（砂2・水2）を **COLS への比率で**決定的配置。毎フレーム少量供給するので、
  マウスを触らなくても常にセルが動き、ベンチが安定する。

## HUD（HTMLオーバーレイ `#hud`・約120msごと更新）

- `FPS`（直近60フレームの移動平均）
- `Grid`（`COLS x ROWS = セル数`）
- `Active`（空気以外のセル数）＋ そのフレームで移動したセル数 `moved/frame`
- `Brush`（現在のブラシ素材: sand/water/wall）
- `Upload`（テクスチャ更新機構 `ImageData->canvas, Texture.update()` ＋ 転送量 KB/frame）

## 実装メモ

- **グリッド表現**: 素材は flat な **`Uint8Array`**（`grid[y*cols+x]`）。濃淡は `Int8Array tint[]`
  に決定的に焼き込み、移動時に素材と一緒に運ぶ（`moveTo`/`swap`）。
- **テクスチャの取り回し**: KAPLAY の `loadSprite` は同名上書きの保証が曖昧なため、**解像度変更
  ごとに一意名でロード**し、内部 `Texture` を取り直す。通常フレームは `loadSprite` を呼ばず
  `tex.update(canvas)` のみ（GL テクスチャの直接更新）。
- **拡大表示**: `texFilter:'nearest'`＋`crisp:true` でセルがドットとしてくっきり見える。
- **入力**: `k.onMousePress`/`onMouseMove`/`onMouseRelease` でドラッグを線分補間して塗る。
  KAPLAY のマウス座標は画面座標と一致するので比率換算してセル座標へ。右ボタンは消去。

## Codex 生成所感

- KAPLAY は 2D アーケード向けの軽量エンジンで、スプライト表示・入力は素直に書ける。
- 本テーマの肝は「**既製スプライトを描く**」のではなく「**毎フレーム CPU でピクセルを生成して
  GPU へ上げる**」点。KAPLAY は公開 API として動的テクスチャ更新を前面に出していないため、
  `loadSprite` が返す `SpriteData` から内部 `Texture` を拾い `tex.update()` する、というやや
  内部寄りの経路になる。ここはバージョン差で `tex` の在処が変わり得るので、`getSprite` 経由の
  フォールバックも用意した。
- 一方シミュレーション本体は他エンジンと完全共通の `Uint8Array` グリッド＋決定的更新で、
  エンジン差は「テクスチャ更新経路」と「拡大時のフィルタ指定」にほぼ集約される。
