# ブロック崩し（マルチボール Breakout） ― Babylon.js 版

テーマ4 共通仕様（`../SPEC.md`）を **Babylon.js** で実装したもの。
3Dエンジンである Babylon.js を使い、**正射影(Orthographic)カメラ + Sprite/Plane** で
2D ブロック崩しを構成している。**物理エンジンは使わず**、位置更新・反射・
AABB×円（最近点）判定をすべて自前実装し、多数ボール × 数百ブロックの
衝突スループットを測れるようにしている。

## 起動方法
画像アセットを `Image` で読むため、`file://` では読込制限で
アセットが読めない場合がある（その場合でもフォールバック図形で**必ず起動**する）。
アセット画像をきちんと表示したい場合はローカルサーバ経由推奨。

```bash
# このフォルダ（4/Babylon.js/）で
python -m http.server 8000
# → ブラウザで http://localhost:8000/index.html
```

ダブルクリック（`file://`）でも単色フォールバック図形で動作する。
（同梱の `../assets/` は空のため、既定では下記フォールバック描画になる）

### フォルダ構成 / アセット
- アセットは `../assets/`（= `4/assets/`）から SPEC のファイル名で読み込む。
  - `paddle.png` / `ball.png` / `brick.png`(明色, HPで tint) / `hit_spark.png` / `bg_breakout.png`
- 画像が存在しない・読込失敗時は **単色図形にフォールバック**：
  - パドル = 白い矩形、ボール = 白丸、ブロック = HP色矩形（赤/橙/緑）、
    spark = 黄バースト、背景 = 暗色

## 使用バージョン
- Babylon.js: **CDN 最新安定版** `https://cdn.babylonjs.com/babylon.js`（グローバル `BABYLON`）
- 追加ライブラリなし・ビルド不要

## 操作
- パドル移動: 矢印キー `←`/`→` または `A`/`D`（水平のみ・画面内クランプ）
- ボール: **自動発射**（手動ロックなし。負荷比較のため常にプレイ継続）
- `+` / `-`: 同時ボール数を ±5（下限1・上限500）。性能比較の主軸。
  - キーは `'+','=','-','_'` を許容（参照実装と同じ keydown 処理）。

## 仕様数値（SPEC 準拠）
- パドル: 幅 **96** × 高さ **18**、中心 y=**510** 固定、移動速度 **600 px/s**、左右クランプ
- ボール: 半径 **8**、速さ **380 px/s**（一定）、発射は上方向ランダム角（左右±60°）
- 反射:
  - 左右壁・天井で反射
  - パドルで反射（常に上方向。中心からのオフセット ±1 を ±60° に写像、速さは 380 に再正規化）
  - 下端（y > 540 + r）を抜けたら**ロスト → パドル上から再発射**して同時数を維持（`Lost` をカウント）
- ブロック: **15列 × 9行 = 135個**、56×20 + 間隔4、上オフセット60、グリッド中央寄せ
  - 上3行=HP3 / 中3行=HP2 / 下3行=HP1。HP で色変化（HP3=赤 / HP2=橙 / HP1=緑）
  - `brick.png` は明色1枚を **HP色で tint（乗算）** して使う（Sprite は `sprite.color`、Plane は HP色マテリアル）
  - 命中で HP-1、HP0 で破壊し **Score +10**、`hit_spark` を一瞬表示
- ボール×ブロック判定: **AABB（矩形）× 円（最近点）**。貫通の浅い軸で反転面を決定（左右面=vx反転 / 上下面=vy反転）
  - 1ボールにつき1フレーム1ブロックまで（最初の命中で break）
- **全ブロック破壊で盤面を再生成**（ベンチを継続させるため）
- 同時ボール数: 初期 **3**、`+`/`-` で **±5**（下限1・上限 **500**）

## HUD（画面左上オーバーレイ, 約0.1秒ごと更新）
- `FPS`（実測, 指数移動平均）
- `Objects`（ボール + 残ブロック + エフェクトの合計）
- `Score`
- `Balls`（現在の同時ボール数 / 設定値）
- `Bricks`（残ブロック数）
- `Lost`（落下→再発射した回数。負荷比較の参考）
- `Render`（`Sprite(tex)` / `Plane(fallback)` 描画モード表示）

## 実装メモ（座標系 y 下向き）
- **座標系**: 正射影カメラを `orthoLeft=0, orthoRight=960, orthoTop=0, orthoBottom=540`
  に設定し、ワールド座標を**画面ピクセル座標（左上原点・y下向き）**へ一致させた。
  `orthoTop < orthoBottom` とすることで y が下向きになり、2D の座標計算を
  そのまま 3D 空間に書ける。ボールの上向き発射は y 下向き座標なので `vy < 0`。
- **描画**: テクスチャがあれば `SpriteManager`/`Sprite`（GPU バッチで大量ボールに強い）、
  無ければ `Mesh`（矩形=`CreatePlane(1x1)` を scaling、円=`CreateDisc`）+
  `StandardMaterial(emissive, disableLighting)` にフォールバック。
  両者を同一インターフェース（`setPos/setVisible/setSize/setRectSize/setColor`）でラップ。
- **HPごとの色**: Sprite では `sprite.color`（Color4）の tint（乗算）で明色テクスチャを着色。
  Plane フォールバックでは HP 色の `emissiveColor` マテリアルへ差し替え。
- **プール**: ボール・ブロック・エフェクトは dispose せずプールで再利用し GC スパイクを抑制。
- **物理なし**: 反射は面ごとの速度反転、めり込みは押し出し補正。反射後に速さを 380 へ再正規化。
- **ボール同士の衝突は行わない**（SPEC通り。性能/簡潔さのため省略）。

## Codex / AI コーディングでの生成しやすさ所感（3Dエンジンで2Dを書く難しさ）

- **最大のつまずきは座標系**。Babylon は 3D 前提で y が上向き・原点中央が標準。
  ブロック崩しの「左上原点・y下向き・ピクセル単位」に合わせるには正射影カメラの
  `orthoTop/orthoBottom` を**意図的に反転**（top=0, bottom=540）させる必要がある。
  ここを決め打ちしないと、上向き発射の符号やパドル反射の向きが延々と逆になる。
- **Sprite の tint が Babylon 流の肝**。HP色の表現は、純2Dなら塗り色を変えるだけだが、
  Babylon の `SpriteManager` 経路では `sprite.color`（Color4）の乗算 tint を使うのが定石。
  一方フォールバックの Mesh 経路では `emissiveColor` マテリアルを差し替える必要があり、
  **同じ「色を変える」操作が描画経路ごとに別 API** になる。統一ラッパ（`setColor`）に
  押し込まないと AI はどちらか片方だけ書きがち。
- **矩形メッシュは Plane(1x1) + scaling が扱いやすい**。ブロックやパドルの可変サイズは
  `CreatePlane({width:1,height:1})` を `scaling.x/y` で伸ばすと、Sprite の `width/height`
  と同じ感覚で寸法指定できる。円は `CreateDisc`。形状ごとにサイズ指定の意味が変わる
  （Disc は半径基準）点は注意が必要。
- **当たり判定ロジック自体は 3D と無関係**で、AABB×円（最近点クランプ）+ 貫通軸比較は
  純2Dと同じく書ける。難所は判定ではなく「2D 用に 3D API をどう初期化するか」に集中する。
- **ライティング**。デフォルトのマテリアルはライト依存で暗くなるため、2D では
  `emissiveColor` + `disableLighting=true` が定石。Sprite は元々 unlit なので問題ないが、
  フォールバック Mesh は明示しないと色が安定しない。`backFaceCulling=false` も
  （カメラが背面側から見るため）忘れると「図形が見えない」事故になる。
- 総じて、**ロジックより「2D 化のための 3D 初期設定 + 描画経路の二重化」が生成の肝**。
  Pixi/Phaser のような純2Dエンジンに比べ、AI 生成時に与えるべき前提知識が一段多い。
