# ブロック崩し（マルチボール） ― Phaser 4 版

テーマ4 共通仕様（`../SPEC.md`）のブロック崩し（Breakout / Arkanoid 風, マルチボール）を Phaser 4 で実装したもの。
性能比較（FPS / 表示オブジェクト数）が目的で、**多数のボール × 多数のブロックの反射・当たり判定**を毎フレーム回す。

## 起動方法

1. このフォルダの `index.html` をブラウザで開く（ダブルクリック可）。
   - `index.html` と `game.js` が同フォルダにあれば動作する完全自己完結。
   - Phaser 4 は CDN から読み込む: `https://cdn.jsdelivr.net/npm/phaser@4/dist/phaser.min.js`（グローバル `Phaser`）。
2. アセットは `../assets/`（= `4/assets/`）から `SPEC.md` のファイル名で読み込む。
   - **画像が無くても必ず起動する**。ロード失敗時は `Graphics` で単色図形テクスチャを生成してフォールバックする
     （パドル=白角丸、ボール=白丸、ブロック=ほぼ白[HP色で tint]、spark=黄バースト、背景=暗色）。
   - `file://` で画像が CORS/未存在になってもフォールバックで起動するので、ローカルサーバは必須ではない。
     （ただし実画像で確認したい場合は `python -m http.server` 等で配信推奨。）

## 使用バージョン

- Phaser **4**（CDN: `phaser@4/dist/phaser.min.js`、`Phaser.AUTO` レンダラ = WebGL 優先）
- 物理エンジンは未使用（Arcade等は使わず、位置更新・反射・AABB×円判定を自前実装）。

## 操作

- パドル移動: 矢印キー `←`/`→` または `A`/`D`（水平のみ、画面内クランプ）
- ボール: オート発射（手動ロックなし。常時プレイ継続で負荷を一定に保つ）
- `+` / `-`: 同時ボール数を ±5（下限1・上限500）

## 仕様の数値（SPEC厳守）

| 項目 | 値 |
|---|---|
| 画面 | 960 x 540 固定 |
| パドル | 96 x 18、中心 y=510、速度 600 px/s |
| ボール | 半径 8、速さ 380 px/s（一定）、発射は上方向 ±60° |
| パドル反射 | 中心オフセットで角度可変（最大75°）、常に上方向、速さ380に再正規化 |
| ブロック | 15列 × 9行 = 135個、56x20 + 間隔4、上オフセット60 |
| ブロックHP | 上3行=HP3(赤) / 中3行=HP2(橙) / 下3行=HP1(緑) |
| 当たり判定 | AABB(矩形)×円（最近点）。当たり面で vx/vy 反転、1ボール1フレーム1ブロック |
| スコア | ブロック破壊 +10 |
| 同時ボール数 | 初期3（±5、下限1・上限500） |

## HUD（左上・常時表示）

- `FPS`（直近30フレームの移動平均）
- `Objects`（ボール + 残ブロック + エフェクトのアクティブ合計）
- `Score`
- `Balls`（現在の同時ボール数 / 設定値、参考として Lost 回数も併記）
- `Bricks`（残ブロック数）

## 実装メモ

- **更新はデルタタイム基準**（`update(time, delta)` の `delta/1000` 秒で全移動を計算）。
- **物理エンジン不使用**: 位置更新・壁/天井/パドル反射・ブロック衝突をすべて自前実装。
  - 壁/天井: 半径 `r` を考慮してクランプし、該当軸の速度成分を反転。
  - パドル: 当たった位置のパドル中心からのオフセット `[-1,1]` で反射角を決定（端ほど横に鋭く、最大75°）。
    常に上方向へ送り、速さは 380 px/s に再正規化（`cos/sin × BALL_SPEED`）。
  - ブロック: **AABB×円（最近点）**。円中心を矩形にクランプして最近点を求め、距離が半径以下なら衝突。
    `|dx| > |dy|` なら左右面=vx反転、そうでなければ上下面=vy反転。矩形内部にめり込んだ場合は
    貫入量の小さい軸で押し戻して反転。1ボールにつき1フレーム最初の命中で `break`。
- **オブジェクトプール**: `Phaser.GameObjects.Group` + `getFirstDead(false)` でボール・ブロック・エフェクトを
  再利用し、GC負荷とインスタンス生成コストを抑制（高負荷=ボール最大500個時の比較を意図）。
- **ブロックの色は tint**: `brick` テクスチャはほぼ白（near-white）で生成し、HP に応じて `setTint`
  （HP3=赤0xff4444 / HP2=橙0xffa23a / HP1=緑0x55cc66）。被弾で HP-1 のたびに tint を更新。
- ボールが下端（y - r > 540）を抜けたら **Lost をカウント → パドル上から即時再発射**し、同時ボール数を維持
  （ゲームオーバーにしない。負荷を一定に保つため）。`+`/`-` で設定値を上げた分は `fillBalls()` が補充。
- **全ブロック破壊で盤面を再生成**（`buildBrickField()` を呼び直し、ベンチを継続）。
- **ボール同士の衝突は行わない**（性能/簡潔さのため省略。SPEC明記どおり）。
- 破壊時に `hit_spark` を一瞬表示（プールから再利用、拡大フェードアウト）。
- フォールバックテクスチャは `BootScene` で `loaderror` を捕捉 → `Graphics.generateTexture` で生成。

## Codex / AI コーディングでの生成しやすさ所感

- **API の素直さ**: 高い。`this.add.image` / `Group` / `getFirstDead` / `Graphics.generateTexture` /
  `setTint` / `Phaser.Math.Clamp` など命名が直感的で、Phaser 3 の知識がほぼそのまま通用する。
  ブロック崩しはシーンライフサイクル（`preload`/`create`/`update`）に綺麗に収まり、AI が雛形を出しやすい。
- **ブロック崩し（Phaser 4）特有のポイント**:
  1. **ブロックの tint 着色**: 「明色1枚を HP 色で乗算」を満たすには、フォールバックテクスチャを必ず
     **near-white** で作る必要がある。ここを灰色や有色で作ると `setTint` の乗算結果が暗くなり、
     赤/橙/緑が濁る。AI はブロックを最初から色付き矩形で描きがちなので、「白く描いて tint」と明示が有効。
  2. **AABB×円（最近点）反射**: 当たり判定そのものより、**当たった面の判別と速度反転**が肝。
     円中心の矩形クランプで最近点を出し、`|dx| vs |dy|` で左右面/上下面を分ける定石が綺麗にハマる。
     ただし高速ボールや内部めり込み（dx=dy=0）のケアを忘れると貼り付き/トンネリングが出るので、
     貫入量の小さい軸での押し戻しを足した。AI はこの内部ケースを落としがち。
  3. **多数ボールのプーリング**: 500個規模では生成/破棄コストとGCが効いてくるため、Group + `getFirstDead`
     で使い回し、下端ロスト時も `killObject` → 同オブジェクトを `respawnFromPaddle` で復活させて
     インスタンス数を一定に保った。プール再利用時の `setActive(true).setVisible(true)` 復帰漏れに注意。
  4. **キーコード**: `+`/`-` は `KeyCodes.PLUS`(187) / `MINUS`(189) で、Shift 無関係に物理キーで発火。
     テンキーは `NUMPAD_ADD` / `NUMPAD_SUBTRACT`。AI は `keydown` の `event.key==='+'` を併用しがちで
     二重発火を招くため、`keydown-PLUS` 等のみに統一した。
  5. **物理を使うか否か**: 大量オブジェクトの性能比較では Arcade Physics のオーバーヘッドを避けたいので、
     反射・判定を自前にした。AI はデフォルトで物理（`this.physics`）を使いたがるので、明示的に外す指示が有効。
- **Phaser 4 固有**: 本実装が使う範囲（GameObjects/Group/Graphics/Input/Scale/Math/setTint）は Phaser 3 と
  互換で、CDN 版もグローバル `Phaser` で同じ。バージョン差でハマる箇所はほぼ無かった。
- **必要だった調整**: tint を効かせる near-white テクスチャ、AABB×円の面判別と内部めり込みケア、
  二重キー発火の排除、プール再利用時の状態復帰。いずれも定型で、AI 生成後の軽微な修正で済む。
