# ブロック崩し(マルチボール) ― PixiJS v8 版

共通仕様 `../SPEC.md`（マルチボール Breakout／性能比較用）の PixiJS 実装です。
**多数のボール × 最大数百個のブロック**の反射・当たり判定を毎フレーム回し、衝突処理スループットの限界を測ります。

## 起動方法

### 方法A: ダブルクリック
`index.html` をブラウザで開くだけで起動します（CDN から PixiJS を読み込みます）。

ただし `PIXI.Assets.load` が `file://` で画像を読む際、ブラウザによっては CORS/プロトコル制限で画像取得に失敗することがあります。その場合でも **単色図形フォールバックで必ず起動します**（仕様要件）。画像を確実に出したい場合は方法Bを推奨。

### 方法B: ローカルサーバ（推奨）
このフォルダの **1つ上**（`4/`）をルートにしてサーバを立てると、`../assets/` 参照が正しく解決します。

```bash
# 4/ で実行
python -m http.server 8000
# → http://localhost:8000/PixiJS/index.html
```

## 操作
- パドル移動: 矢印 `←`/`→` または `A`/`D`（水平のみ、画面内クランプ）
- ボール: **自動発射**（手動ロックなし。負荷比較のため常にプレイ継続）
- `+` / `-`: **同時ボール数を ±5**（下限1・上限500）。性能比較の主軸。

## 使用バージョン / 技術
- PixiJS **v8**（CDN: `https://cdn.jsdelivr.net/npm/pixi.js@8/dist/pixi.min.js`、グローバル `PIXI`）
- 追加ライブラリなし。`index.html` + `game.js` のみで自己完結。
- 画面 **960x540 固定**、`resolution:1 / autoDensity:false`（DPR の影響を排し性能比較を素直に）。
- **物理エンジンは不使用**。位置更新・壁/パドル反射・当たり判定はすべて自前の数式。

### v8 破壊的変更の注意
- 初期化が `new PIXI.Application(opts)` ではなく **`new PIXI.Application()` → `await app.init(opts)`** の2段階。
- 描画キャンバスは `app.view` ではなく **`app.canvas`**。
- `PIXI.Graphics` は **メソッドチェーン式**（`.roundRect(...).fill(0x..)` / `.circle(...).fill(...)`）。旧 `beginFill()/drawRect()/endFill()` は廃止。
- テクスチャ化は `app.renderer.generateTexture({ target: g, width, height })`（オブジェクト引数）。
- `TilingSprite` も `new PIXI.TilingSprite({ texture, width, height })` のオブジェクト引数形式。

## アセット
`../assets/` から SPEC のファイル名で読み込み（`paddle.png` / `ball.png` / `brick.png` / `hit_spark.png` / `bg_breakout.png`）。
各ファイルを個別に `try/catch` し、**無いものだけ** `PIXI.Graphics` 由来のフォールバックテクスチャに差し替えます。

| エンティティ | フォールバック図形 |
|---|---|
| パドル | 白の角丸矩形 |
| ボール | 白い円 |
| ブロック | **ほぼ白い角丸矩形**（HP色で `tint` 乗算するため明色） |
| 破壊エフェクト | 黄バースト（中心白） |
| 背景 | 暗色 + 散らした星 |

ブロックの色は **HP に応じて `sprite.tint`** で表現（HP3=赤 `0xff4444` / HP2=橙 `0xffa23a` / HP1=緑 `0x55cc66`）。同一テクスチャを tint だけ変えて共有するのでバッチングが効きます。

## HUD（必須・性能比較の要）
HTML オーバーレイ（左上）で常時表示：
- **FPS**（直近60フレームの移動平均）
- **Objects**（ボール ＋ 残ブロック ＋ エフェクトの合計。内訳も併記）
- **Score**
- **Balls**（現在の同時ボール数 / 設定値）
- **Bricks**（残ブロック数）
- **Lost**（下端を抜けたボールの累計）

HUD は `requestAnimationFrame` ごとではなく約120msごとに DOM 更新し、文字列生成コストがベンチを汚さないようにしています。

## 実装メモ
- **ゲームループ**: `app.ticker` の `deltaMS` を使い、移動・反射・タイマすべてデルタタイム基準。
- **入力**: `keydown/keyup` で `keys{}` を保持する自前実装。`+`/`-` はテンキー・`=`・`-` 各コードに対応し、`Equal`/`NumpadAdd` などを拾う。
- **反射**:
  - 左右の壁・天井は速度成分の符号反転 + めり込み補正で処理。
  - パドルは **中心からのオフセット (-1..+1)** を反射角 (±60°) に写像し、`cos/sin` で速度を再構成（常に上向き・速さ一定）。端ほど横に鋭く飛ぶ。
- **ボール × ブロック判定**: **AABB(矩形) × 円**。矩形内でボール中心に最も近い点 (`clamp`) との距離で接触判定し、x/y それぞれの重なり量（侵入深度）を比べて浅い軸＝当たった面とみなし `vx`/`vy` を反転。反射後は `BALL_SPEED` へ正規化。1ボール 1フレーム 1ブロックまで（最初の命中で `break`）。
- **ロスト処理**: 下端 (`y - r > H`) を抜けたら `Lost++` し、同じボールをパドル上から再発射 → **同時ボール数を維持**（負荷一定、ゲームオーバーにしない）。
- **盤面再生成**: 残ブロックが 0 になったら自動で再構築し、ベンチを継続。
- **ボール同士の衝突は行わない**（性能/簡潔さのため省略。総当たりは「ボール × ブロック」のみ）。
- **エンティティ**: 各要素は `{ sprite, ...state }`。除去は配列末尾とのスワップ＋ `sprite.destroy()` で O(1)。
- **フォールバックテクスチャ**は `renderer.generateTexture()` で一度だけ生成しキャッシュ → 大量 Sprite が同一テクスチャを共有しバッチング。ブロックは tint 違いで共有。

## Codex / AI コーディングでの生成しやすさ所感

**v8 新API の罠**
- 初期化が `new PIXI.Application()` → **`await app.init(opts)`** の2段階。AI は v7 以前の記憶で `new PIXI.Application({...})` を出しがちで、これだと v8 では描画されず（黙って失敗）ハマる典型ポイント。`app.view` → **`app.canvas`** の改名も同様。
- `PIXI.Graphics` の **メソッドチェーン式**（`.roundRect(...).fill(...)`）。旧 `beginFill/drawRect/endFill` を混ぜると無言で図形が出ない。
- `tint` は v8 でも健在で、**明色テクスチャ1枚を HP 色で乗算**してブロック3色を1テクスチャで賄える。これは Pixi の Sprite/tint が素直で AI も書きやすい部分。

**描画ライブラリゆえの自前実装量**
- Pixi は「描く」だけなので、**ゲームループ・入力・反射・AABB×円判定・盤面生成・HUD のすべてが自前**。Breakout は STG と違い「矩形×円の最近点 + 面判定」がロジックの肝で、AI は最近点 `clamp` までは出せても **「侵入深度の浅い軸で反射面を決める」面判定**を落としやすい（角に当たると変な反射になるバグの温床）。ここは明示的に指示すると安定する。
- パドル反射の「中心オフセット → 角度」写像は素の三角関数で、ライブラリ非依存ゆえ AI 生成は安定。速さの再正規化を忘れると徐々に加速/減速するので明記が有効。
- 総じて：**ロジックは書きやすいが、(1) v8 初期化/Graphics 新API のバージョン差異、(2) 矩形×円の面判定** の2点を正しく指示するのが成功の鍵。`tint` でのブロック色分けは Pixi の素直さが活きる場面。
