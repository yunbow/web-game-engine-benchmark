# テーマ6: タワーディフェンス (経路探索) ― Babylon.js 版

グリッド上のタワーディフェンスを **Babylon.js** で実装したもの。敵がスタート(左端中央)から
A\* で求めた経路を辿ってゴール(右端中央)へ向かい、タワーが射程内の敵を撃つ。2Dゲームだが
Babylon は3Dエンジンのため、**正射影(Orthographic)カメラ + SpriteManager/Sprite** で
30x17 タイルのグリッドを 1 画面に再現している。

本テーマの主眼は「**グリッド A\* 経路探索 ＋ 多数ユニットの経路追従更新 ＋ タワー射撃**」を
同時に回したときの性能。テーマ1〜5が描画/自前衝突中心だったのに対し、ここは **CPU側の AI ロジック
(A\*)と多数ユニットの追従** が負荷の主役になる。

## 起動方法

`../assets/` を相対パスで読むため、ローカルHTTPサーバ経由で開く（file:// でも起動はするが
画像が読めずフォールバック図形になる）。

```bash
# このフォルダ (6/Babylon.js) の1つ上 (6/) で実行すると ../assets も配信される
cd c:/work/claude/local-works/web-game/6
python -m http.server 8000
# ブラウザで http://localhost:8000/Babylon.js/ を開く
```

または VS Code の Live Server などでも可。
画像アセットが無い環境でも **単色図形でフォールバック**して必ず起動する。

## 使用バージョン

- Babylon.js: CDN 最新版 (`https://cdn.babylonjs.com/babylon.js`)
- 追加ビルド・依存なし。`index.html` + `game.js` の2ファイルのみ。

## 操作

| 操作 | 動作 |
|---|---|
| 左クリック | 通路タイルにタワー設置 (25 Gold)。経路を塞ぐ配置は失敗 |
| 右クリック | タワー撤去 (返金なし)。経路を再計算 |
| `+` / `-` | 同時出現する敵数の上限を ±10 (下限10・上限500) |
| `R` | リスタート (マップ・ゴールド・ライフ初期化) |
| クリック (GAME OVER中) | リスタート |

## 仕様準拠（数値）

- キャンバス **960x540** 固定 / タイル **32x32** / グリッド **30x17**（= 960x544 px, 1画面・スクロール無し）。
- マップは決定的擬似乱数 (mulberry32, seed=0x7DEF) で生成。`0=通路, 1=壁`。
  - 外周を壁で囲み、スタート(左端中央)/ゴール(右端中央)の開口部のみ空ける。内部に約22%で壁ブロックを散らす。
  - 生成後に A\* で連結性を確認し、塞いでしまった場合は中央行の壁を決定的に取り除いて**必ず経路を残す**。
- 経路は **グリッド A\*（4方向・コスト1・マンハッタン距離ヒューリスティック・バイナリヒープ）** で
  スタート→ゴールを計算。**壁 ＋ タワーの占有タイル**を通行不可に扱う。
- 敵 (creep): 半径 **10**（描画 ~24）、移動 **70 px/s**、HP **30**。経路（タイル中心列）に沿って線形追従。
  - **0.5s 間隔**でスポーンし、生存数が上限(負荷値)未満のときのみ供給（常に上限付近を維持）。
  - ゴール到達で消滅し **Lives -1**。HP0 撃破で **Gold +5 / Score +10** ＋ `hit_spark` 表示。
- タワー: 射程 **96 px**、連射 **0.6s**、弾ダメージ **10**。毎フレーム射程内の
  **最も進行度の高い(ゴールに近い=残ウェイポイントが少ない)敵**を狙い `projectile` を発射。
- 弾: 弾速 **320 px/s** で発射時の標的方向へ直進。距離 < 敵半径10＋弾半径6 で命中・消滅・ダメージ。
- 資源: 初期 **Gold 120 / Lives 20 / Score 0**。Lives 0 で **GAME OVER**（`R`/クリックで再開）。
- 物理エンジンは未使用。経路追従・弾の直進・距離判定はすべて自前。デルタタイム基準（上限 0.05s クランプ）。
- 敵数(負荷): 初期 **30**、`+`/`-` で ±10（下限10・上限500）。

## 実装メモ ― 正射影2D + A\* + 多数ユニット追従

- **正射影カメラで 2D 画面を再現**：`orthoLeft/Right/Top/Bottom` を `0..960 / 0..540` に設定し
  （`orthoTop < orthoBottom` で y 下向き）、画面座標 = ワールド px を 1:1 で対応させた。テーマ5と異なり
  グリッド全体が 1 画面に収まるため**スクロールはせず ortho 窓は固定**。ロジックは終始「左上原点・+y下方向」
  の px 空間で完結する。
- **ポインタ → グリッドは単純な割り算**：ortho が px 等倍なので、`getBoundingClientRect()` で得た画面座標を
  `Math.floor(px / 32)` するだけでタイル座標になる（座標逆変換やレイキャスト不要）。CSS スケールにも備えて
  `canvas.width / rect.width` で補正している。右クリックは `contextmenu` を `preventDefault` して撤去に使う。
- **A\* (バイナリヒープ)**：4方向・コスト1・マンハッタン距離。open set は配列のバイナリヒープで実装し、
  `gScore/fScore/cameFrom/closed` を TypedArray で持って GC 圧を抑えた。`blocked` 集合に**タワーの占有セル**を
  渡すことで「壁＋タワー＝通行不可」を一括表現している。
- **経路の二段運用**：(1) 表示用のグローバル経路（スタート→ゴール）を `currentPath` に保持し薄い矩形で
  ハイライト。(2) 各敵は**自分の現在セルから**ゴールまでの経路を個別に持つ。タワー設置/撤去のたびに
  `recomputeGlobalPath(true)` でグローバル経路を更新し、**生存中の全敵が現在地から A\* を取り直す**
  （= `Path recalcs` が一気に増える）。これが本テーマの負荷の核。
- **設置の連結性保証**：タワーを仮に `towerSet` へ追加 → `hasPathOnGrid()` で A\* し、ゴール到達不能になるなら
  ロールバックして設置を拒否する。ゴール隣接の唯一の隘路など「塞ぐ配置」は弾かれ、必ず経路が残る。
- **経路追従**：敵はウェイポイント（タイル中心）列を線形に進む。1フレームの移動量が大きい場合に備え、
  `remain` を消費しながら複数ウェイポイントをまたぐループにしてある。末尾到達でゴール処理。
- **狙撃ターゲット選択**：タワーは射程内（距離²比較）で「残ウェイポイントが最も少ない＝ゴールに近い」敵を選ぶ。
  弾は発射時の方向へ直進し、標的優先＋全敵走査で距離命中判定する。
- **プール再利用**：敵 (creep) / 弾 (projectile) / スパーク (hit_spark) はスプライトプールで使い回し、
  上限 500 体まで増やしても生成/破棄コストを抑える。静的タイル(通路/壁)はリスタート時のみ再構築する。
- **フォールバック**：起動時に `new Image()` で各アセット URL を試し（`checkImage()`）、欠落なら
  `SpriteManager` の代わりに単色 `Plane`(矩形)/`Disc`(円) を `emissive`・`disableLighting` で表示する。
  統一ラッパ `{setPos,setVisible,setSize,setAlpha,dispose}` により Sprite と Mesh を区別せず扱える。
  経路ハイライトは画像を持たない専用キー `phl`（明るい青緑・半透明）で**必ず図形描画**になる。
- **z 重なり順**：タイル=5(奥)、ゴール/スタート=4、経路ハイライト=3、タワー=-1、敵=-2、弾=-3、スパーク=-4(手前)。

## Codex 生成所感 ― Babylon で グリッド経路探索TD を書く

- **最大の難所は描画ではなく「CPU側ロジック(A\*)と多数追従の同居」**。テーマ5までは描画/自前物理が主役だったが、
  本テーマは A\* 経路探索が中心。Babylon は3Dエンジンなので経路探索の支援は無く、バイナリヒープ A\* を素で書く。
  ここはエンジン非依存なので、どのエンジン版でもロジックは概ね共通になる。
- **3Dエンジンで 2D グリッドを扱う旨味は「ポインタ→セル変換の素直さ」**。ortho を px 等倍にすると、
  クリック座標は割り算1回でタイルになる。3D の picking/ray を使う必要がなく、テーマ5で培った ortho 設定が
  そのまま効く。スクロールしない分テーマ5よりカメラ周りは単純。
- **負荷の効き方が独特**。`+` で敵上限を上げると追従更新が増えるのは当然だが、**タワーを1つ置くだけで
  全敵が一斉に A\* を再実行**するため、`Path recalcs` が敵数分跳ね上がる。多数敵 × 頻繁な設置が最悪ケースで、
  ここが性能比較の見どころ。30x17 と盤面が小さいので A\* 自体は軽量だが、再計算回数で効かせる設計。
- **SpriteManager のインスタンシングは大量敵描画に強い**。500 体の creep + 多数の弾を出しても描画側は破綻しにくく、
  ボトルネックが「描画」ではなく「経路再計算」側に寄るので、ベンチ題材として CPU ロジックを綺麗に切り出せる。
- **設置の連結性保証は仕様の肝**。A\* を「仮置き→検証→ロールバック」に流用するだけで「経路を塞ぐ配置の禁止」を
  実装でき、追加のフラッドフィル等が不要だったのは A\* を持っている強み。総じて Babylon 固有の苦労は
  ortho セットアップとフォールバックくらいで、本テーマの本体はエンジン非依存のアルゴリズムだった。
