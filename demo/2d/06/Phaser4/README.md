# テーマ6 タワーディフェンス (経路探索) ― Phaser 4 版

グリッド上のタワーディフェンス。敵 (creep) がスタート地点から **A\* で求めた経路**を辿って
ゴール (自陣) へ向かい、タワーが射程内の敵を撃つ。タワーを置く / 撤去するたびに経路を再計算し、
**生存中の全 creep が現在地から経路を追従し直す**。本テーマの性能比較の核は、テーマ1〜5の
「描画/自前衝突」中心とは異なり、**CPU 側の AI ロジック (グリッド A\*) ＋ 多数ユニットの経路追従更新
＋ タワー射撃** を同時に回したときの負荷を測ることにある。

## 起動方法

ブラウザのセキュリティ制約 (file:// での CORS / アセット読込) を避けるため、
簡易 HTTP サーバ経由での起動を推奨します。

```bash
# このフォルダ (6/Phaser4/) で
python -m http.server 8000
# → http://localhost:8000/ をブラウザで開く
```

または任意の静的サーバ (`npx serve` など) でも可。
アセット画像が無くてもフォールバック描画で必ず起動します。

## 使用バージョン

- Phaser **4** (CDN: `https://cdn.jsdelivr.net/npm/phaser@4/dist/phaser.min.js`)
- 追加ライブラリなし。`index.html` + `game.js` のみ。
- **物理エンジン非使用** (Arcade 等を使わず、経路追従・弾の直進・距離判定をすべて自前実装)。

## 操作

| 入力 | 動作 |
|---|---|
| 左クリック | 通路タイルにタワーを設置 (コスト 25 Gold)。経路を塞ぐ配置は失敗 |
| 右クリック | タワーを撤去 (返金なし・経路を再計算) |
| `+` / `-` (テンキー可) | 同時出現する敵数の上限 (負荷) を 10 体ずつ増減 |
| `R` | リスタート (マップ・ゴールド・ライフ初期化) |

- 敵を撃破すると **Gold +5 / Score +10**、`hit_spark` 表示。
- 敵がゴールへ到達すると **Lives -1** で消滅。**Lives 0 で GAME OVER** (`R` で再開)。
- 右クリックメニューは `this.input.mouse.disableContextMenu()` で抑止。

## HUD

`setScrollFactor(0)` でカメラ非追従に固定し、左上に常時表示 (SPEC.md の必須項目を網羅)。

- `FPS` (実測・移動平均 30 サンプル)
- `Enemies` (生存数 / 上限) / `Towers` (設置数) / `Projectiles` (生存弾数)
- `Path recalcs` (経路再計算の累計回数) / `Path len` (現在のスタート→ゴール経路のタイル長)
- `Gold` / `Lives` / `Score`
- 画面下に操作ヒント行 (設置 / 撤去 / +/- / R)。中央に GAME OVER 表示。

## 実装メモ

### グリッド A\* (本テーマの核)

- `astar(map, idx, start, goal, isBlocked)` で 4 方向 A\* を実装。コストは一律 1、ヒューリスティックは
  マンハッタン距離。`isBlocked(x,y)` に「壁 **OR** タワーの置かれたセル」を渡し、敵はタワータイルも
  通行不可として扱う。
- オープンリストは **バイナリヒープ** (配列実装) で管理。多数の再計算 (敵数上限ぶん) に耐えるよう、
  `closed` フラグで重複ノードを排除しつつ最小 fScore を O(log n) で取り出す。
- **経路の保証**: マップ生成時に start→goal が必ず解けるよう A\* で検証し、塞いでしまった内部壁を
  決定的な走査順で間引く。タワー設置時も「仮設置 → A\* で再探索 → 解なしなら設置キャンセル」とし、
  **ゴールへ到達不能になる配置を禁止**する。
- 経路の再計算は次の 3 契機: ①設置成功、②撤去、③再計算後に**各生存 creep を現在セルから再探索**。
  いずれも `Path recalcs` カウンタを加算する (creep ごとの再探索も 1 回ずつ計上するため、敵数 ×
  設置回数に比例して増え、A\* 負荷がそのまま数字に表れる)。

### 多数ユニットの経路追従

- creep は **オブジェクトプール** (`active` フラグで使い回し) で管理。スポーンは start から 0.5s 間隔で、
  生存数が上限未満のときだけ供給し、常に上限付近の敵が経路追従している状態を維持する (ベンチ安定化)。
- 各 creep は「タイル中心の列」(px 座標) と現在向かうウェイポイント index `wp` を保持し、毎フレーム
  `delta` 秒ぶん線形に進む。1 フレームで複数ウェイポイントを跨ぐケースも `while` ループで消化する。
- タワー設置/撤去で経路が変わると `repathAllCreeps()` が全生存 creep に対し**現在セルから A\* を再計算**
  し、新しい経路の先頭から追従し直させる。現在セルがタワー/壁になっていた場合は 4 近傍の通行可能セルへ
  寄せてから探索する。

### タワー射撃 / 弾

- タワーは `Map` (セル id → {spr, fireTimer, ...}) で保持。毎フレーム射程 96px 内を走査し、
  **最も進行度が高い (ゴールまでの残りウェイポイント数が少ない) 敵**を狙って 0.6s 間隔で発射。
- 弾もオブジェクトプール。発射時の標的を追尾し、距離が 1 ステップ以下または `敵半径10 + 弾半径6` を
  下回ったら命中とみなしてダメージ 10。標的が先に消滅したら弾も破棄する。距離判定はすべて自前。

### 描画 (タイル / 経路オーバーレイ)

- マップは静的なので **Blitter** を通路用・壁用に 1 枚ずつ用意し、生成時に一括で Bob を配置
  (タイルは動かないため毎フレーム再構築は不要)。ゴールは単独スプライト。
- **経路ハイライト**は `Graphics` オーバーレイで「画像不要」に実装 (SPEC 指定)。現在経路の各セルを
  半透明矩形で塗り、タイル中心を線でつないで可視化する。ホバー中タイルにはタワー射程 96px の
  プレビュー円も `Graphics` で描く。
- 深度は depth で明示管理 (通路0 / 壁1 / base2 / 経路3 / 射程4 / タワー5 / creep6 / 弾7 / spark8 / HUD1000)。

### アセット / フォールバック

- `../assets/` (= `6/assets/`) から SPEC のファイル名 (`creep.png` `tower.png` `projectile.png`
  `tile_path.png` `tile_wall.png` `base.png` `hit_spark.png`) で読込。`BootScene` で `loaderror` を
  捕捉し、欠落キーは `Graphics.generateTexture` で焼いた図形にフォールバック
  (敵=赤丸 / タワー=青矩形 / 弾=黄丸 / 通路=濃灰 / 壁=灰 / ゴール=緑旗 / spark=白丸)。
  **画像が 1 枚も無くても必ず起動する。**

## Codex 生成所感 (タワーディフェンス + A\* の実装しやすさ)

- **A\* は素のロジック**: 経路探索は Phaser の API にほぼ依存せず純粋な JS で書けるため、エンジン差が
  出にくい部分。バイナリヒープ + `closed` フラグの定番構成をそのまま書けばよく、Codex 生成でも安定した。
  むしろ難所は「再計算の契機を漏れなく拾う」「全 creep を一斉に追従し直させる」というゲーム側の配線で、
  ここを `repathAllCreeps()` に集約したことで `Path recalcs` の計上も一箇所にまとまった。
- **ポインタ入力**: `this.input.on('pointerdown', ...)` + `pointer.rightButtonDown()` で左右クリックを
  判別し、`this.input.mouse.disableContextMenu()` で右クリックメニューを抑止する流れは数行で済む。
  タイル座標は `Math.floor(pointer.x / TILE)` で直接得られ、カメラ非スクロールなので変換も不要だった。
- **Blitter と Graphics の使い分け**: 静的タイルは Blitter で一括バッチ、動的な経路ハイライトと射程
  プレビューは Graphics オーバーレイ、と役割を分けられた。経路の可視化を「画像不要」で実装する要件に
  Graphics がそのまま噛み合う。
- **物理エンジン非使用**: 経路追従 (線形補間) も弾の直進・距離当たりも自前で、Arcade を持ち出す必要は
  なかった。多数ユニットを毎フレーム素朴に更新するループはオブジェクトプールと相性がよく、500 体規模の
  負荷比較でも素直にスケールする。
- **深度管理**: レイヤを Container / Blitter / Graphics で分け depth を明示するだけで重なり順が安定し、
  HUD を `setScrollFactor(0)` + 高 depth で固定する手順もテーマ5と共通。
- 総じて Phaser 4 は「グリッド AI ＋ 多数ユニット ＋ ポインタ操作 ＋ 図形オーバーレイ」を、描画 (Blitter/
  Graphics) と入力 (pointer events) の標準 API だけで迷いなく両立でき、A\* 本体はエンジン非依存に書けた。
