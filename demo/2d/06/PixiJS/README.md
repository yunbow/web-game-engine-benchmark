# テーマ6 タワーディフェンス（経路探索）― PixiJS v8 版

グリッド上のタワーディフェンスを **PixiJS v8** で実装。30×17 タイル（960×544px）の
固定マップ上で、敵（creep）がスタート→ゴールを **A\* 経路探索**で辿り、タワーが射程内の
敵を撃つ。タワー設置/撤去のたびに**全敵が現在地から一斉に再経路付け**するため、
「同時に経路追従している敵数 × 経路再計算回数」を負荷として測る性能比較用デモ。

本テーマの主軸はテーマ1〜5の「描画／自前衝突」と異なり、**CPU 側 AI ロジック
（A\* 経路探索 ＋ 多数ユニットの追従更新 ＋ タワー射撃）** にある。

## 起動方法

ローカルにHTTPサーバを立ててブラウザで開く（`PIXI.Assets.load` が `file://` では CORS で失敗するため）。

```bash
# このフォルダ(6/PixiJS/)の1つ上(6/)で実行すると ../assets/ も解決できる
cd c:/work/claude/local-works/web-game/6
python -m http.server 8000
# → http://localhost:8000/PixiJS/ を開く
```

`../assets/` に画像が無くても **Graphics 図形フォールバック** で必ず起動する。

## 使用バージョン

- PixiJS **v8**（CDN: `https://cdn.jsdelivr.net/npm/pixi.js@8/dist/pixi.min.js`）
- 追加ライブラリなし。`index.html` + `game.js` の2ファイル構成。
- **v8 注意**: 初期化は `const app = new PIXI.Application(); await app.init({...});` の2段階。
  キャンバスは `app.view` ではなく **`app.canvas`**。Graphics は新チェーンAPI
  （`gr.rect(...).fill(color)` / `.stroke({width,color})` / `.circle(...)` / `.poly(...)`）。
  フォールバックは `app.renderer.generateTexture({ target: graphics })` でテクスチャ化し Sprite 再利用。
- **マウス入力は `app.canvas` の DOM `pointerdown`** を直接使用。`event.button`（0=左 / 2=右）で
  設置／撤去を判別し、`app.canvas.addEventListener('contextmenu', e => e.preventDefault())` で
  右クリックメニューを抑止する。
- **物理エンジンは未使用**。A\*・経路追従・弾の直進・距離判定はすべて自前。

## 操作

| 入力 | 動作 |
|---|---|
| **左クリック** | 通路タイルにタワーを設置（25 Gold）。経路を塞ぐ配置は失敗 |
| **右クリック** | タワーを撤去（返金なし・経路再計算） |
| `+` / `-`（テンキー可） | 同時出現する敵数の上限を ±10（初期30・下限10・上限500） |
| `R` | リスタート（マップ・Gold・Lives 初期化） |

## 仕様数値（全エンジン共通）

- キャンバス **960×540**、タイル **32×32**、グリッド **30×17**。
- マップは固定シード（`mulberry32(20250615)`）で**決定的生成**。
  `0=通路(配置可) / 1=壁(配置・通行不可)`、外周は壁・内部に散らした壁ブロック。
  スタート=左端中央、ゴール=右端中央。生成後に A\* で疎通確認し、塞いだ壁は除去して必ず通れる状態にする。
- **経路探索**: グリッド **A\***（4方向・コスト1・**マンハッタン距離ヒューリスティック**）。
  壁**および**タワーの置かれたタイルを通行不可として扱う。設置/撤去のたびに再計算。
- **敵（creep）**: 半径 **10**、移動 **70 px/s**、HP **30**。経路（タイル中心列）を線形追従。
  - ゴール到達 → 消滅 **Lives -1**。HP0 で撃破 → **Gold +5 / Score +10** ＋ `hit_spark` 表示。
  - タワー設置/撤去で経路が変わると、**各敵は現在セルから再計算した経路に追従し直す**（再計算カウンタ +1）。
- **タワー**: 射程 **96 px**、連射間隔 **0.6 s**、弾ダメージ **10**。射程内の**最も進行度が高い**敵を狙う。
  - 弾速 **320 px/s**。敵中心との距離 < 敵半径10 + 弾半径6 で命中・消滅。
- **ウェーブ**: 0.5s ごとにスタートからスポーン。生存数が上限（負荷値）未満のときのみ供給。
- **資源**: 初期 **Gold 120 / Lives 20 / Score 0**。Lives 0 で **GAME OVER**（`R` で再開）。

## HUD（HTMLオーバーレイ `#hud`・約120msごと更新）

- `FPS`（直近60フレームの移動平均）
- `Enemies`（生存数 / 上限）/ `Towers`（設置数）/ `Projectiles`（生存弾数）
- `Path recalcs`（経路再計算の累計回数）/ `Path len`（現在のスタート→ゴール経路のタイル長）
- `Gold` / `Lives` / `Score`

HUD はキャンバス外の HTML `<pre>` 要素で描画。Pixi の描画負荷と切り離して計測できる。

## 実装メモ

- **A\*（本デモの核）**: `aStar(map, sx, sy, gx, gy, towerBlocked)` を自前実装。30×17=510 セルと
  小さいので、open リストは「配列を都度 min を線形探索」する素朴な優先度キューで十分軽い。
  `closed` / `inOpen` / `gScore` / `fScore` を TypedArray で持ち、`came` 配列から経路を復元する。
  壁は `map`、タワーは `towerBlocked`（`Set<cellId>`）の両方を通行不可として判定。
- **マップ疎通保証**: 決定的に壁ブロックを撒いたあと、A\* がスタート→ゴールの解を返すまで
  「中央水平ラインの壁」を左から順に1つずつ通路化する素朴ループで、必ず通れる初期マップにする。
- **タワー設置の検証**: 通路タイル / Gold 充足 / スタート・ゴールセルでない を確認後、
  **仮設置（`towerBlocked` に追加）して A\* を回し、解が残るときだけ確定**。塞ぐ配置はロールバックして拒否。
  確定時は経路を更新し、`pathRecalcs++` ＋ **全 creep を `repathCreep`（現在セルから A\*）** で追従し直す。
- **creep 経路追従**: 各 creep は `path`（タイル列）と `wp`（次に向かうウェイポイント index）を持つ。
  毎フレーム `CREEP_SPEED*dt` ぶんを「現在ウェイポイントへ向けて」消費し、到達したら `wp++`。
  1フレームで複数ウェイポイントを跨げるよう while ループで残り移動量を配分する。
- **タワー射撃**: 射程内（距離²で比較）で **`wp`（消化済みウェイポイント数＝進行度）が最大の敵**を狙い、
  目標方向へ弾を直進発射。弾は最近接の敵に当たると消滅しダメージを与える（距離ベース簡易判定）。
- **プール再利用**: creep / projectile / spark の各 Sprite は破棄せず `*Pool` 配列へ退避して再利用。
  上限500体・大量の弾でも Sprite 生成は初回のみで、GC とドローコール増を抑える。
  タイルは 30×17 全セルぶんの Sprite を最初に確保し、種別に応じてテクスチャを貼り分ける（固定）。
- **経路ハイライト**: 現在のスタート→ゴール経路を `pathLayer` の `Graphics` に薄い塗り＋中心線で重ねる
  （`redrawPath`）。毎フレームではなく**経路更新時のみ**再描画してコストを抑える。
- **マウス座標→タイル**: `app.canvas.getBoundingClientRect()` で CSS スケールを補正し、`world` の
  平行移動ぶんを引いてタイル座標へ変換。
- **フォールバック**: 各アセットを `PIXI.Assets.load` で試行し、失敗時は `generateTexture` 生成の
  図形テクスチャに差し替え（敵=赤丸 / タワー=青矩形 / 弾=黄丸 / 通路=濃灰 / 壁=灰 / ゴール=緑旗 / spark=白丸）。

## Codex 生成所感（v8 で経路探索 TD を組む観点）

- **v8 初期化の罠（再掲）**: `new PIXI.Application(opts)` は無視され `app.canvas` が生えない。
  必ず `new PIXI.Application()` → `await app.init(opts)` の2段階。`app.view` も `app.canvas` に改名。
- **マウス入力は DOM イベントで素直に取れる**: Pixi の `eventMode`/`FederatedPointerEvent` を使わず、
  `app.canvas` に直接 `pointerdown` を張り `event.button` で左右判別＋`contextmenu` 抑止が一番手堅い。
  CSS スケール（等倍縮小表示）ぶんは `getBoundingClientRect()` で補正する点だけ注意。
- **描画はほぼタダ・負荷は CPU 側の A\* と追従に集中**: Pixi は当たり判定にも経路にも一切干渉しない
  ので、A\*・経路追従・射撃を完全に論理側で組める。500体 × タワー設置のたびに全敵再計算という
  最重ケースでも、510セルの A\* は素朴実装で十分軽く、`Path recalcs` を HUD で完全に可視化できた。
  ボトルネックは「再計算回数 × セル数」で素直に効くので、`+/-` での負荷比較が安定して取れる。
- **プール再利用が効く題材**: creep/弾は生成消滅が激しいが、Sprite を退避再利用すれば
  ドローコールと GC を抑えられる。タイルは全セル固定 Sprite で貼り分けるだけで済む。
- **経路ハイライトを Graphics 1枚で**: 現在経路を `pathLayer` の単一 `Graphics` に重ね、
  経路更新時のみ `clear()`→再構築。毎フレーム描き直さないので大量敵下でも軽い。
- 総じて v8 は「描画は超軽量・ロジックは完全自前」という分担が明快で、**経路探索ベンチの
  題材としては A\* と追従更新をストレートに JS で書け、負荷を HUD でクリーンに計測できる**構成だった。
