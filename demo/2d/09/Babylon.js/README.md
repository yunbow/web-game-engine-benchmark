# テーマ9: アイソメトリック都市/農場（深度ソート × タイル奥行き描画）― Babylon.js 版

斜め見下ろし（アイソメトリック）のマップ上で、地面タイル・木・家・歩き回るユニットを
正しい前後関係で描く都市/農場風シーンを **Babylon.js** で実装したもの。
2Dシーンだが Babylon は3Dエンジンのため、**正射影(Orthographic)カメラ + SpriteManager/Sprite**
で 2D アイソメ画面を px 等倍で再現している。「3Dエンジンで アイソメ投影 ＋ 可視カリング ＋
**可視オブジェクトの毎フレーム深度ソート（z-order）** ＋ 多数の移動ユニット を同時に回した性能」を
測るのが主眼。トップダウン直交カリング（テーマ3/5）とは別軸の、**斜め投影＋毎フレーム前後ソート**コストを比較する。

## 起動方法

`../assets/` を相対パスで読むため、ローカルHTTPサーバ経由で開く（file:// でも起動はするが
画像が読めずフォールバック図形になる）。

```bash
# このフォルダ (9/Babylon.js) の1つ上 (9/) で実行すると ../assets も配信される
cd c:/work/claude/local-works/web-game/9
python -m http.server 8000
# ブラウザで http://localhost:8000/Babylon.js/ を開く
```

または VS Code の Live Server などでも可。
画像アセットが無い環境でも **菱形ポリゴン等の図形でフォールバック**して必ず起動する。

## 操作

| キー | 動作 |
|---|---|
| `←` `→` `↑` `↓` / `WASD` | カメラスクロール（ワールド中心を移動・クランプ） |
| `+` / `-` | 歩き回るユニット数を ±20（下限0・上限2000） |
| `G` | グリッド線表示トグル（可視タイル菱形の境界線） |
| `R` | リセット（カメラ中心とユニットを初期化） |

## 仕様準拠（数値）

- キャンバス **960x540** 固定。アイソメ菱形 **TILE_W=64 / TILE_H=32**。
- マップ **64x64 タイル**。決定的擬似乱数 (mulberry32, seed=0x1509) で生成。
  - 地面 `0=草(grass)`, `1=土/畑(soil)`, `2=水(water)`：全面草 → 楕円の池(5個)を水に → 矩形の畑区画(8個)を土に。
  - 静的オブジェクト：草地に確率10%で木、草/畑に確率3%で家を散布（**水の上には置かない**）。
- **アイソメ投影（SPEC 共通式）**：`screenX=(gx-gy)*32`, `screenY=(gx+gy)*16`、**深度キー=gx+gy**。
- カメラ：中心のワールド grid 座標 `(camGx,camGy)` を `[0,63]` でクランプ。スクロールは ortho 窓の平行移動。
- ユニット：初期 **60**、`+`/`-` で **±20**（0..2000）。移動 **40px/s 相当**（grid 連続座標で約1.25 tile/s）。
  - 各ユニットは固定シードの `mulberry32` を1個持ち、決定的な目的地へ直進、着いたら次の決定的目的地へ。
    **水タイルは避ける**（目的地候補は水・障害物でないタイルから決定的に選ぶ／踏みそうなら選び直す）。
- デルタタイム基準（`engine.getDeltaTime()/1000`, 上限 0.05s でクランプ）。Math.random 不使用。

## HUD（HTMLオーバーレイ, 約0.1s更新）

- `FPS`（指数移動平均）
- `Tiles drawn`（そのフレームで描画した可視地面タイル数）
- `Objects sorted`（深度ソートした可視オブジェクト＋ユニット数）
- `Units`（現在数 / 設定値）
- `Camera`（カメラ中心のワールド grid 座標 `(gx, gy)`）
- `Render`：`Sprite(tex)`（画像あり）/ `Mesh(fallback)`（画像欠落→図形描画）

## 実装メモ ― アイソメ投影 / 深度ソート / カリング

### アイソメ投影と正射影カメラ
- 正射影カメラの `orthoLeft/Right/Top/Bottom` を `0..960 / 0..540` に設定し（`orthoTop<orthoBottom`
  で y 下向き）、画面座標 = ワールド px を 1:1 対応させる。SPEC の式 `screenX=(gx-gy)*32`,
  `screenY=(gx+gy)*16` をそのまま使い、`isoX/isoY` でワールド px を求める。
- **カメラスクロールは ortho 窓の平行移動**：カメラ中心 grid を ortho 窓の中心へ写し、
  毎フレーム `orthoLeft/Right/Top/Bottom` を画面オフセット分ずらして窓を動かす。
  スクロール入力（px 移動量）はワールド px → grid（`sum=py/16=gx+gy`, `dif=px/32=gx-gy`）へ
  逆変換して `(camGx,camGy)` を `[0,63]` でクランプし、はみ出しを防ぐ。

### ★ Babylon での深度順の実現（本テーマの最難所）★
Babylon の `Sprite` は **同一 SpriteManager 内で描画リストへ積まれた順に近い順序で描かれ、
画面 y による自動深度ソートはしてくれない**。複数 SpriteManager 間や Sprite と Mesh の混在では
なおさら描画順を制御したい。そこで本実装は **「可視オブジェクトを gx+gy でソートし、その順に
`position.z` を単調割り当てする」** 方式を採った（SPEC が推奨する最も堅牢な手法 (a)）。

- 正射影でも **z は深度バッファ(depthTest)に効く**。カメラは `z=-100` で +z 方向を見るので
  **z が大きいほどカメラから遠い（奥）**。Sprite/Mesh いずれも z でソートして描けば正しく遮蔽される
  （テーマ5でも背景=z60 / タイル=z0 / 敵=z-2 / 自機=z-3 と同じ z 手動割当てで層を作っており、本テーマはその発展）。
- **2層方式**（SPEC 推奨）：
  1. **地面タイル**：高さ0で互いに重ならないため**ソート不要**。可視範囲だけ種別ごとの
     SpriteManager + プールへ割り当て、`z=GROUND_Z(50)` の**最奥固定**で描く（カリングのみ）。
  2. **オブジェクト(木/家)＋ユニット**：背が高く前後が入れ替わるため、**可視分だけを毎フレーム
     `drawList` に集めて `gx+gy` 昇順にソート**し、ソート順 `i=0..n-1` に対して
     `z = OBJ_Z_FAR(40) → OBJ_Z_NEAR(5)` を線形補間で割り当てる（奥=大きい z、手前=小さい z）。
     同 depth はオブジェクト→ユニットの順で安定ソート。ソート数を `Objects sorted` に出す。
- **なぜ全タイルをソートしないか**：64x64=4096 タイル（＋オブジェクト）を毎フレームソートすると重い。
  地面はソート不要・オブジェクト/ユニットのみソート、という2層分割で「再ソート対象 = 可視オブジェクト＋ユニット」
  に絞れる。`+` でユニットを増やすほどこの再ソート対象が増え、ソート/再配置コストが `FPS` に現れる。

### カリング（可視範囲だけを対象）
- 画面四隅のワールド px を `(gx+gy)=py/16` と `(gx-gy)=px/32` に逆変換し、その min/max から
  可視 grid 矩形 `[gxMin..gxMax, gyMin..gyMax]`（＋余白2タイル）を求める（`visibleGridBounds`）。
- その候補矩形の各タイルについて、さらに**タイル中心の画面座標が画面内か**を菱形半分の余白付きで
  判定して描画／スキップする（粗い AABB → 精密判定の2段）。`Tiles drawn` は最終的に描いた実数。
- オブジェクト/ユニットも同じ可視 grid 矩形＋画面内判定で間引いてから `drawList` に積む。
  画面外のユニットは `isVisible=false` にしてソート対象からも外す。

### フォールバック図形
- 起動時に `new Image()` で各アセット URL を試し（`checkImage`）、欠落なら SpriteManager の代わりに
  単色 Mesh を `emissive`・`disableLighting` で表示する。統一ラッパ `{setPos,setVisible,setSize,dispose}`
  により Sprite と Mesh をロジックから区別せず扱える（テーマ5と同型）。
- **アイソメ菱形**は4頂点ポリゴン（上/右/下/左）を `VertexData` で自作（草=緑/土=茶/水=青）。
  木=茶幹(矩形)＋緑円(`Disc`)、家=灰本体＋茶の菱形屋根、ユニット=橙円(`Disc`)。
  いずれも単位サイズで作り `MergeMeshes`／`clone`＋`scaling` で実サイズへ。

### グリッド線（G）
- `G` トグル時のみ、可視タイルの菱形4辺を `LineSystem` で `z=GRID_Z(48)`（地面とオブジェクトの中間）に描く。
  カメラ移動で位置が変わるため毎フレーム作り直す（オフ時は何も生成しない簡易実装）。

## Codex 生成所感 ― Babylon でアイソメ深度ソートを書く

- **最大の難所は「Babylon の Sprite が y で自動深度ソートしない」点の解決**。Pixi の `zIndex+sortableChildren`
  や Phaser の `depth`、Godot の `YSort` のような専用機構が Sprite には無い。結局
  「**可視分を `gx+gy` でソートして `position.z` を painter 順に単調割り当て**」が一番素直で堅牢だった。
  正射影でも z が深度バッファに効くので、Sprite/Mesh 混在でも z 一本で前後関係を統一できる。
- **2層分割（地面=ソート不要・最奥固定 / オブジェクト＋ユニット=毎フレーム再ソート）が効く**。
  地面は高さ0で重ならないので最奥固定で済み、ソート対象を可視オブジェクト＋ユニットだけに絞れる。
  これで `+` でユニットを増やしたときのコスト増加が「再ソート＋再配置」に集中し、ベンチの軸として素直に効く。
- **アイソメのカリングは逆変換が肝**。直交グリッドと違い「画面矩形 → grid 矩形」が回転した菱形になるので、
  `(gx+gy)`/`(gx-gy)` の min/max から候補矩形を作り、各タイルで精密判定する2段構えにした。
  余白を取らないと菱形の角がスクロール時にちらつく。
- **大量スプライト描画は3Dエンジン由来の強み**。SpriteManager のインスタンシングで地面 600 枚前後＋
  ユニット数百体を描いても破綻しにくい。ボトルネックは描画よりむしろ「毎フレームの JS 側ソート＋
  z 割り当て」で、ユニット数を上げると `Objects sorted` の増加に比例して `FPS` が落ちる挙動が観測しやすい。
- 総じて Babylon は、深度順を自前で組む手数はかかるが、正射影のピクセルパーフェクト性と
  スプライト大量描画性能のおかげで、アイソメ深度ソートのベンチ題材として素直に成立する。
