# テーマ9 アイソメトリック都市/農場（深度ソート × タイル奥行き描画）― LittleJS 版

斜め見下ろし（アイソメトリック）の都市/農場シーンを **LittleJS** で実装したもの。
64x64 タイルの決定的マップ（草/土/水＋木/家）を可視範囲だけカリングして描き、
**アイソメ投影 ＋ 可視カリング ＋ 多数ユニットの毎フレーム深度ソート（painter's algorithm）**
の同時負荷を測る。比較の核は「**毎フレーム再ソートする可視オブジェクト＋ユニット数**」。

## 起動方法

CDN 読込のため `file://` で直接開いても動くが、`../assets/` の相対参照と
ブラウザのキャッシュ挙動を安定させるため簡易HTTPサーバ経由を推奨。

```bash
# このフォルダ (9/LittleJS) で
python -m http.server 8000
# → ブラウザで http://localhost:8000/ を開く
```

`../assets/` フォルダは空でも起動する（画像欠落時は菱形ポリゴン等の図形フォールバック）。
SPEC のアセット（`tile_grass.png` 等）を `../assets/` に置けば自動で使用される。

## 使用バージョン / CDN

- **LittleJS**（`@latest` 解決時点 / classic global build）
- 主URL: `https://unpkg.com/littlejsengine`
  失敗時フォールバック: `https://cdn.jsdelivr.net/npm/littlejsengine/dist/littlejs.min.js`
- 素の unpkg は classic global build（グローバル `engineInit` 等を生やす版）を解決するため
  `<script>` 直読みできる。読込失敗時のみ `error` イベントを捕捉して jsDelivr ミラーへ切替。

## 操作

- **← → ↑ ↓ / WASD**: カメラをワールド上でスクロール（中心はマップ内にクランプ）。
- **+ / -**（メイン行 or テンキー）: 歩き回るユニット数を ±20（下限 0・上限 2000・初期 60）。
  性能比較の主軸（毎フレーム再ソートする対象が増える）。
- **G**: グリッド線（菱形の輪郭）表示トグル。
- **R**: リセット（マップ・オブジェクト・ユニットを初期状態へ再生成）。

## 仕様数値（全エンジン共通）

| 項目 | 値 |
|---|---|
| キャンバス | 960 x 540 固定 |
| タイル菱形 | TILE_W=64, TILE_H=32 |
| マップ | 64 x 64（決定的生成）|
| 地面 | 0=草 / 1=土・畑 / 2=水 |

## HUD（常時表示・左上, HTML `#hud` overlay）

- `FPS`: 実測の移動平均（エンジン `frameRate` を指数移動平均で平滑化）。
- `Tiles drawn`: そのフレームで描画した可視地面タイル数（カリング後）。
- `Objects sorted`: 深度ソートした可視オブジェクト＋ユニット数（比較の核）。
- `Units`: 現在のユニット数 / 設定値。
- `Camera(gx,gy)`: カメラ中心のワールド・グリッド座標。
- `[sprites]` / `[shapes fallback]`: 画像使用かフォールバックかを明示。

## 実装メモ

### アイソメ投影（SPEC 共通式）
グリッド `(gx, gy)` → 画面オフセット px:

```
isoX = (gx - gy) * (TILE_W/2)   // = (gx-gy)*32
isoY = (gx + gy) * (TILE_H/2)   // = (gx+gy)*16
depth = gx + gy
```

カメラ中心 `(camGX, camGY)` の iso オフセットを引いて画面中心（ワールド原点）へ合わせる。
逆変換 `gx = (ix/HALF_W + iy/HALF_H)/2, gy = (iy/HALF_H - ix/HALF_W)/2` で
画面四隅 → グリッド窓を求め、可視範囲カリングに使う。

### Y軸 / 座標系（最重要）
SPEC のアイソメ式は**画面ピクセル・Y下向き**（`isoY` が大きいほど画面下）で定義される。
一方 LittleJS のワールドは **Y軸が上向き**。テーマ5 同様 `cameraScale=1`（1ワールド単位=1px）とし、
iso 計算・カリング・深度ソートはすべて **Y下向き px の一貫モデル**で行い、
描画関数（`drawTile`/`drawRect`/`drawPoly`）へ座標を渡す**一点だけ**で Y 符号を反転する
（`worldFromGrid()` が `vec2(ix, -iy)` を返す）規約に閉じ込めた。
これで「アイソメで奥（`isoY` 小）= 画面上 = LittleJS の Y 大」という自然な見た目になり、
「上を押すと下に行く」「片側だけカリングが欠ける」事故を防いだ。
背の高い木/家/ユニットは接地点（セル中心）から上（＝ Y+）へ伸ばして描く。

### 深度ソートの実現方法（painter's algorithm）
**LittleJS には Pixi の `zIndex` や Phaser の `depth`、Godot の `YSort` のような z-order 機構は無い。**
描画順は「`draw*` 関数を呼んだ順序」そのもので決まる（後に呼んだもの＝手前）。
そこで SPEC 推奨の **2層方式**を採る:

1. **第1層 — 地面タイル**（`gameRender`）: 可視範囲を二重ループでカリングし、`drawTile`/`drawIsoDiamond`
   で描く。地面は高さ0で重ならないため**ソート不要**。実描画数を `Tiles drawn` に集計。
2. **第2層 — 可視オブジェクト＋ユニット**（`gameRenderPost`）: 可視範囲に入るものを配列 `drawList`
   に収集し、`depth = gx + gy` **昇順**に `Array.sort`（同 depth は静的オブジェクト→ユニットで安定化）。
   その順に奥→手前で描く。ユニットは連続座標で `gx+gy` が毎フレーム変わるため**毎フレーム再ソート**が要る
   （ここが性能比較の核）。ソート対象数を `Objects sorted` に出す。

地面を先・高さのあるものを後に描くので、ユニットが家/木の手前/奥に来たとき自然に遮蔽される。

### カリング
カメラ中心 ± 画面半分（+余白2タイル）を iso 逆変換してグリッド窓 `[gxMin..gxMax]×[gyMin..gyMax]`
を求め、地面はこの範囲だけ描画、オブジェクト/ユニットも範囲外を `continue` でスキップ。
全 4096 タイルや全ユニットをソートせず、**可視分だけ**を `gx+gy` でソートする。

### ユニット（決定的うろつき）
各ユニットは固定シードを持ち、`pickTarget()` で決定的に次の目的地グリッドへ向かう（水タイルは
選び直し）。到達したら次の決定的目的地へ。進行先が水セルなら目的地を選び直して回避する。
速度は 40px/s 相当（画面1グリッドステップ≒ `TILE_W/2` px とみなして grid/s に換算）。
`+`/`-` は事前生成した候補配列の**先頭 N 体**を採るので、増減しても配置・並びが決定的。

### 決定的生成 / 図形フォールバック
`mulberry32` の決定的乱数（マップ `20260615`、畑/木/家/ユニットも別シード固定）。`Math.random` 不使用。
川は対角＋正弦の蛇行、池・畑パッチ・木・家の集落を決定的に散布。
`textureInfos[i].size.x > 1` で画像読込を判定（`spriteReady`）。読込済みなら `tile(0,size,i)` で `drawTile`、
未読込なら図形（地面=菱形 `drawPoly` 4頂点ポリゴン[草=緑/土=茶/水=青] / 木=茶幹＋緑円 /
家=灰箱＋赤屋根 / ユニット=橙丸）。**画像が1枚も無くても必ず起動する。**

## Codex 生成所感（アイソメ深度ソートを軽量エンジンで書く所感・罠）

- **「z-order 機構が無い」ことが最大の論点**。Pixi/Phaser/Godot は zIndex/depth/YSort を立てれば
  エンジンが並べ替えてくれるが、LittleJS は**描画呼び出し順がそのまま z**。逆に言えば
  painter's algorithm を自分で書くだけで、ソートコスト（`Objects sorted` 件数の `Array.sort`）が
  どこで効くかが完全に可視化される。性能比較用途ではむしろ素直だった。
- **Y軸上向き × アイソメ Y下向き式の二重座標が事故りやすい**。SPEC 式は Y下向き前提なので、
  iso 計算は Y下向きのまま貫き、`worldFromGrid()` の出口一点でだけ `-iy` する規約に閉じ込めた。
  カリングの逆変換も同じ Y下向き系で書けるので、符号の混入を最小化できた。
- **菱形フォールバックは `drawPoly` の4頂点で素直に描ける**。当初は回転矩形での近似を考えたが、
  classic build に `drawPoly(points, color, lineWidth, lineColor, pos)` があり、上/右/下/左の
  4頂点ローカルオフセット＋`pos` 平行移動で正しい 64x32 菱形が出せた。グリッド線も同 API の
  線幅指定（塗り `CLEAR_WHITE`）で実現。
- **カリングは斜め投影だと「矩形のグリッド窓」では完全一致しない**（画面の長方形に対しグリッド窓は
  菱形領域）。余白を2タイル足した bounding box で安全側に倒し、`Tiles drawn` がやや多めに出るが
  抜けは無い、という割り切り。厳密な菱形カリングは比較の本質ではないので採らなかった。
- 総じて LittleJS は「薄い API を正しく組む」エンジン。深度ソート機構を持たない分、
  2層 painter's を手書きすることになるが、ソート対象数と FPS の相関が HUD に直結して出るのが、
  アイソメ深度ソートのベンチ題材として扱いやすかった。
