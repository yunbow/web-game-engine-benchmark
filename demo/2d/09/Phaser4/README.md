# テーマ9 アイソメトリック都市/農場（深度ソート × タイル奥行き描画）― Phaser 4 版

斜め見下ろし（アイソメトリック）のマップ上に、地面タイル・木・家・歩き回るユニットを
**正しい前後関係（深度ソート）** で描く都市/農場風シーン。64x64 タイルのマップを
**画面に入るアイソメ範囲だけカリング**して地面を描き、背の高いオブジェクト＋移動ユニットを
**毎フレーム `gx+gy` 昇順に再ソート**して Phaser の `depth` で重ね順を決める比較用サンプル。
テーマ3のトップダウン直交カリングとは別軸の「斜め投影＋毎フレームの前後ソート」コストを測る。

## 起動方法

ブラウザのセキュリティ制約（file:// での CORS / アセット読込）を避けるため、
簡易 HTTP サーバ経由での起動を推奨します。

```bash
# このフォルダ (9/Phaser4/) で
python -m http.server 8000
# → http://localhost:8000/ をブラウザで開く
```

または任意の静的サーバ（`npx serve` など）でも可。
アセット画像が無くてもフォールバック図形描画で必ず起動します。

## 使用バージョン

- Phaser **4**（CDN: `https://cdn.jsdelivr.net/npm/phaser@4/dist/phaser.min.js`）
- 追加ライブラリなし。`index.html` + `game.js` のみ。

## 操作

| キー | 動作 |
|---|---|
| `←` `→` `↑` `↓` または `WASD` | カメラ（画面オフセット）をスクロール。中心はワールド内にクランプ |
| `+` / `-`（テンキー可） | 歩き回るユニット数（負荷）を ±20 体 |
| `G` | グリッド線オーバレイのトグル |
| `R` | リセット（決定的に同じ初期状態へ） |

勝敗は無く、描画統計を HUD に出す箱庭デモ。

## 仕様数値表（全エンジン共通）

| 項目 | 値 |
|---|---|
| 画面 | 960 x 540（固定） |
| タイル菱形 | TILE_W=64 / TILE_H=32 |
| マップ | 64 x 64（決定的生成・固定シード） |
| 地面種別 | 0=草(grass) / 1=土(soil) / 2=水(water) |
| 静的オブジェクト | 木 / 家（水タイル以外に決定的散布） |
| ユニット移動速度 | 40 px/s 相当（グリッド連続座標） |
| ユニット数 | 初期 60 / ±20 / 下限 0・上限 2000 |
| 深度キー | `gx + gy`（小=奥・先に描く / 大=手前・後に描く） |

## HUD

`setScrollFactor(0)` でカメラ非追従に固定し、左上に常時表示。

- `FPS`（実測・移動平均 30 サンプル）
- `Tiles drawn`（可視地面タイル数）
- `Objects sorted`（深度ソートした可視オブジェクト＋ユニット数 ＝ ベンチの核）
- `Units`（現在 / 設定値）
- `camera world`（カメラ中心のワールドグリッド座標 `(gx, gy)`）
- 画面下に操作ヒント行。

## 実装メモ

### アイソメトリック投影（SPEC 厳密準拠）

- グリッド座標 `(gx, gy)` → 画面ローカル座標:
  - `screenX = (gx - gy) * (TILE_W/2) = (gx - gy) * 32`
  - `screenY = (gx + gy) * (TILE_H/2) = (gx + gy) * 16`
- 画面中央にワールド中心 `(camGX, camGY)` が来るよう、描画時にカメラオフセット
  `ox = 960/2 - isoX(camGX,camGY)`, `oy = 540/2 - isoY(camGX,camGY)` を全座標に加算する。
- カリングのための **逆投影**（オフセット除去後の `sx,sy` → `gx,gy`）:
  - `gx = sx/64 + sy/32`, `gy = sy/32 - sx/64`。
  画面四隅（背高オブジェクトぶんの余白付き）を逆投影し、その AABB をグリッド範囲として走査する。

### 深度ソート（Phaser `depth`）― ベンチの核

- SPEC 推奨の **2 層方式**:
  1. **地面**（高さ0・菱形同士は重ならない）はソート不要。種別ごとの **Blitter** に
     可視ぶんだけ Bob を作って描く（`Tiles drawn`）。
  2. **背の高いオブジェクト（木/家）＋ユニット** は可視ぶんを 1 本のリストに集め、
     `gx + gy` 昇順に安定ソートして共有 Image プールへ反映。各スプライトに
     `setDepth(10 + (gx+gy) + i*1e-4)` を設定し、Phaser の depth 機構で前後関係を実現する
     （同 `gx+gy` 値は並び順の微小オフセットで安定化）。`Objects sorted` に対象数を表示。
- **ユニットは連続グリッド座標**を持ち、毎フレーム `gx+gy` が変化するため
  **毎フレーム再ソート**が必要になる（ここがエンジン間比較の核）。ユニット数を `+`/`-` で
  増やすほど、毎フレームソート＋depth 反映する可視スプライト数が増え `FPS` に差が出る。
- スプライトは静的オブジェクトとユニットを **単一の共有プール**に混在させ、
  ソート後の並び順に位置・テクスチャ・depth を上書きして使い回す（プールはフレーム間で再利用）。
  足元を菱形の床中心に合わせるため `origin=(0.5, 1)` で `y` を `isoY + TILE_H/2` に置く。

### カリング（地面・オブジェクト共通）

- 地面は毎フレーム可視グリッド範囲 `(gx0..gx1, gy0..gy1)` を逆投影で求め、その矩形内のタイルだけ
  Blitter に再構築する。64x64=4096 タイルのうち常に可視ぶん（数百タイル）しか描かない。
- オブジェクト/ユニットも同じ可視範囲（背高ぶん +1 タイル余白）でカリングしてからソートする。
  全 6 万規模の総当たりソートはせず、**カリング後の可視ぶんだけ**を `gx+gy` で並べる。

### 移動ユニット（決定的にうろつく）

- 各ユニットは index を seed に混ぜた `mulberry32` で初期位置と巡回目的地を決定的に得る。
  目的地へ 40px/s 相当（アイソメ 1 グリッド ≈ 35.78px に換算したグリッド速度）で直進し、
  到着したら次の決定的目的地へ。次セルが **水タイルなら目的地を選び直して回避**する。
- `+`/`-` の増減は index 由来のシードで配置が再現されるため、増やしても決定的に同じ並びになる。

### 決定的生成（シード固定）

- マップ（seed `0x9C17A`）: 粗い格子点の決定的ハッシュを双線形補間した値ノイズで
  草/土/水を割り当て（低い所＝水、中間＝土、高い所＝草）。`Math.random` は不使用。
- オブジェクト散布（seed `0x70B1EC75`）: 水以外のタイルに木（草地に多め）・家（稀）を決定的散布。
- ユニット（seed `0x5E1D17 ^ index`）: 初期位置・巡回目的地を決定的に生成。

### アセット / フォールバック

- `../assets/`（= `9/assets/`）から SPEC のファイル名で読込:
  `tile_grass.png` / `tile_soil.png` / `tile_water.png`（64x32 菱形）、
  `tree.png`（48x64）、`house.png`（64x64）、`villager.png`（24x32）。
- `BootScene` で `loaderror` を捕捉し、欠落キーは `Graphics.generateTexture` で焼いた図形に
  フォールバック: 地面=アイソメ菱形ポリゴン（草=緑/土=茶/水=青）、木=茶幹+緑円、
  家=灰の菱形柱（屋根付き）、ユニット=橙丸。**画像が 1 枚も無くても必ず起動する。**

## Codex 生成所感（アイソメ深度ソートの実装しやすさ）

- **投影と逆投影**: SPEC の式（`(gx-gy)*32`, `(gx+gy)*16`）は素直に関数化でき、カリングの逆投影も
  単純な連立で導ける。画面四隅を逆投影して AABB を取る方式は実装が短く、斜め範囲のカリングを
  矩形走査に落とせるのが扱いやすかった。
- **Phaser `depth` での z-order**: `setDepth(gx+gy)` を可視スプライトに毎フレーム設定するだけで
  前後関係が完結する。同値の安定化に微小オフセットを足す程度の工夫で済み、Pixi の
  `sortableChildren` や Godot の `YSort` 相当を最小コードで再現できた。
- **2 層方式の効き**: 地面をソート対象から外し Blitter でバッチ描画、背の高いものだけ
  ソートする SPEC 推奨方式により、ソート対象を可視オブジェクト＋ユニットの数百〜数千に抑えられる。
  `Objects sorted` をそのまま HUD に出せて性能比較と好相性。
- **共有プール**: 静的オブジェクトとユニットを 1 プールに混ぜ、ソート後の並びで使い回す構造は
  GC 圧を抑えつつ depth 反映を一括化でき、Codex 生成でも安定して書けた。
- **フォールバック**: `Graphics.generateTexture` で菱形ポリゴンや菱形柱の家を焼けるため、
  アセット欠落時でもアイソメの奥行きが分かる図形描画を手軽に用意できる点は Phaser の強み。
- 総じて Phaser 4 は「アイソメ投影 + 可視カリング + 毎フレーム depth 再ソート」を
  少ない API（blitter / generateTexture / setDepth）で素直に両立できた。
