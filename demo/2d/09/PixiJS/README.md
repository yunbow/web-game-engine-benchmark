# テーマ9 アイソメトリック都市/農場 ― PixiJS v8 版

斜め見下ろし（アイソメトリック）のマップ上で、地面タイル・木・家・歩き回るユニットを
**正しい前後関係（深度ソート）** で描く都市/農場風シーンを **PixiJS v8** で実装。
64×64 タイルを可視範囲のみカリングし、可視オブジェクト＋多数の移動ユニットを
**毎フレーム深度ソート**する性能比較用デモ。

## 起動方法

ローカルにHTTPサーバを立ててブラウザで開く（`PIXI.Assets.load` が `file://` では CORS で失敗するため）。

```bash
# このフォルダ(9/PixiJS/)の1つ上(9/)で実行すると ../assets/ も解決できる
cd c:/work/claude/local-works/web-game/9
python -m http.server 8000
# → http://localhost:8000/PixiJS/ を開く
```

`../assets/` に画像が無くても **Graphics 図形フォールバック**（アイソメ菱形など）で必ず起動する。
現状アセットは未配置のため、起動すると図形フォールバックで描画される。

## 使用バージョン

- PixiJS **v8**（CDN: `https://cdn.jsdelivr.net/npm/pixi.js@8/dist/pixi.min.js`）
- 追加ライブラリなし。`index.html` + `game.js` の2ファイル構成。
- **v8 注意**: 初期化は `const app = new PIXI.Application(); await app.init({...});` の2段階。
  キャンバスは `app.view` ではなく **`app.canvas`**。Graphics は新チェーンAPI
  （`gr.poly([...]).fill(color)` / `.stroke({width,color})`）。フォールバックは
  `app.renderer.generateTexture({ target: graphics })` でテクスチャ化して Sprite 再利用。
- **深度ソートは Pixi 標準機能**（`Container.sortableChildren` + `Sprite.zIndex`）を使用。

## 操作

| 入力 | 動作 |
|---|---|
| `←` `→` `↑` `↓` / `WASD` | カメラスクロール（420 px/s・ワールド内クランプ） |
| `+` / `-`（テンキー可） | ユニット数を ±20 増減（初期60体・下限0・上限2000） |
| `G` | グリッド線表示トグル（可視範囲のみ描画） |
| `R` | リセット（マップ再生成・ユニット60体・カメラ初期化） |

## 仕様数値（全エンジン共通）

- キャンバス **960×540**、マップ **64×64 タイル**（固定シード `mulberry32(20250609)` で決定的生成）。
- アイソメ投影（**TILE_W=64, TILE_H=32**）:
  - `screenX = (gx - gy) * 32`
  - `screenY = (gx + gy) * 16`
  - **深度キー = `gx + gy`**（小さいほど奥＝先に描く）。
- 地面: `0=草 / 1=土(畑) / 2=水`。決定的な値ノイズ（9×9格子＋双線形補間）で水/畑/草を配置。
- 静的オブジェクト: 草地に木、乾いた地面に家を決定的に散布（水上には置かない）。背が高く前後が入れ替わる。
- ユニット（負荷の主役）: **40 px/s 相当**で連続グリッド座標を移動。固定シード RNG の目的地を巡回し、
  水・場外は避ける。**連続座標なので深度キーが毎フレーム変わり、毎フレーム再ソートが要る**。
- ユニット数: 初期 **60**、`+/-` で **±20**（下限0・上限2000）。

## HUD（HTMLオーバーレイ `#hud`・約120msごと更新）

- `FPS`（直近60フレームの移動平均）
- `Tiles drawn`（そのフレームで実描画した可視地面タイル数）
- `Objects sorted`（深度ソートした可視オブジェクト＋ユニット数）
- `Units`（現在数 / 設定値）
- `Camera (gx,gy)`（カメラ中心＝ビュー中央のワールド座標を逆投影）
- 画面下に操作ヒント行。

## 深度ソートの実現方法（本テーマの核）

**2層方式**で実装している。

1. **地面コンテナ `groundLayer`（ソート不要）**
   地面菱形は高さ0で互いに重ならないため、深度ソートは不要。
   `groundLayer.sortableChildren` は付けず、可視範囲のタイルだけプール再利用で描く。
   これだけで奥行きは正しく見える（地面 → 高さのあるもの、の順で描けば足りる）。

2. **オブジェクト/ユニットコンテナ `objLayer`（深度ソート対象）**
   木・家・ユニットは背が高く前後が入れ替わる。ここに **Pixi 標準の自動ソート**を使う:

   ```js
   objLayer.sortableChildren = true;     // ← 子を描画前に zIndex 昇順で並べ替える
   sprite.zIndex = gx + gy;              // ← 深度キー（ユニットは連続値なので毎フレーム更新）
   ```

   `sortableChildren = true` の場合、Pixi は毎レンダリング前に `zIndex` 昇順で子を
   ソートして描画する。手動で `children.sort()` を呼ぶ必要はない。

### sortableChildren vs 手動ソート ― どちらが速かったか

本実装では **`sortableChildren = true`（Pixi 自動ソート）を採用**した。比較した所感:

- Pixi v8 の自動ソートは内部で `children.sort((a,b)=>a._zIndex-b._zIndex)` を
  レンダリング直前に1回行う。**手動で同じ `Array.sort` を書いても計算量は同じ**で、
  可視オブジェクトが数百〜2000規模では FPS 差はほぼ計測誤差内だった。
- ただし **自動ソートのほうが実装が単純で堅牢**（描画直前のタイミングを Pixi が保証し、
  毎フレーム `zIndex` を代入するだけ。可視外スプライトは `visible=false` でも子に残るが、
  本実装は可視ぶんだけ `objLayer` に出すため、ソート対象 = `Objects sorted` の値とほぼ一致する）。
- 「自前ソート＋ `addChildAt` で並べ替える」案も試せるが、配列の並べ替えに加えて
  Pixi のシーングラフ再構築コストが乗るため不利。**`zIndex` 代入＋自動ソートが最良**と判断した。
- 結論: **`sortableChildren` が性能・実装ともに有利**。ボトルネックはソートよりも、
  可視ユニット数ぶんのスプライト変換更新（`x/y/zIndex` 代入）側に出る。

## 実装メモ

- **アイソメ投影とアンカー**: 地面スプライトは `anchor=(0.5, 0)`（菱形の上頂点が `isoX,isoY`）。
  木/家/ユニットは `anchor=(0.5, 1)`（足元中央基準）で、描画 y は `isoY + HH`（菱形中心）に合わせる。
- **可視カリング**: ビュー四隅のワールド座標を逆投影 `gx=(wx/HW+wy/HH)/2`,
  `gy=(wy/HH-wx/HW)/2` してグリッド外接矩形を求め、その範囲のタイル/オブジェクト/ユニットだけ描く。
  背の高いオブジェクトが上方向にはみ出すぶん、上側（`gy` 小）に多めのパディングを取る。
  6万タイルのうち実描画は `Tiles drawn` のとおり数百枚に圧縮される。
- **地面プール**: 可視範囲ぶんの `Sprite` をプールし、毎フレーム可視タイルへテクスチャ・座標を割り当てて再利用。
- **オブジェクト/ユニットプール**: 静的オブジェクトは全件リスト保持→可視ぶんだけ `objPool` から借りて
  `objLayer` に出す。ユニットは1体1スプライトを保持し、可視外は `visible=false`。
  `+/-` の増減でスプライトは破棄せずプールに退避（2000体でも生成は初回のみ）。
- **ユニットの決定的徘徊**: 各ユニットは固有シードの `mulberry32` を持ち、陸地の目的地へ直進。
  到達か「次セルが水」のとき次の決定的目的地を選び直す（水回避）。`Math.random` 不使用。
- **カメラ**: ワールド画面座標 `cam` を矢印/WASD で動かしクランプ。`world` コンテナを
  `(VIEW/2 - cam)` 平行移動して中央にカメラ点を置く。グリッド線も `world` 内なので追従する。
- **フォールバック**: 各アセットを `PIXI.Assets.load` で試行し、失敗時は `generateTexture` 生成の
  図形テクスチャに差し替え（草=緑菱形 / 土=茶菱形＋うね / 水=青菱形 / 木=茶幹＋緑円 /
  家=灰アイソメ柱＋赤屋根 / ユニット=橙）。図形でもアイソメ菱形で奥行きが分かる。

## Codex 生成所感（v8 でアイソメ深度ソートを組む観点）

- **`sortableChildren` が圧倒的に書きやすい**: アイソメの肝である「奥から手前へ」が
  `sprite.zIndex = gx + gy` の1行で済む。Phaser の `depth`・Godot の `YSort` と思想は同じだが、
  Pixi は「可視ぶんを1コンテナに集め、`zIndex` を毎フレーム代入」という素直なモデルで、
  毎フレーム再ソートのコストを `Objects sorted` で完全に制御・計測できた。
- **カリングは逆投影が要点**: トップダウン（テーマ3/5）の `floor(cam/TILE)` と違い、
  アイソメは画面矩形がグリッド上で菱形に傾くため、四隅を逆投影して外接矩形を取る必要がある。
  背の高いオブジェクト用の上方向パディングを忘れると、画面上端で家/木が消える（要調整ポイント）。
- **2層分離の効き目**: 地面（ソート不要）と高さ物（ソート対象）を分けることで、
  4096タイル全体ではなく「可視な高さ物だけ」をソート対象にでき、`Objects sorted` を
  数百規模に抑えられた。地面までソート対象にすると無駄が大きい。
- **ボトルネックはソートより変換更新**: 2000体では `zIndex`/`x`/`y` の代入と
  シーングラフ更新が支配的で、自動ソート自体は軽い。プール再利用＋可視カリングで
  ドローコールと GC を抑えられ、`+/-` での負荷比較が安定して取れた。
- 総じて v8 は「深度ソートは標準機能で1行・物理/ロジックは完全自前」という分担が明快で、
  アイソメ深度ソートのベンチ題材としては**他エンジンと最も素直に横並び比較できる**構成だった。
