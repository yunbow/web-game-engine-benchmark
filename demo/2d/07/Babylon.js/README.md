# テーマ7: 物理パズル (投擲物理) ― Babylon.js + Matter.js 版

積み上げた箱（剛体）の山に発射体をぶつけて崩す 投擲物理の物理パズルを、
**Babylon.js（描画）＋ Matter.js（2D剛体物理）** で実装したもの。
Babylon は3Dエンジンで内蔵物理（Havok/Ammo）も3D前提のため、本テーマの純2D剛体は
**Matter.js（CDN）に委譲**し、Babylon は「正射影カメラ + SpriteManager/Sprite」で
描画専用に徹している。「3Dエンジンを2Dレンダラとして使い、外部2D物理エンジンと毎フレーム
同期したときの統合相性と剛体数スケール」を測るのが主眼。

## 起動方法

`../assets/` を相対パスで読むため、ローカルHTTPサーバ経由で開く（file:// でも起動はするが
画像が読めずフォールバック図形になる）。

```bash
# このフォルダ (7/Babylon.js) の1つ上 (7/) で実行すると ../assets も配信される
cd c:/work/claude/local-works/web-game/7
python -m http.server 8000
# ブラウザで http://localhost:8000/Babylon.js/ を開く
```

または VS Code の Live Server などでも可。
画像アセットが無い環境でも **単色図形でフォールバック**して必ず起動する。

## 使用バージョン

- Babylon.js: CDN 最新版 (`https://cdn.babylonjs.com/babylon.js`) ― 描画専用
- Matter.js: `0.19.0` (`https://cdn.jsdelivr.net/npm/matter-js@0.19.0/build/matter.min.js`) ― 2D剛体物理
- 追加ビルドなし。`index.html` + `game.js` の2ファイルのみ。

## 操作

| 操作 | 動作 |
|---|---|
| 左ドラッグ&リリース | スリングショット（引いた方向の反対へ、距離に比例した初速で発射） |
| クリック（ほぼ動かさず） | クリック地点へ向けて固定初速で発射 |
| `Space` | オートショット トグル（0.8s 間隔で決定的な角度/初速を自動発射） |
| `+` / `-` | 箱（剛体）数を ±20（下限20・上限600）／変更時に再構築 |
| `R` | 構造物・スコアを決定的に再構築 |

## 使用した物理エンジンと統合方法（最重要）

- **物理エンジン: Matter.js（2D剛体）を CDN 併用**。Babylon は3Dエンジンで内蔵物理も
  3D（Havok/Ammo, 3軸）前提のため、2Dの箱スタック/反発/スリープを素直に扱えない。
  そこで剛体シミュレーション（重力・接触・スタック・反発・摩擦・スリープ）はすべて Matter に委譲し、
  テーマ4/5のような自前AABBは一切書いていない。
- **毎フレームの流れ**：`Matter.Engine.update(physics, ms)` で物理を1ステップ進め
  （`dt` 秒 → ms、スパイク抑制で上限 0.033s にクランプ）、その後 **各 Matter ボディの
  `position.x/y` と `angle` を Babylon スプライト/プレーンへ転写**（`syncSprites()`）。
  物理 px → 正射影ワールド px は **1:1** なので座標スケール変換は不要。
- **Y 軸の一致**：正射影カメラを `orthoTop=0 < orthoBottom=540` に設定して
  「左上原点・+y 下向き」にした。Matter も左上原点・+y 下向きなので **両者の座標系が一致**し、
  Y 反転なしで位置同期できる。回転だけは API 差を吸収する：
  - Babylon **Sprite** の `angle` は反時計回り → Matter の時計回り angle に対し **符号反転** (`-rad`)。
  - フォールバック **mesh** は `rotation.z` をそのまま使う（ortho が y 下向きなので一致）。
- **重力**：Matter の `gravity.scale=0.001`・`gravity.y=1.0` で約 1000 px/s² 相当。
  箱が約1〜1.5sで落ち着く見た目に調整。`positionIterations=8 / velocityIterations=6` で
  大量スタックでもめり込みにくくしている。

## 仕様準拠（数値）

- キャンバス **960x540** 固定。座標は左上原点・px・画面下が地面。
- **静的床**：上面が画面下端から **上 48px**（中心 y を床高で調整）。左右に静的な壁、
  天井も静的（高速発射体の場外抜け防止）。これらは再構築でも作り直さず使い回す。
- **構造物（決定的, mulberry32 seed=0x7B0C）**：1箱 **34x34**。箱数 `n` から列数
  `cols ≈ round(sqrt(n*1.4))`（左限界 360px でクランプ）を決め、**「最も低く中央寄りの列」を
  決定的に選んで**ピラミッド/格子状に積む。頂上付近の **2〜4個をターゲット箱**（色違い）にする。
  箱は `DYNAMIC` 剛体、床/壁は `STATIC`。
- **崩し判定**：箱の重心が初期位置から **64px 以上**移動したら「崩した」とみなして加点（箱ごとに1回）。
  通常箱 **+10**、ターゲット箱 **+50**。
- **発射体**：剛体の円（半径 **12**）。
  - 左ドラッグ&リリース＝引いた距離に比例（上限初速 24）した逆向き初速。
  - クリック＝クリック地点へ固定初速。
  - `Space` オートショット＝0.8s 間隔で決定的な角度（-20°〜-55°）・初速（16〜22）。
  - 同時 **最大8発プール**、超過時は **古いものから（FIFO）破棄**。発射体は箱より高密度で当たると崩れる。
- **場外スリープ除去**：床より下/壁の外へ出て **`body.isSleeping`** になった剛体（箱・発射体）を除去
  （剛体数の暴走防止）。箱の加点済みフラグは保持。除去は負荷値（設定箱数）には影響しない。
- **箱数（負荷）**：初期 **60**、`+`/`-` で **±20**（下限 **20**・上限 **600**）。変更で決定的に再構築。
- デルタタイム基準（`engine.getDeltaTime()/1000`、上限 0.05s でクランプ）。

## 実装メモ ― 3Dエンジン × 外部2D物理の同期

- **描画と物理の責務分離**。Babylon 側は「ボディが今どこにいるか」だけを知ればよく、
  位置・回転の単方向転写（物理→描画）に限定した。入力（発射）と再構築のときだけ物理側へ書き込む。
  この一方向同期が、3Dエンジンと外部物理を組み合わせるときの最もクリーンな設計だった。
- **正射影 px 等倍が効く**。`ortho` を `0..960 / 0..540` に張ると、Matter の出力 px をそのまま
  `position.x/y` に入れられ、スケール変換コードがゼロになる。テーマ5の自前物理版と同じ
  「左上原点・+y下」空間でロジックが完結する。
- **回転の符号だけ要注意**。Babylon の Sprite angle は反時計回り、Matter の angle は（y下向き画面で）
  時計回りなので、Sprite には `-angle`、フォールバック mesh の `rotation.z` には `+angle` を渡す。
  ここを取り違えると箱が「逆さに転がる」ように見える。
- **スリープと除去で剛体数を制御**。`sleepThreshold` を短めにして安定したスタックを早く眠らせ、
  `Active`（覚醒数）が落ち着くようにした。場外へ抜けて眠った body は除去し、剛体数の暴走を防ぐ。
- **フォールバック**：起動時に `new Image()` で各アセット URL を試し（`checkImage()`）、欠落なら
  `SpriteManager` の代わりに単色 `Plane`(矩形)/`Disc`(円) を `emissive`・`disableLighting` で表示。
  統一ラッパ `{setPos,setVisible,setSize,setAngle,dispose}` により、Sprite と Mesh をロジックから
  区別せず扱える（テーマ5から `setAngle` を追加）。

## Codex 生成所感 ― なぜ3Dエンジンで外部2D物理を使うのか

- **Babylon の内蔵物理は3D前提で、2D箱崩しには過剰かつ不向き**。Havok/Ammo は3軸の剛体・拘束を
  解くため、Z を固定して擬似2Dにすることは可能だが、設定が増え、2D専用の broad/narrow phase や
  スリープ実装の素直さでは matter.js に劣る。SPEC が「各エンジンの“素直な選択”を使う」と定める通り、
  Babylon では **matter.js を CDN 併用**するのが最も自然で、PixiJS 版と同じ統合パターンになる。
- **統合の核は座標系・Y軸・角度のすり合わせ**。幸い Matter も Babylon ortho（y下設定）も
  左上原点・+y下で揃えられ、px 等倍なので位置はノーコスト同期。残るは前述の回転符号だけで、
  ここを文書化しておけば移植も容易。
- **性能特性の比較ポイント**：剛体数を `+` で 600 まで増やすと、Matter のソルバ（接触解決・拘束反復）が
  支配的になる。Babylon 側は SpriteManager のインスタンシング描画が効くため、ボトルネックは
  ほぼ物理側に寄る ＝ 「描画は3Dエンジン由来で強いが、2D物理は外部ライブラリ任せ」という
  本テーマ特有の性能プロファイルが観察できる。`Active`（覚醒数）が安定後に落ち着くか張り付くかで
  Matter のスリープ挙動が読み取れる。
- 総じて、3Dエンジンを2Dレンダラとして使い外部物理と繋ぐ構成は、**責務分離が明快で実装は素直**だが、
  「2つのライブラリの座標規約を最初に1:1へ合わせ込む」初期コストが全体の難所だった。
