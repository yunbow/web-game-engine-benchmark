# テーマ7 物理パズル (投擲物理) ― LittleJS + matter.js 版

積み上げた箱(剛体)の山に発射体をぶつけて崩す 投擲物理の物理パズルを
**LittleJS(描画/入力)＋ matter.js(剛体物理)** で実装したもの。
本テーマの主題は「**本物の2D剛体物理エンジンを“使った”ときの統合相性と剛体数スケール**」で、
テーマ4(ブロック崩し)・テーマ5(横スクロール)の「物理エンジン不使用・自前AABB」の対になる。
重力・接触・スタック・反発・スリープはすべて matter.js に委譲し、LittleJS は描画と入力に専念する。

## 起動方法

CDN 読込のため `file://` で直接開いても動くが、`../assets/` の相対参照と
ブラウザのキャッシュ挙動を安定させるため簡易HTTPサーバ経由を推奨。

```bash
# このフォルダ (7/LittleJS) で
python -m http.server 8000
# → ブラウザで http://localhost:8000/ を開く
```

`../assets/` フォルダは空でも起動する(画像欠落時は単色図形フォールバック)。
SPEC のアセット(`box.png` 等)を `../assets/` に置けば自動で使用される。

## 使用バージョン / CDN

- **LittleJS**(`@latest` 解決時点 / classic global build) ― 描画・入力・ループ
  - 主URL: `https://unpkg.com/littlejsengine`
    失敗時フォールバック: `https://cdn.jsdelivr.net/npm/littlejsengine/dist/littlejs.min.js`
  - 素の unpkg は classic global build(グローバル `engineInit` 等を生やす版)を解決するため
    `<script>` 直読みできる。読込失敗時のみ `error` イベントを捕捉して jsDelivr ミラーへ切替。
- **matter.js 0.19.0** ― 剛体物理
  - `https://cdn.jsdelivr.net/npm/matter-js@0.19.0/build/matter.min.js`
  - `game.js` より**前**に読込み、グローバル `Matter` を参照する。

## 操作

- **左ドラッグ&リリース**: スリングショット発射。発射台アンカーから、引っ張った**逆方向**へ、
  引っ張り距離に比例した初速で半径12の円(発射体)を撃つ。ドラッグ中はゴム/予測線を表示。
- **クリック(ほぼ動かさない)**: クリック地点方向へ固定初速で発射(簡易操作)。
- **Space**: オートショット トグル。ON の間 **0.8s 間隔**で発射台から決定的な角度/初速で自動発射
  (マウス無しでベンチを回せる)。
- **+ / -**(メイン行 or テンキー): シーンの箱(剛体)数を増減(性能比較の主軸)。増減のたびに**再構築**。
- **R**: 構造物・スコアをリセット(決定的に再構築)。

## 仕様数値(全エンジン共通)

| 項目 | 値 |
|---|---|

| 床 | 画面下端から上 48px の静的床 + 左右に静的壁 |

| 場外除去 | 床より下/壁の外へ出て **スリープした**剛体を除去(剛体数の暴走防止) |

## HUD(常時表示・左上, HTML `#hud` overlay)

- `FPS`: 実測の移動平均(エンジン `frameRate` を指数移動平均で平滑化)。
- `Bodies`: 剛体総数(箱 + 発射体 + 床1 + 壁2)。括弧内に「箱数 / 設定値」。
- `Active`: 覚醒(awake)している剛体数。matter の `body.isSleeping === false` を数える。
- `Shots`: 生存中の発射体数 / 上限8(累計発射数も併記)。
- `Score` / `Engine`: スコア と 使用物理エンジン名 `Matter (CDN)`。
- `[sprites]` / `[shapes fallback]`: 画像使用かフォールバックかを明示。`AUTO:ON/OFF` でオート状態。

## 実装メモ

### 物理エンジンの選択(なぜ matter.js か ― 正直な記載)
SPEC は LittleJS について「**公式 Box2D プラグイン(`box2d.js` / `littlejs.box2d`)を使用。
利用不可な場合は matter.js を CDN 併用してよい(README に明記)**」としている。
本実装は **matter.js (CDN) を採用**した。理由:

- LittleJS 公式 box2d プラグインは別配布の wasm/ESM ビルドで、本ベンチの
  「classic global build を `<script>` 直読み + 単一 `game.js`」という構成と相性が悪い
  (ESM/`type=module` への切替や wasm の取り回しが必要)。
- matter.js は**単一の min.js を `<script>` 直読み**でき、グローバル `Matter` を介して
  Engine/World/Bodies/Body/Composite に素直にアクセスできる。ベンチ構成にそのまま載る。

その代わり、**物理は matter、描画は LittleJS** という二段構えになるため、
両者の **Y軸の向きが逆**という固有の罠が生じる(下記)。これは PixiJS/Babylon の
matter 併用構成と同じ論点であり、本ベンチでは「描画エンジンに matter を後付けすると
何が起きるか」をそろえて観測できる。

### ★ 二重Y軸の取り扱い(最重要) ★
- **matter.js は Y軸が下向き**(画面下=Y大、重力は +Y へ)。これは matter の自然な規約。
- **LittleJS のワールドは Y軸が上向き**(画面下=Y小、画面上=Y大)。
- そこで2つの座標空間を完全に分離する:
  - **(A) Matter 空間**: 左上原点・px・Y下向き。物理はすべてここで解く。`gravity.y = +1.0`。
  - **(B) LittleJS ワールド空間**: `cameraScale = 1`(1ワールド=1px)・Y上向き。
    カメラを画面中心 `(480, 270)` に固定し、ワールド座標 = 左上原点 px(Yだけ上向き)に揃える。
- 変換は3関数に閉じ込めた:
  - `m2l(mx,my) = vec2(mx, VIEW_H - my)` … Matter → LittleJS(**Y反転**、x はそのまま)。
  - `l2mY(ly) = VIEW_H - ly` … LittleJS(マウス等) → Matter(Y逆変換)。
  - `m2lAngle(a) = -a` … Y反転に伴い**角度の符号も反転**(箱の回転表示で重要)。
- 入力 `mousePos` は LittleJS ワールド座標(Y上向き)で来る。発射体の初速も
  LittleJS の方向ベクトルで作り、Matter へ渡すとき y 成分の符号を反転する
  (`vmy = -dirY * power`)。この一点を間違えると「上にドラッグしたのに下へ飛ぶ」事故になる。

### Matter ワールド構築
- `Engine.create({ enableSleeping: true })` でスリープ有効化(`Active` 観測のため)。
- 静的: 床(`isStatic` 矩形, 中心 Matter-Y = `VIEW_H - GROUND_H/2`)、左右の壁(画面外, 高さ2倍で飛び越え防止)。
- 動的: 箱を `Bodies.rectangle(34,34)` で格子配置。列数は箱数から `√n*1.2` を 4..22 にクランプして決定的に決定。
  row 0 を床上面に乗せ、上の row ほど Matter-Y を小さく(=上へ)する。
- `+`/`-`/`R` では**ワールドを作り直す**(`Engine.create` から再構築)ことで決定性を担保。
  固定シード `mulberry32(70260615)` でターゲット箱の位置も決定的(`Math.random` 不使用)。

### 物理ステップ
- `Engine.update(engine, 1000/60)` を毎フレーム呼ぶ(固定 16.67ms 寄り)。
  LittleJS 側の `setGravity(0)` で LittleJS 物理は完全に無効化し、二重重力を防ぐ。
- 発射体の初速は `Body.setVelocity`(px/step)。スリングは引っ張り距離 × 0.12 を 1.5..20 にクランプ。

### スコアリング / 場外除去
- 加点: 各箱の重心が初期位置(Matter 座標)から **64px 超**移動したら 1 回だけ加点
  (通常 +10 / ターゲット +50)。`scored` フラグで重複加点を防ぐ。
- 除去: 箱/発射体が床より下・壁の外に出て**かつ `isSleeping`** なら `Composite.remove`。
  発射体は寿命(12s)超過でも除去。剛体数の暴走を防ぎつつ、設定箱数の負荷値自体は維持。

### 描画 / テクスチャ判定 / フォールバック
- 描画は LittleJS ワールド空間。各剛体の `position`/`angle` を `m2l`/`m2lAngle` で変換して
  `drawTile`(画像あり)/ `drawRect`・`drawCircle`(図形)で描く。
  ※ `drawCircle` の size は**直径**指定なので半径12は `BALL_R*2` を渡す。
- `textureInfos[i].size.x > 1` で読込済みを判定(`spriteReady`)。1枚も無くても必ず起動し、
  図形フォールバック(箱=茶 / ターゲット=橙 / 発射体=赤丸 / 地面=緑茶 / 壁=灰 / 発射台=灰 / 背景=空色)で描く。

## Codex 生成所感(描画エンジン + 外部物理エンジン併用の所感・罠)

- **最大の論点は二重Y軸**。matter(Y下)と LittleJS(Y上)が逆向きなので、
  「物理は Matter 空間で完結、描画の瞬間だけ `m2l` で反転」という**単方向の変換境界**を
  徹底するのが効いた。入力(mousePos)だけは逆変換 `l2mY` が要るので、変換関数を3つに固定し、
  それ以外の場所で生の `VIEW_H - y` を書かないルールにしてバグを封じた。角度も Y反転で
  符号が逆になる点を忘れると箱が「逆回転」して見えるので `m2lAngle` で一括反転。
- **テーマ4/5 との対比が綺麗に出る**。あちらは軸分離AABBを完全自前で書いたが、本テーマは
  重力・接触・スタック・反発・スリープを matter に丸投げするだけで“箱が崩れて積み直る”挙動が
  得られる。代わりに性能は完全に matter のソルバ依存で、`+`/`-` で箱を増やすと
  `Active`(覚醒剛体)が安定後にスリープへ落ちるか張り付くかが如実に出る ―
  これがまさに SPEC の狙う「ソルバ/スリープ実装差」の観測点。
- **LittleJS 自体の物理を切る**のを忘れない(`setGravity(0)`)。切らないと LittleJS の
  EngineObject 物理と matter が二重に効いて破綻する。本実装は LittleJS の EngineObject を
  一切使わず、剛体は matter の Body だけを真実とし、LittleJS は毎フレーム読んで描くだけにした。
- **`+`/`-`/`R` で都度ワールドを作り直す**のは富豪的だが、部分的な body 追加/削除より
  決定性が確実(同じシードで同じ山)。比較ベンチでは「同じ初期配置からの体数だけの違い」を
  保てることが最優先なので、再構築方式を選んだ。
- 総じて「**描画エンジン + 外部剛体物理の後付け**」は、Y軸変換境界さえ1箇所に閉じ込めれば
  驚くほど素直。matter の API(Engine/World/Bodies/Body/Composite)が薄く明快で、
  LittleJS の薄い描画 API と組み合わせると、性能の所在(描画 vs 物理ソルバ)が
  HUD の FPS / Active にそのまま現れ、ベンチ題材として扱いやすかった。
