# テーマ7 物理パズル (投擲物理) ― PixiJS v8 + Matter.js 版

積み上げた箱（剛体）の山に発射体をぶつけて崩す 投擲物理の物理パズルを
**PixiJS v8（描画）+ Matter.js（剛体物理, CDN）** で実装。
箱（剛体）数を `+/-` で 20〜600 までスケールさせ、覚醒（awake）剛体数の落ち着き方を
横並び比較するための性能比較用デモ。

## 起動方法

ローカルにHTTPサーバを立ててブラウザで開く（`PIXI.Assets.load` が `file://` では CORS で失敗するため）。

```bash
# このフォルダ(7/PixiJS/)の1つ上(7/)で実行すると ../assets/ も解決できる
cd c:/work/claude/local-works/web-game/7
python -m http.server 8000
# → http://localhost:8000/PixiJS/ を開く
```

`../assets/` に画像が無くても **Graphics 図形フォールバック** で必ず起動する（アセット0でも動作確認済み）。

## 使用バージョン

- PixiJS **v8**（CDN: `https://cdn.jsdelivr.net/npm/pixi.js@8/dist/pixi.min.js`）― 描画専用。
- Matter.js **0.19.0**（CDN: `https://cdn.jsdelivr.net/npm/matter-js@0.19.0/build/matter.min.js`）― 剛体物理。
  `index.html` で **game.js より前**に読み込む。`index.html` + `game.js` の2ファイル構成。
- **v8 注意**: 初期化は `const app = new PIXI.Application(); await app.init({...});` の2段階。
  キャンバスは `app.view` ではなく **`app.canvas`**。Graphics は新チェーンAPI
  （`gr.rect(...).fill(color)` / `.stroke({width,color})`）。フォールバックは
  `app.renderer.generateTexture({ target: graphics })` でテクスチャ化して Sprite 再利用。
- **物理は完全に Matter.js 任せ**（重力・接触・スタック・反発・摩擦・スリープ）。自前 AABB は書かない。

## 操作

| 入力 | 動作 |
|---|---|
| 左ドラッグ&リリース | スリングショット。ドラッグの逆向き・距離比例の初速で発射体（円 r12）を発射 |
| クリックのみ（ドラッグ < 6px） | クリック地点へ向け固定初速で発射 |
| `Space` | オートショット ON/OFF。ON の間 **0.8s 間隔**で決定的な角度/初速を自動発射 |
| `+` / `-`（テンキー可） | 箱（剛体）数を ±20 増減（初期60・下限20・上限600）。スコアはリセット |
| `R` | 構造物・スコアを決定的に再構築 |

## 仕様数値（全エンジン共通）

- キャンバス **960×540**。重力 下向き（見た目で箱が約1〜1.5sで落ち着くよう px 系に調整）。
- 地面: 画面下端から上 **48px** に水平な静的床。左右にも静的な壁（場外流出防止）。天井は無し。
- 箱: **34×34 px** の剛体（`DYNAMIC`）。固定シードで右側にピラミッド状に決定的に積む。
  頂上付近に **ターゲット箱（橙・加点）** を 2〜6個 決定的配置。
- 発射体: 半径 **12** の剛体円。**同時最大8発**プール、上限超過で最古から消す。
- 加点: 箱の重心が初期位置から **64px 以上**移動したら「崩した」とみなし1回だけ加点。
  通常箱 **+10** / ターゲット箱 **+50**。
- 場外（床より下/壁の外）に出た剛体は **スリープ後に除去**（剛体数の暴走防止）。
- 箱数初期 **60**、`+/-` で **±20**（下限20・上限600）。

## HUD（HTMLオーバーレイ `#hud`・約120msごと更新）

- `FPS`（直近60フレームの移動平均）
- `Bodies`（箱数 / 設定値。括弧内に床/壁/発射体を含む総ボディ数）
- `Active`（覚醒している剛体数。`body.isSleeping === false` を数える。静的ボディは除外）
- `Shots`（生存中の発射体数 / 累計発射数・上限8）
- `Score` / `Engine: Matter (CDN)`（＋オートショットの ON/off）
- 画面内ヒント行（ドラッグ=発射 / Space=オート / +/-=箱数 / R=再構築）

HUD はキャンバス外の HTML `<pre>` 要素で描画。Pixi/Matter の負荷と切り離して計測できる。

## 実装メモ ／ 物理エンジン統合方法

- **なぜ外部物理ライブラリ（Matter.js）か**: PixiJS は**描画専用**で物理機能を持たない。
  本テーマは「剛体物理エンジンを積極利用したときの統合相性と剛体数スケール」を測るのが主題なので、
  テーマ4/5 のような自前 AABB は使わず、2D剛体物理の定番 **Matter.js** を CDN 併用した
  （SPEC 指定）。Pixi 側は一切物理に関与せず、Matter のボディ姿勢を**毎フレーム描画へ写すだけ**。
- **ボディ↔スプライト同期（本デモの核）**:
  1. 毎フレーム `Matter.Engine.update(engine, deltaMS)`（スパイク時は 32ms にクランプ）で物理を進める。
  2. 各箱/発射体について、対応する Matter ボディの `body.position`（中心座標）を Pixi の
     `sprite.x/y` に、`body.angle`（rad）を `sprite.rotation` に代入する。
     Pixi スプライトは `anchor.set(0.5)`（中心基準）にしているので、Matter の中心座標と無変換で一致する。
  これで「論理＝Matter / 描画＝Pixi」を完全分離でき、Pixi は座標を写すだけで回転スタックも自然に描ける。
- **構造物の決定的生成**: 箱数 n からピラミッドの最下段幅を決定的に算出し、下広・上狭の格子で積む
  （`computeStackLayout`）。配置に `mulberry32` 由来の ±0.3px 微小ジッタのみ加え、`Math.random` は不使用。
  ターゲット箱は y が小さい上段プールから固定シードで選ぶ。`+/-`・`R` は常に同じレイアウトを再現する。
- **発射**: 左ドラッグの変位ベクトルの**逆向き**に距離比例（係数 0.22・上限速度クランプ）で初速を与える
  スリングショット。6px 未満のドラッグはクリック扱いで、クリック地点方向へ固定初速。
  発射体は `Body.setVelocity` で初速を与え、`balls` 配列を FIFO プール（最大8）として古い順に押し出す。
- **オートショット**: `Space` トグル。`autoTimer` を `deltaMS` で積算し 0.8s ごとに、`mulberry32` の
  決定的系列から角度（-55°〜-15°）と初速（18〜24）を引いて発射。マウス無しでベンチを回せる。
- **崩し加点**: 箱ごとに初期重心 `(x0,y0)` を保持し、現在重心との距離が 64px 以上で1回だけ加点
  （`scanDisplacement`）。加点済み箱は `alpha=0.7` で可視化。接触イベントではなく**変位**で判定する簡易実装
  （SPEC 許容）なので、衝突解決の詳細に依存せず安定する。
- **場外除去**: ボディ重心が画面外マージン（±200px）に出て、かつ `body.isSleeping` の箱を除去。
  発射体は場外なら即除去。除去しても `boxSet`（設定値）は変えないので負荷値は維持される。
- **スリープと Active**: `engine.enableSleeping = true`。HUD の `Active` は全ボディ走査で
  `!isStatic && !isSleeping` を数える。発射直後は山全体が覚醒し、数百ボディでも安定後に順次スリープして
  `Active` が落ちる様子を観測できる。`sleepThreshold` を 30 に下げて落ち着きを早めた。
- **フォールバック**: 各アセットを `PIXI.Assets.load` で試行し、失敗時は `generateTexture` 生成の
  図形テクスチャに差し替え（箱=木目茶 / ターゲット=橙＋星 / 発射体=赤丸 / 地面=緑茶タイル /
  発射台=灰Yフォーク / 背景=空色＋雲）。画像0でも必ず起動する。

## Codex 生成所感（v8 + 外部物理ライブラリを統合する観点）

- **Pixi と Matter の責務分離が極めて明快**: Pixi は「Matter の `position`/`angle` を `sprite.x/y/rotation`
  に写すだけ」。`anchor.set(0.5)` で中心基準を合わせれば座標変換すら不要で、テーマ4/5 の自前物理に比べ
  当たり判定コードがゼロになる。回転を持つスタック崩しが**数十行の同期ループ**で成立するのは外部物理の利点。
- **v8 初期化の罠（再掲）**: `new PIXI.Application(opts)` は無視され `app.canvas` が生えない。
  必ず `new PIXI.Application()` → `await app.init(opts)` の2段階。`app.view` も `app.canvas` に改名。
- **Matter の単位系の調整がやや手探り**: Matter は内部的に「速度＝1ステップあたりの px 移動量」「重力＝
  `gravity.y * gravity.scale`」という独自スケールで、`px/s²` で考える Pixi 流の直感と合わない。
  `gravity.scale=0.001`（既定）に対し初速 18〜26 程度・密度 0.0018〜0.004 で「箱が1〜1.5sで落ち着く」
  見た目に合わせ込んだ。ここはエンジン固有の勘所で、Phaser4 内蔵 Matter との差は基本ない（同じライブラリ）。
- **可変フレームと固定ステップ**: `Engine.update` に生の `deltaMS` を渡すと重い瞬間に貫通（tunneling）し得る。
  `Math.min(dtMs, 32)` でクランプし、薄い壁（厚み40px）で場外流出を防いだ。厳密には固定タイムステップ＋
  サブステップが望ましいが、ベンチ用途ではクランプで十分安定した。
- **剛体数スケールの素直さ**: `+/-` で 600 箱まで積んでもボトルネックは Matter の拘束ソルバ側に出て、
  Pixi の描画（Sprite 数百枚＋回転）は余裕。`Active`（覚醒数）が安定後にスリープへ収束するか張り付くかで
  ソルバ/スリープ実装差が読み取れる、というテーマ意図どおりの比較軸が綺麗に出る構成だった。
- 総じて v8 は「描画は超軽量・物理は外部ライブラリに丸投げ」という分担が明快で、**統合コストはほぼ
  同期ループのみ**。テーマ4/5（自前物理）の対極として「物理は他人任せ・描画だけ最適化」という
  比較軸がはっきり出る、投擲物理ベンチの題材として相性が良い。
