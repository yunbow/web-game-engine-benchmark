# テーマ5: 横スクロールアクション ― Babylon.js 版

横スクロールのジャンプアクション（横スクロールジャンプアクション）を **Babylon.js** で実装したもの。
2Dゲームだが Babylon は3Dエンジンのため、**正射影(Orthographic)カメラ + SpriteManager/Sprite** で
2Dタイルマップを再現している。「3Dエンジンで 広い横長マップ描画(カリング) ＋ 自前の重力/AABB物理
＋ 多数の敵の更新 を同時に回した性能」を測るのが主眼。

## 起動方法

`../assets/` を相対パスで読むため、ローカルHTTPサーバ経由で開く（file:// でも起動はするが
画像が読めずフォールバック図形になる）。

```bash
# このフォルダ (5/Babylon.js) の1つ上 (5/) で実行すると ../assets も配信される
cd c:/work/claude/local-works/web-game/5
python -m http.server 8000
# ブラウザで http://localhost:8000/Babylon.js/ を開く
```

または VS Code の Live Server などでも可。
画像アセットが無い環境でも **単色図形でフォールバック**して必ず起動する。

## 使用バージョン

- Babylon.js: CDN 最新版 (`https://cdn.babylonjs.com/babylon.js`)
- 追加ビルド・依存なし。`index.html` + `game.js` の2ファイルのみ。

## 操作

| キー | 動作 |
|---|---|
| `←` `→` / `A` `D` | 移動 (歩き 180 px/s) |
| `Shift` | ダッシュ (288 px/s, ×1.6) |
| `Space` / `↑` / `W` | ジャンプ (接地時のみ, 可変ジャンプ) |

## 仕様準拠（数値）

- キャンバス **960x540** 固定 / タイル **32x32** / マップ **200x17**（= 6400x544 px）。
- マップは決定的擬似乱数 (mulberry32, seed=0xB17A) で生成。
  - `0=空, 1=地面(ground), 2=ブロック(brick), 3=土管(pipe)`。**進入不可 solid = 1,2,3**。
  - 最下段は地面、所々に穴(gap)、空中にブロック足場、地上に土管、左右端は壁。
- 物理: 重力 **1800 px/s²**、デルタタイム基準（`engine.getDeltaTime()/1000`, 上限 0.05s でクランプ）。
- 自機: 当たり判定 **24x44**（描画 32x48）。歩き **180** / ダッシュ **288** / ジャンプ初速 **-640**。
  **可変ジャンプ**（上昇中にキーを離すと上昇を 0.40 倍に減衰）。AABB を **軸分離**（x→y）で解決し接地フラグを持つ。
- 敵(goomba): **28x28**、水平歩行 **60**、重力あり。壁衝突で反転＋ガケ落下回避。
  踏みつけ（自機 vy>0 で上面に乗る）→ 撃破 **Score +100**・スパーク・跳ね返り **vy=-380**。
  横接触 → 被弾（ノックバック＋無敵 **1.0s**, **HP-1**）。
- コイン: 決定的配置。接触で取得 → `Coins++` / **Score +50**（取得で消える）。`Entities` に含む。
- 自機 HP 初期 **3**。穴落下（y > マップ下端+余白）または HP0 で**スポーン地点へ復帰**
  （HPを3に戻す。ベンチ継続のためスコア・敵は保持）。
- 敵数: 初期 **20**、`+`/`-` で ±10（下限0・上限 **500**）。足場のある地形に決定的にスポーン。

## 実装メモ ― 正射影2D + カリング + 自前物理

- **正射影カメラで 2D 画面を再現**：`orthoLeft/Right/Top/Bottom` を `0..960 / 0..540` に設定し
  （`orthoTop < orthoBottom` で y 下向き）、画面座標 = ワールド px を 1:1 で対応させた。
  ロジックは終始「左上原点・+y下方向」の px 空間で完結させ、座標変換を不要にしている。
- **横スクロールは ortho 窓の平行移動**：自機を概ね中央に置く `cameraX` を `[0, worldW-960]` で
  クランプし、毎フレーム `orthoLeft = cameraX` / `orthoRight = cameraX + 960` をずらして窓を動かす。
  背景(空)スプライトも `cameraX` 分だけ動かして常に画面を覆う。テーマ3が `camera.position` を
  動かしていたのに対し、本テーマは横スクロール量を直接 ortho 範囲に反映している。
- **カリングの実装**：タイル種別(ground/brick/pipe)ごとに `SpriteManager` を1つ用意し、各 manager に
  **スプライトプール**を持つ。毎フレーム、`cameraX` から可視列範囲 `[x0,x1]`（約 30 列 ×17 行 ≈ 510 枚）
  だけをプールへ割り当て直し（位置更新＋`isVisible`）、余ったプールは非表示にする。空タイル(0)はスキップ。
  6400×544 の全タイルを毎フレーム生成すると重いので、この使い回しが事実上必須。`Tiles drawn` で実数を確認できる。
- **自前の重力 / 軸分離 AABB**：物理エンジンは使わない。`moveBody()` が x 移動 → 衝突解決 → y 移動 →
  衝突解決の順で処理し、めり込んだ分だけタイル境界へ押し戻す。落下中の y 衝突で `grounded=true` を立て、
  ジャンプは接地時のみ許可。終端速度をクランプして高速落下時のタイルすり抜けを防いでいる。
- **敵 AABB と踏みつけ判定**：敵も同じ `moveBody()` で重力・壁反転を処理。自機×敵は AABB 重なり後に
  「自機が落下中(vy>0)かつ下端が敵上半分」で踏みつけ撃破、そうでなければ横接触被弾、と分岐する。
- **フォールバック**：起動時に `new Image()` で各アセット URL を試し（`checkImage()`）、欠落なら
  `SpriteManager` の代わりに単色 `Plane`(矩形)/`Disc`(円) を `emissive`・`disableLighting` で表示する。
  統一ラッパ `{setPos,setVisible,setSize,dispose}` により、Sprite と Mesh をロジックから区別せず扱える。
- **z 重なり順**：背景=60(奥)、タイル=0、敵/コイン=-2、自機=-3、スパーク=-4(手前) と z を手動割当てして解決。

## Codex 生成所感 ― Babylon で 横スクロール プラットフォーマを書く

- **最大の難所は「自前物理 ＋ ortho スクロール ＋ カリング」の同居**。テーマ3は徘徊＋カリングだけだったが、
  本テーマは重力・軸分離AABB・可変ジャンプ・踏みつけ・ノックバックといったアクション特有の物理を全部
  手書きする必要がある。Babylon は3D物理(Havok/Ammo)を持つが、2Dタイル相手にそれを使うのは過剰で、
  結局「タイル格子に対する矩形の軸分離押し戻し」を自前で書くのが一番素直で軽い。
- **軸分離の押し戻しはバグの温床**。x→y の順で別々に解決し、衝突時はタイル境界へスナップする実装に落ち着いた。
  境界判定の `-0.0001` オフセットや終端速度クランプが無いと、高速落下や角での引っかかり・すり抜けが出る。
  ここは2D専用エンジン（Phaser のアーケード物理等）なら用意されている部分で、Babylon では手数が増える。
- **横スクロールは ortho 範囲を直接動かすのが手軽**。`orthoLeft/Right` を `cameraX` でずらすだけで
  ピクセル等倍のまま 6400px の横長ワールドをスクロールでき、エンティティ座標はワールド px のまま置ける。
  背景だけはカメラに追従させる必要がある（ワールド固定だと流れていってしまう）点に注意。
- **カリングは3Dエンジン2Dタイルの定番コスト**。種別ごとの SpriteManager + プール再利用で
  「可視分(約510枚)だけ毎フレーム再配置」する設計はテーマ3と同型。SpriteManager のインスタンシングは
  大量描画に強く、`+` で敵を 500 体まで増やしても破綻しにくいのは3Dエンジン由来の強み。
- 総じて、Babylon は「描画パイプライン・カリング・2D物理」を自分で組むコストがかかる反面、
  正射影カメラのピクセルパーフェクト性とスプライトの大量描画性能は、性能ベンチ題材としては素直に効く。
