# テーマ8 パーティクル / 魔法エフェクトデモ ― Phaser 4 版

多数のエミッタが**加算合成のパーティクル(火花・魔法)**を噴き出すエフェクトデモ。
画面内を**決定的な軌道で周回する 4 個の発光オーブ**から火花を連続噴出し、
マウス移動でトレイル、左クリックで放射状の爆発バーストを出す。
性能比較の核は「**画面上の生存パーティクル総数**」で、`+`/`-` で目標上限を増減して
**加算ブレンド描画 + パーティクル更新のスループット**を測る比較用サンプル。
Phaser の内蔵 GPU パーティクル(`GameObjects.Particles.ParticleEmitter`)を使用する。

## 起動方法

ブラウザのセキュリティ制約(file:// での CORS / アセット読込)を避けるため、
簡易 HTTP サーバ経由での起動を推奨します。

```bash
# このフォルダ (8/Phaser4/) で
python -m http.server 8000
# → http://localhost:8000/ をブラウザで開く
```

または任意の静的サーバ(`npx serve` など)でも可。
アセット画像が無くてもフォールバック描画で必ず起動します。

## 使用バージョン

- Phaser **4**(CDN: `https://cdn.jsdelivr.net/npm/phaser@4/dist/phaser.min.js`)
- 追加ライブラリなし。`index.html` + `game.js` のみ。
- **パーティクル機構: Phaser 内蔵 `GameObjects.Particles.ParticleEmitter`(GPU バッチ描画)**。

## 操作

| 操作 | 動作 |
|---|---|
| マウス移動 | カーソル位置に小さな追従トレイルエミッタ |
| 左クリック | クリック地点で爆発バースト(一度に 120〜200 個を放射状放出) |
| `Space` | オート花火トグル(ON 中は 0.5s 間隔で決定的位置に爆発・マウス無しでベンチ可) |
| `B` | ブレンドモード切替(ADD ⇄ NORMAL) |
| `+` / `-`(テンキー可) | 目標パーティクル数(上限)を ±2000 |
| `R` | リセット |

- スコア / ライフは無し(デモのため。代わりにパーティクル統計を HUD 表示)。

## 仕様数値表(全エンジン共通)

| 項目 | 値 |
|---|---|
| 画面 | 960 x 540(固定) |
| 背景 | 暗色 `#08080f`(発光が映えるよう暗く) |
| 常設オーブ | 4 個・決定的な正弦/円運動軌道・連続火花エミッタ |
| パーティクル寿命 | 0.6〜1.4s |
| 寿命変化 | size 大→小 / alpha 1→0 / 色 暖色→寒色グラデ |
| 力 | 軽い重力(gravityY 60〜120)+ 寿命減衰 |
| 爆発バースト | 一度に 120〜200 個を放射状放出 |
| オート花火 間隔 | 0.5s |
| 目標パーティクル数 | 初期 2000 / ±2000 / 下限 500・上限 50000 |
| 合成 | 加算(ADD)既定・`B` で NORMAL 切替 |

## HUD

`setScrollFactor(0)` でカメラ非追従に固定し、左上に常時表示。

- `FPS`(実測・移動平均 30 サンプル)
- `Particles`(現在生存しているパーティクル総数 = 全エミッタの `getAliveParticleCount()` 総和)
- `Target`(目標上限)
- `Emitters`(常設オーブ数 4 + アクティブバースト数)
- `Blend`(ADD / NORMAL)/ `Mode`(GPU / CPU のいずれか → 本実装は **GPU**)
- 画面下に操作ヒント行(クリック=爆発 / Space=オート / B=ブレンド / +/-=量 / R=リセット)。

## 実装メモ

### 使用したパーティクル機構(GPU / CPU)

- **Phaser 内蔵 `GameObjects.Particles.ParticleEmitter`** を使用。Phaser 4 では
  `this.add.particles(x, y, texture, config)` がエミッタを**直接返す**(旧 v3 の
  ParticleEmitterManager は廃止)。各エミッタは単一テクスチャの粒子を**バッチ描画**するため、
  WebGL レンダラ下では描画は **GPU バッチ**(同一テクスチャ + 同一ブレンドで 1 ドローコールにまとまる)。
  HUD の `Mode` は **GPU** と表示。
  - ただし粒子の**位置/alpha/size の更新自体は CPU**(JS)側で行われる「CPU 更新 + GPU 描画」型。
    完全な GPU シミュレーション(Babylon の `GPUParticleSystem` 等)とは異なる点に注意。
    大量(数万)では JS 側の更新コストが FPS を支配しやすい。
- **加算ブレンド**: `blendMode: 'ADD'`(= `Phaser.BlendModes.ADD`)。重なるほど明るくなり発光感が出る。
  `B` キーで全エミッタ + オーブ本体スプライトを `NORMAL` に一括切替し、描画コスト/見た目を比較できる。

### 寿命に沿う変化(size / alpha / 色)

- `lifespan: { min: 600, max: 1400 }`(0.6〜1.4s)。
- `scale: { start: 1.1, end: 0.05 }` で **大→小**、`alpha: { start: 1.0, end: 0.0 }` で **1→0**。
- 色グラデは Phaser4 エミッタの `color`(配列で寿命に沿って補間)を使用し、
  **暖色(白→橙→赤)→寒色(紫→青)** へ遷移(`colorEase: 'quad.out'`)。
- 力は `gravityY`(オーブ 60 / バースト 120)で軽い重力を与え、初速 `speed` のランダム方向と合わせて
  自然な拡散にする。

### 生存数(Particles)の制御 ― 性能比較の主軸

- Phaser エミッタの**生存数を直接固定する API は無い**ため、**発生レートで定常生存数を間接制御**する。
  - 定常生存数 ≈ `rate(個/秒) × 平均寿命(秒)`、`rate = quantity / (frequency/1000)`。
  - 平均寿命は (0.6+1.4)/2 = **1.0s** なので、`Target` 個を狙うには毎秒 `Target` 個発生させればよい。
- `applyTargetRate()` で **オーブ 4 個の合計が Target の約 85%** を定常で確保するよう
  各オーブの `frequency`(8〜120ms にクランプ)と `quantity` を逆算して設定。残り 15% は
  トレイル/バーストの上乗せ余地。`+`/`-` で `Target` を変えるたびに再計算し、`Particles` 実測値が
  上限付近で安定する。
- **生存数の実測**は `emitter.getAliveParticleCount()` を全エミッタ(オーブ + トレイル + バーストプール)で
  合算して HUD に表示。

### プール / 再利用

- 爆発バーストは `emitter.explode(count, x, y)` で**一括放出**する単発エミッタ。
  毎回生成するとエミッタオブジェクトの GC が増えるため、**`burstPool` で停止中のエミッタを使い回す**
  (`_busy` フラグ管理。最長寿命ぶん経過後に解放)。Phaser のパーティクル粒子自体もエミッタ内部で
  プールされ、`maxParticles`/`frequency` の範囲で再利用される(生成/破棄の GC を避ける)。
- 常設オーブとトレイルは生成済みエミッタを使い回し続ける(`emitting` の on/off のみ制御)。

### オーブの決定的軌道

- 各オーブは中心 (cx,cy)・振幅 (ax,ay)・角速度 (wx,wy)・位相を **index 由来の `mulberry32`** で決定的に生成し、
  `x = cx + cos(t·wx + φ)·ax`、`y = cy + sin(t·wy + 1.3φ)·ay` の**正弦/円運動の合成**で周回。
  マウス無しでも常にパーティクルが流れ、ベンチが安定する。

### 決定的生成(シード固定)

- オーブ軌道・バースト個数・オート花火位置はすべて `mulberry32` 製の**決定的疑似乱数**で生成し、
  `Math.random` は不使用。`R`(リセット)では PRNG も初期シードに戻すため、操作列が同じなら
  描画も再現される。
  - パーティクル初速の方向ぶれ(`speed`/`angle` のランダム)は Phaser エミッタ内部の乱数に依存する
    (ここは Phaser 側 API のため決定性の対象外)。軌道・発生位置・個数は決定的。

### アセット / フォールバック

- `../assets/`(= `8/assets/`)から SPEC のファイル名(`particle_spark.png` / `particle_smoke.png` /
  `orb.png` / `bg_dark.png`)で読込。`BootScene` で `loaderror` を捕捉し、欠落キーは
  `Graphics.generateTexture` で焼いた図形テクスチャにフォールバック:
  - パーティクル = **放射状グロー(中心白→外周透明)の白丸**(同心円の重ね塗りで近似)
  - オーブ = **発光円**(ハロー + コア)
  - 背景 = **暗色 + 微弱な星**(512x512 タイル)
- **図形フォールバック時も加算合成(ADD)は維持**する。**画像が 1 枚も無くても必ず起動する。**

## Codex 生成所感(Phaser 内蔵パーティクルの実装しやすさ)

- **API の素直さ**: Phaser 4 で `this.add.particles(x,y,texture,config)` がエミッタを直接返す形に
  なり、v3 のマネージャ階層が消えて書き味が良くなった。`scale`/`alpha`/`color`/`lifespan` を
  `{start,end}` や配列で渡すだけで「大→小・1→0・色グラデ」が宣言的に書け、寿命変化の記述は最短だった。
- **加算ブレンド**: `blendMode: 'ADD'` 一行で発光感が出る。`setBlendMode` でエミッタ単位に切替できるため
  ADD⇄NORMAL の比較トグルも容易。
- **生存数の制御が間接的**: 最大の難所。Phaser エミッタは「生存数を N に保つ」直接 API を持たず、
  `frequency`/`quantity`/`maxParticles` という**発生側のパラメータ**しかない。そのため
  `定常生存数 ≈ rate × 平均寿命` の式からレートを逆算する方式を採った。`Target` を厳密に張り付かせるのは
  難しく、実測 `Particles` は上限の周辺で揺れる(寿命に min/max のばらつきがあるため)。
  この「狙い撃ちのしづらさ」自体が、生存数を直接指定できるエンジン(PixiJS の自前更新等)との比較ポイント。
- **GPU の度合い**: Phaser のパーティクルは「**CPU で更新 + GPU でバッチ描画**」型で、Babylon の
  `GPUParticleSystem` のような完全 GPU シミュレーションではない。数万規模では JS 側の更新ループが
  FPS を支配しやすく、`Mode: GPU` は**描画がバッチされる**意味として表示している(README に明記)。
- **プール**: 単発バーストはエミッタ自体をプールして使い回す形にした。粒子そのものは Phaser 内部で
  プールされるため、エミッタの再生成さえ抑えれば GC はほぼ気にならなかった。
- 総じて、見た目と寿命変化は最短で書ける一方、**「目標生存数に張り付かせる」制御だけはエンジン非依存の
  数式で補う**必要があり、そこが内蔵パーティクル系エンジン共通の比較ポイントになると感じた。
