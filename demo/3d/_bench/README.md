# Bench Harness (3D) — 自動FPS計測

3D の **10テーマ × 4 JSエンジン（three.js / Babylon.js / PlayCanvas / A-Frame）** の
**同一負荷でのFPSを無人で自動巡回計測**し、CSVに出すツール。Godot は別系統（Web書き出し）のため対象外。
2D版（`2d/_bench/`）と計測コアは同一で、対象リストと一部ラベルのみ 3D 用。

## 仕組み（なぜゲーム本体を改変しないか）

- 各ゲームを `<iframe>` で順に起動。
- **FPSはハーネスが iframe内の `requestAnimationFrame` を直接駆動して実測**する。各ゲームの自己申告HUDに依存しないため、HUDの有無・形式に関係なく全エンジンを同条件で測れる。
- 負荷は **合成キーで "+" を送る**（`key:'+' / code:'Equal' / keyCode:187 / which:187` を window・document・canvas に dispatch）。3d各ゲームは `e.key.toLowerCase()==='+'`（`=`/`]`も同義）で負荷増加を判定するため、この1種で全エンジンに効く。
- `Objects`（HUDに `Objects` 行があるエンジン＝主にテーマ1）・初期転送量・初期化時間も併せて収集。

### 3d の負荷主軸はテーマごとに異なる

`+`/`-` が増減させる対象はテーマで違う（FPS計測自体は共通だが、CSVの解釈時に留意）:

| テーマ | 負荷主軸 |
|---|---|
| 01 | 小惑星インスタンス数 |
| 02 | 剛体（箱）数 |
| 03 | スキンドキャラ（glTF）数 |
| 04 | GPUパーティクル数 |
| 05 | 描画距離 / LOD |
| 06 | 動的シャドウ光源数 |
| 07 | ボクセルチャンク数 |
| 08 | PBRオブジェクト数（+Bloom） |
| 09 | A*ナビ群衆エージェント数 |
| 10 | レイキャスト本数（LIDAR） |

> 02〜10 はプレイヤー操作の無いビジュアル/物理ベンチなので、2D版のような「無入力で自機が被弾→オートリスポーン」の考慮は不要（より素直に計測できる）。

## 前提：`3d/` をルートに配信すること

テーマをまたいで巡回するため、**単一オリジン**が必要。`3d/` をHTTPルートにする
（ゲームの `../assets/` 参照は `3d/<テーマ>/<engine>/` から見て `3d/<テーマ>/assets/` に解決される）。

```bash
cd 3d
python -m http.server 8000
# → http://localhost:8000/_bench/ を Chrome で開く
```

> 単体テーマ配信（`cd 01`）ではハーネスから他テーマへ辿れない。計測時は必ず `3d/` ルートで。

## 使い方

1. 上記サーバを起動し `http://localhost:8000/_bench/` を **Chrome** で開く。
2. テーマ／エンジン／負荷レベル／タイミングを設定して **「▶ 計測開始」**。
3. 終了後 **「⤓ CSVダウンロード」**（`bench_results_3d.csv`）。

### パラメータ

| 項目 | 既定 | 意味 |
|---|---|---|
| 負荷レベル | `0,10,25` | デフォルト状態から累積で "+" を送る回数。SPECのSTEPは全エンジン共通なので同レベル＝同負荷設定。 |
| boot | 3500ms | iframe `load` 後の起動安定待ち。重いテーマ（A-Frame/Babylon、03のglTF群、08のPBR+Bloom）は伸ばす。 |
| settle | 1500ms | 負荷変更後の整定待ち（生成充填）。 |
| sample | 8000ms | FPSサンプリング長。 |
| keyGap | 40ms | "+" 連打の間隔。 |

### 出力CSV列

`theme, engine, level, fps_avg, fps_1pct_low, objects, transferKB, initMs, frames, error`

- `fps_avg` … サンプル区間の平均フレームレート。
- `fps_1pct_low` … フレーム時間の99パーセンタイルから算出した1%ロー（カクつきの指標）。
- `objects` … HUDに `Objects` 行があるエンジンのみ。多くのテーマは固有ラベル（Bodies/Particles/Lights/Rays 等）なので `null` になりうる。
- `transferKB` / `initMs` … 初期転送量・DOMContentLoadedまでの時間。

## 注意・既知の制約

- **このタブを前面に置いたまま放置**すること。背面タブはブラウザが rAF を絞るため計測が無効になる。
- **FPSはモニタのリフレッシュレートで頭打ち**（60/120/144Hz）。エンジン間の相対比較や「FPSが落ち始める負荷」は有効だが、上限値そのものは表示依存。固定リフレッシュ環境で計測すること。
- 公平性のため、**同一マシン・同一ブラウザ・canvas 960×540・DPR=1・AC電源/冷却済み**で1セッション通し計測する。
- 起動失敗・タイムアウトは該当行の `error` 列に記録される（boot を伸ばして再試行）。
